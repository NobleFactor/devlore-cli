# Platform and Package Management

> **Direction note (2026-06-04 — supersedes the `op.Platform` framing in this document).** The platform contract
> lives in `pkg/platform` as an **op-free** package (it imports nothing from `pkg/op`); `pkg/op` imports *it*, the
> same shape as `pkg/result` / `pkg/status`. Read **`op.Platform` as `platform.Platform`** and **`op.PackageManager`
> as `platform.PackageManager`** throughout. There is **no `op.Platform` type and no `platformData`** — identity
> serialization into the graph (the cross-platform-production use case) is deferred to
> [#282](https://github.com/NobleFactor/devlore-cli/issues/282). The Composite router, purl routing, the driver
> catalog, and state-query verification described below are implemented as written, in `pkg/platform`; the package
> verbs return a platform-local `Receipt` (`{Purl, PriorVersion, Version, Err}`), and the `pkg.Provider` veneer
> adapts those into `op.Receipt`.

## Core Idea

A **Platform** is the target a graph runs against — `os`, `arch`, `distro`, plus the package and service managers
that platform provides. It is a **capability** injected onto the `RuntimeEnvironment` (the same pattern as the
`ui` capability): the app constructs it, it flows onto the environment, and a stateless `platform` provider exposes
it to Starlark.

Package management hangs off the platform as a **Composite**: `op.Platform.PackageManager()` returns a single
`op.PackageManager` — a **router** — that routes each requested package to the right backend driver by its
**purl**, fans the work out, and returns one **receipt per package**. `pkg.Provider` is a thin veneer over that
router. The consumer never selects a package manager; the purl does.

```
RuntimeEnvironment.Platform : op.Platform
        │  .PackageManager()
        ▼
op.PackageManager  ← the per-platform Composite router (one contract at every level)
        ├── apt   (native, platform-bound)
        ├── brew  (native, platform-bound)
        ├── npm   (shared, cross-platform)
        └── …     leaf drivers
```

## The platform capability — `platform.Platform`

`platform.Platform` is an **interface** in `pkg/platform`: identity (`OS`, `Arch`, `Distro`, `Version`,
`Hostname`), `DefaultConcurrency`, a `ServiceManager()`, a single `PackageManager()` returning the platform's
router, and `DefaultPurlType()` / `ResolvePurlType()` for veneer-side purl normalization. There is no
consumer-facing manager *selection* surface (`DefaultPackageManager` / `AvailablePackageManagers` /
`PackageManagerByName`) — that collapses into the router's internal routing table.

**Cross-platform production is the driving requirement.** A graph authored on macOS that targets Ubuntu must carry
its target platform (`os: linux, distro: ubuntu, arch: amd64`) baked into the serialized graph document, then run
on a real Ubuntu host where the live managers are attached. Identity fields serialize; managers are runtime-only
and never serialized. **This serialization is deferred to
[#282](https://github.com/NobleFactor/devlore-cli/issues/282)** — the current cut ships `platform.Platform` as a
runtime capability, and baking it into the graph document is the Scenario-1 capstone.

Construction is **named specs → seal**: each named factory (`Debian()`, `Fedora()`, `Darwin()`, `Windows()`, …)
returns a fresh, mutable `*Spec`; `New(spec)` seals it into a `platform.Platform`. `Detect()` returns the host
spec. The catalog is the named systems devlore supports; distro families cover their members by descent
(Debian → Ubuntu, Mint; Fedora → RHEL, CentOS, Alma, Rocky).

`pkg/platform` is a **standalone, op-free package** — it imports nothing from `pkg/op`; `pkg/op` imports it.

## Package management is a Composite

`op.PackageManager` is the **same contract at every level**:

```go
type PackageManager interface {
    Install(packages []*Resource, kwargs map[string]any) (receipts []*Receipt, err error)
    Remove(packages  []*Resource, kwargs map[string]any) (receipts []*Receipt, err error)
    Upgrade(packages []*Resource, kwargs map[string]any) (receipts []*Receipt, err error)
}
```

Three pure layers:

- **Leaf = mechanism.** `aptManager`, `npmManager`, … do the operation on the list they are handed and return one
  `Receipt` per package, then stop. A leaf knows nothing of routing or of other managers.
- **Composite = routing + fan-out + unified receipts.** The platform's router groups the incoming list by purl,
  fans each slice out to its leaf concurrently, and concatenates the leaves' receipts into one unified result. It
  *is* an `op.PackageManager`, so the consumer sees one uniform surface.
- **`pkg.Provider` = thin veneer.** Adapts the Starlark/graph call, hands the list down, hands the unified receipts
  up. No policy, no convergence, no manager selection.

```
pkg.Provider.Install(resources, kwargs)              ← thin veneer
        ▼
op.PackageManager [composite].Install(resources, kwargs)
        │  group by purl type
        ├── apt → aptManager.Install([git=2.39], kwargs) → Receipt{git}
        ├── npm → npmManager.Install([typescript], kwargs) → Receipt{typescript}
        └── pip → pipManager.Install([black], kwargs)      → Receipt{black}
        ▼
unified []*Receipt  ── returned straight back out through pkg.Provider
```

## Routing by purl, normalized at construction

Every package is a purl. The purl's **type** (`apt`/`deb`, `dnf`/`rpm`, `brew`, `npm`, `pip`, …) selects the
driver. A `pkg.Resource` built from a bare purl with no manager (`"git"`) is **normalized at construction** by
inserting the platform's default native manager into both its URI and name — `"git"` on Debian becomes
`apt`-qualified (`pkg:apt/git`), while `"npm:typescript"` stays `npm`. So **every `Resource` that reaches the
router already carries a resolved manager**, routing is total and deterministic, and there is no `manager`
argument anywhere. An unknown purl type fails *that package's* receipt, not the whole call. (Resource identity —
the versionless catalog entry, version-as-state — is [§4.1 Resource Identity](4.1-resource-identity.md).)

## The driver catalog — native per platform, language managers shared

A platform's routing table is `nativeDrivers ∪ sharedLanguageDrivers` (`●` = platform default):

| Manager | Debian | Fedora | Darwin | Windows |
|---|:--:|:--:|:--:|:--:|
| `apt` *(repo + local `.deb`)* | ● | | | |
| `dnf` *(repo + local `.rpm`)* | | ● | | |
| `brew` | | | ● | |
| `port` | | | ✓ | |
| `winget` | | | | ● |
| `snap` | ✓ | ✓ | | |
| `flatpak` | ✓ | ✓ | | |
| `npm` / `pip` / `gem` / `cargo` / `go` | ✓ | ✓ | ✓ | ✓ |

**Native drivers** are platform-bound. **Language drivers** are cross-platform singletons declared **once** and
consumed by every platform — `cargo install X` behaves identically on every host, so there is no per-platform
copy. Adding a manager later is one leaf driver plus a routing-table entry: no consumer change, no signature
change.

## Purl reference matrix

Every `pkg.Resource` is identified by a [purl](https://github.com/package-url/purl-spec), and the purl **type** is
the routing key: the composite router (`op.Platform.PackageManager()`) dispatches each package to the leaf driver
registered for its type. Where the purl spec defines a type, devlore is **spec-compliant**; for managers the spec
does not cover (macOS, Windows, sandboxed Linux), devlore defines an **extension** type. All catalog URIs are
**versionless** — `@version` is state, not identity (see *Idempotence*, below); the examples show the full purl form
for reference.

The grammar is one shared base plus a per-type refinement (RFC 5234 ABNF):

```abnf
purl        = "pkg:" type "/" [ namespace "/" ] name [ "@" version ]
              [ "?" qualifiers ] [ "#" subpath ]
type        = 1*( lower / DIGIT / "." / "+" / "-" )   ; the routing key, lowercased
namespace   = segment *( "/" segment )
name        = 1*pchar
version     = 1*pchar
qualifiers  = qualifier *( "&" qualifier )
qualifier   = key "=" value
lower       = %x61-7A
token       = 1*( ALPHA / DIGIT / "-" / "." / "_" / "+" )
lower-token = 1*( lower / DIGIT / "-" / "." / "+" )
```

### Per-type refinements (ABNF)

```abnf
; deb → apt       (vendor namespace, lowercased name; arch / distro / repository_url qualifiers)
deb-purl     = "pkg:deb/" lower-token "/" lower-token [ "@" version ] [ "?" deb-quals ]
deb-quals    = deb-qual *( "&" deb-qual )
deb-qual     = "arch=" token / "distro=" token / "repository_url=" 1*pchar

; rpm → dnf       (vendor namespace, CASE-SENSITIVE name; arch / epoch / distro qualifiers)
rpm-purl     = "pkg:rpm/" lower-token "/" token [ "@" version ] [ "?" rpm-quals ]
rpm-quals    = rpm-qual *( "&" rpm-qual )
rpm-qual     = "arch=" token / "epoch=" 1*DIGIT / "distro=" token

; alpm → pacman   (vendor namespace, lowercased name; arch qualifier)
alpm-purl    = "pkg:alpm/" lower-token "/" lower-token [ "@" version ] [ "?" "arch=" token ]

; brew → brew     (extension; no namespace; cask qualifier)
brew-purl    = "pkg:brew/" lower-token [ "@" version ] [ "?" "cask=" ( "true" / "false" ) ]

; port → port     (extension; no namespace)
port-purl    = "pkg:port/" lower-token [ "@" version ]

; winget → winget (extension; publisher namespace, case preserved)
winget-purl  = "pkg:winget/" token "/" token [ "@" version ]

; snap → snap     (extension; no namespace; optional channel qualifier)
snap-purl    = "pkg:snap/" lower-token [ "@" version ] [ "?" "channel=" token ]

; flatpak → flatpak (extension; name is the reverse-DNS application id)
flatpak-purl = "pkg:flatpak/" app-id [ "@" version ]
app-id       = segment *( "." segment )

; choco → choco   (extension, PLANNED; no namespace)
choco-purl   = "pkg:choco/" token [ "@" version ]

; scoop → scoop   (extension, PLANNED; bucket namespace)
scoop-purl   = "pkg:scoop/" token "/" token [ "@" version ]

; npm → npm       (PLANNED; scope namespace, @ percent-encoded; case-sensitive name)
npm-purl     = "pkg:npm/" [ "%40" token "/" ] token [ "@" version ]

; pypi → pip      (PLANNED; no namespace; lowercased, underscores → hyphens; file_name qualifier)
pypi-purl    = "pkg:pypi/" lower-token [ "@" version ] [ "?" "file_name=" 1*pchar ]

; gem → gem       (PLANNED; no namespace; case-sensitive; platform qualifier)
gem-purl     = "pkg:gem/" token [ "@" version ] [ "?" "platform=" token ]

; cargo → cargo   (PLANNED; no namespace; case-sensitive)
cargo-purl   = "pkg:cargo/" token [ "@" version ]

; golang → go     (PLANNED; module-path namespace, lowercased)
golang-purl  = "pkg:golang/" lower-token *( "/" lower-token ) "/" lower-token [ "@" version ]
```

### Routing — purl type to provider

The router's table maps **purl type → leaf driver**. A bare name or manager-name prefix supplied by a caller is
normalized to a typed purl at `pkg.Resource` construction (the manager-aware veneer): a bare name takes the
platform's default native type (`git` on Debian → `pkg:deb/git`); a prefix resolves through the **name↔type map**
that bridges the manager name to its purl type. Most are identical (`brew`, `winget`, `snap`, `flatpak`, `port`,
`choco`, `scoop`, `npm`, `gem`, `cargo`), but five differ and the map must carry them: `apt` → `deb`, `dnf` → `rpm`,
`pacman` → `alpm`, `pip` → `pypi`, `go` → `golang`. Routing is by type and total: an unknown type fails *that
package's* receipt, not the whole call.

Elevation (whether a leaf shells out under root/admin) is **not** modeled here — it is the execution-topology
concern specified in [§6 Execution Topology](6-execution-topology.md) (`ElevationAware` / `ElevationProvider` /
`flow.elevate`); package leaves declare the need through that contract when wired, not via the purl identity.

## Idempotence is intrinsic; verification is state-query, never screen-scraping

**Idempotence is not a special case — it is the nature of the entire supported set.** Every manager's install is
already install-or-no-op-or-change-version (`apt-get install -y git=2.39.0`, `brew install …`, `npm install -g …`).
The framework does not *add* idempotence and does not reimplement it per platform; it inherits it from the
managers. (This is why a per-OS "package manager idempotence" treatment is a category error — there is nothing
OS-specific about it.)

The framework's **only** addition is **outcome verification by state query**. Each leaf brackets every operation:

1. **pre-query** the manager's actual state (installed? which version?),
2. run the (idempotent) command,
3. **post-query** the actual state,
4. emit a `Receipt`, **erroring it if the post-state did not reach what the purl requested**.

Crucially, the leaf **queries state — it never screen-scrapes command output**. Success and resulting state are
derived from re-observing the manager (`dpkg -s`, `brew list`, `npm ls -g`, a binary on `PATH`), not from parsing
stdout or trusting an exit code alone. This closes the hazard that opaque native flags can make a command exit `0`
without reaching the requested state (`apt-get install foo-` *removes* `foo`; `apt install --only-upgrade foo`
no-ops when `foo` is absent). The manager's chatter is ignored; its observable state is the truth.

## Verb surface — `Install` / `Remove` / `Upgrade`, plus `Update`

The mutating triad is kept for **consumer expectations** (apt and brew expose all three even though `install` alone
could converge everything), not collapsed to `Install`. Convergence is the manager's own idempotency — `Install`
converges a package to *present at the requested version*, `Remove` to *absent*, `Upgrade` moves named packages to
the latest available.

`Update` refreshes the package index, with two faces. Refresh is normally **automatic and staleness-gated**: a leaf
refreshes its own index before an index-consuming op (`Install` / `Upgrade` / `Search` / `Available`) only when
stale, and only managers with a local index (`apt` / `dnf` / `pacman` / `brew` / `port`) do it — live-store managers
(`snap` / `flatpak` / `winget`) no-op. **`Update()` is the manual force-refresh**, bypassing the staleness gate; the
router fans it out to every leaf. The automatic gate (`driver.ensureFresh`) precedes `Install`/`Upgrade`/`Search`/
`Available` and refreshes only when the index's age exceeds a TTL; `apt`/`pacman` report that age by index mtime,
while `brew`/`dnf` self-manage their freshness and `port` defers (manual `Update` still force-refreshes all five).
Both the gate and the manual force-refresh are implemented and tested.

`kwargs` are **opaque native-installer flags**, passed through verbatim per manager. Cross-tool option
normalization is deliberately out of scope; the state-query verification above is the backstop against a flag that
subverts the outcome.

## Per-package receipts and failure handling

Every call returns **one `Receipt` per package**; partial failure is normal (one package installs, another fails →
two receipts, one error). The leaf is always best-effort within a call and never self-rolls-back. How a failure
becomes `Degraded`, `Failed`, or `Stranded` — error-action dispatch, the four run terminals, compensation, and the
restartable journal — is the framework's job, not the package layer's. See
[§2.2 Phase Execution](2.2-phase-execution.md) and [§2.3 Orchestration Primitives](2.3-orchestration-primitives.md)
for the terminals and error actions, and [§5.1 Reconciliation](5.1-reconciliation.md) /
[§5.2 Recovery Serialization](5.2-recovery-serialization.md) for the journal and restart. The pkg leaf conforms by
returning a faithful per-receipt error; it adds no failure logic of its own.

## Design properties

- **One contract at every level.** Leaf and composite are both `op.PackageManager`; the consumer sees one surface.
- **Purl is the single routing key.** Manager selection is data (the purl), normalized at construction — never an
  argument, never a runtime guess.
- **Observe, don't trust.** State is read by re-querying the manager, never by scraping its output.
- **The platform owns its managers.** Native managers are platform-bound; language managers are shared singletons
  every platform consumes.
- **The target is serializable.** Identity serializes into the graph; live managers attach at run time — the basis
  for cross-platform production.
