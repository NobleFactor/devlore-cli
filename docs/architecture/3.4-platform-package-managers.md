# Platform and Package Management

## Core Idea

A **Platform** is the target a graph runs against ‚Äî `os`, `arch`, `distro`, plus the package and service managers
that platform provides. It is a **capability** injected onto the `RuntimeEnvironment` (the same pattern as the
`ui` capability): the app constructs it, it flows onto the environment, and a stateless `platform` provider exposes
it to Starlark.

Package management hangs off the platform as a **Composite**: `op.Platform.PackageManager()` returns a single
`op.PackageManager` ‚Äî a **router** ‚Äî that routes each requested package to the right backend driver by its
**purl**, fans the work out, and returns one **receipt per package**. `pkg.Provider` is a thin veneer over that
router. The consumer never selects a package manager; the purl does.


## The platform capability ‚Äî `op.Platform`

`op.Platform` is an **interface** in `pkg/op`: identity (`OS`, `Arch`, `Distro`, `Version`, `Hostname`),
`DefaultConcurrency`, a `ServiceManager`, and a single `PackageManager()` returning the platform's router. There is
no consumer-facing manager *selection* surface (`DefaultPackageManager` / `AvailablePackageManagers` /
`PackageManagerByName`) ‚Äî that collapses into the router's internal routing table.

**Cross-platform production is the driving requirement.** A graph authored on macOS that targets Ubuntu must carry
its target platform (`os: linux, distro: ubuntu, arch: amd64`) **baked into the serialized graph document**, then
run on a real Ubuntu host where the live managers are attached. So `op.Platform` is serialized through an
unexported `platformData` ‚Äî **identity fields serialize; managers are runtime-only and never serialized**. A
deserialized graph yields a `platformData` with nil managers; the host re-attaches real ones at run time.

Construction is **named specs ‚Üí seal**: each named factory (`Debian()`, `Fedora()`, `Darwin()`, `Windows()`, ‚Ä¶)
returns a fresh, mutable `*PlatformSpec`; `NewPlatform(spec)` seals it into an `op.Platform`. `Detect()` returns
the host spec. The catalog is the named systems devlore supports; distro families cover their members by descent
(Debian ‚Üí Ubuntu, Mint; Fedora ‚Üí RHEL, CentOS, Alma, Rocky).

`pkg/platform` is the **implementation layer** ‚Äî it implements the `op` contract; `op` does not depend on it.

## Package management is a Composite

`op.PackageManager` is the **same contract at every level**:

```go
type PackageManager interface {
    Install(packages []*Resource, kwargs map[string]any) (receipts []*Receipt, err error)
    Remove(packages  []*Resource, kwargs map[string]any) (receipts []*Receipt, err error)
    Upgrade(packages []*Resource, kwargs map[string]any) (receipts []*Receipt, err error)
}
```

Three pure layers:

- **Leaf = mechanism.** `aptManager`, `npmManager`, ‚Ä¶ do the operation on the list they are handed and return one
  `Receipt` per package, then stop. A leaf knows nothing of routing or of other managers.
- **Composite = routing + fan-out + unified receipts.** The platform's router groups the incoming list by purl,
  fans each slice out to its leaf concurrently, and concatenates the leaves' receipts into one unified result. It
  *is* an `op.PackageManager`, so the consumer sees one uniform surface.
- **`pkg.Provider` = thin veneer.** Adapts the Starlark/graph call, hands the list down, hands the unified receipts
  up. No policy, no convergence, no manager selection.

```
pkg.Provider.Install(resources, kwargs)              ‚Üê thin veneer
        ‚ñº
op.PackageManager [composite].Install(resources, kwargs)
        ‚îÇ  group by purl type
        ‚îú‚îÄ‚îÄ apt ‚Üí aptManager.Install([git=2.39], kwargs) ‚Üí Receipt{git}
        ‚îú‚îÄ‚îÄ npm ‚Üí npmManager.Install([typescript], kwargs) ‚Üí Receipt{typescript}
        ‚îî‚îÄ‚îÄ pip ‚Üí pipManager.Install([black], kwargs)      ‚Üí Receipt{black}
        ‚ñº
unified []*Receipt  ‚îÄ‚îÄ returned straight back out through pkg.Provider
```

## Routing by purl, normalized at construction

Every package is a purl. The purl's **type** (`apt`/`deb`, `dnf`/`rpm`, `brew`, `npm`, `pip`, ‚Ä¶) selects the
driver. A `pkg.Resource` built from a bare purl with no manager (`"git"`) is **normalized at construction** by
inserting the platform's default native manager into both its URI and name ‚Äî `"git"` on Debian becomes
`apt`-qualified (`pkg:apt/git`), while `"npm:typescript"` stays `npm`. So **every `Resource` that reaches the
router already carries a resolved manager**, routing is total and deterministic, and there is no `manager`
argument anywhere. An unknown purl type fails *that package's* receipt, not the whole call. (Resource identity ‚Äî
the versionless catalog entry, version-as-state ‚Äî is [¬ß4.1 Resource Identity](4.1-resource-identity.md).)

**Native drivers** are platform-bound. **Language drivers** are cross-platform singletons declared **once** and
consumed by every platform ‚Äî `cargo install X` behaves identically on every host, so there is no per-platform
copy. Adding a manager later is one leaf driver plus a routing-table entry: no consumer change, no signature
change.

## Idempotence is intrinsic; verification is state-query, never screen-scraping

**Idempotence is not a special case ‚Äî it is the nature of the entire supported set.** Every manager's install is
already install-or-no-op-or-change-version (`apt-get install -y git=2.39.0`, `brew install ‚Ä¶`, `npm install -g ‚Ä¶`).
The framework does not *add* idempotence and does not reimplement it per platform; it inherits it from the
managers. (This is why a per-OS "package manager idempotence" treatment is a category error ‚Äî there is nothing
OS-specific about it.)

The framework's **only** addition is **outcome verification by state query**. Each leaf brackets every operation:

1. **pre-query** the manager's actual state (installed? which version?),
2. run the (idempotent) command,
3. **post-query** the actual state,
4. emit a `Receipt`, **erroring it if the post-state did not reach what the purl requested**.

Crucially, the leaf **queries state ‚Äî it never screen-scrapes command output**. Success and resulting state are
derived from re-observing the manager (`dpkg -s`, `brew list`, `npm ls -g`, a binary on `PATH`), not from parsing
stdout or trusting an exit code alone. This closes the hazard that opaque native flags can make a command exit `0`
without reaching the requested state (`apt-get install foo-` *removes* `foo`; `apt install --only-upgrade foo`
no-ops when `foo` is absent). The manager's chatter is ignored; its observable state is the truth.

## Verb surface ‚Äî `Install` / `Remove` / `Upgrade`; no public `Update`

The surface is the familiar triad, kept for **consumer expectations** (apt and brew expose all three even though
`install` alone could converge everything), not collapsed to `Install`. Convergence is the manager's own
idempotency ‚Äî `Install` converges a package to *present at the requested version*, `Remove` to *absent*, `Upgrade`
moves named packages to the latest available.

There is **no public `Update`**. Refreshing a manager's package index (`apt-get update`) is an internal,
per-leaf, staleness-gated strategy: a leaf refreshes its own index before `Install`/`Upgrade` only when stale.
Managers that self-refresh (`brew`, `dnf`, `winget`; the language managers query their registry live) make it a
no-op.

`kwargs` are **opaque native-installer flags**, passed through verbatim per manager. Cross-tool option
normalization is deliberately out of scope; the state-query verification above is the backstop against a flag that
subverts the outcome.

## Per-package receipts and failure handling

Every call returns **one `Receipt` per package**; partial failure is normal (one package installs, another fails ‚Üí
two receipts, one error). The leaf is always best-effort within a call and never self-rolls-back. How a failure
becomes `Degraded`, `Failed`, or `Stranded` ‚Äî error-action dispatch, the four run terminals, compensation, and the
restartable journal ‚Äî is the framework's job, not the package layer's. See
[¬ß2.2 Phase Execution](2.2-phase-execution.md) and [¬ß2.3 Orchestration Primitives](2.3-orchestration-primitives.md)
for the terminals and error actions, and [¬ß5.1 Reconciliation](5.1-reconciliation.md) /
[¬ß5.2 Recovery Serialization](5.2-recovery-serialization.md) for the journal and restart. The pkg leaf conforms by
returning a faithful per-receipt error; it adds no failure logic of its own.

## Design properties

- **One contract at every level.** Leaf and composite are both `op.PackageManager`; the consumer sees one surface.
- **Purl is the single routing key.** Manager selection is data (the purl), normalized at construction ‚Äî never an
  argument, never a runtime guess.
- **Observe, don't trust.** State is read by re-querying the manager, never by scraping its output.
- **The platform owns its managers.** Native managers are platform-bound; language managers are shared singletons
  every platform consumes.
- **The target is serializable.** Identity serializes into the graph; live managers attach at run time ‚Äî the basis
  for cross-platform production.
