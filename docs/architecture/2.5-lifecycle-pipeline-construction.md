# Lifecycle Pipeline Construction

How a tool command (e.g. `lore deploy`) turns a package definition into a **sealed, serializable execution
graph** ‚Äî built host-independently, saved, loaded, and run on its target operating system. This is the
build-time, authoring counterpart to the runtime model in [2.2 Phase Execution](2.2-phase-execution.md); it
relies on the immutable graph seal (phase-8 step 21) and the projected provider API in
[3.2 The Projected Provider API](3.2-projected-provider-api.md).

## The pipeline model

It is lore's job to **construct lifecycle pipelines**. A package declares a lifecycle composed of **pipelines**
‚Äî one per `Action`. A pipeline is an **ordered sequence of phase subgraphs executed in order; the outputs of
one phase are the inputs to the next.** The phase order is canonical, defined in
`internal/lorepackage/lifecycle.go` (`PhaseOrder(action)`):

| Pipeline (`Action`) | Phase order |
|---|---|
| `Deploy` | `prepare` ‚Üí `install` ‚Üí `provision` ‚Üí `verify` |
| `Upgrade` | `prepare` ‚Üí `upgrade` ‚Üí `migrate` ‚Üí `verify` |
| `Decommission` | `unprovision` ‚Üí `uninstall` ‚Üí `cleanup` |
| `Reconcile` | `scan` ‚Üí `repair` ‚Üí `verify` |

Each phase is authored as a starlark script (`def <phase>(package, phase): ‚Ä¶`) and **materializes as one
discrete subgraph**. A pipeline graph is a package subgraph whose children are the phase subgraphs **in phase
order**, edge-linked so each phase's outputs feed the next.

**Missing phases are excluded.** A phase with no script in the package definition is omitted, and the present
phases are linked directly across the gap ‚Äî e.g. a package with no `provision` script links `install ‚Üí verify`
and never materializes a `provision` subgraph. The phase *order* is canonical; the phase *set* is whatever the
package actually provides.

```
graph (Origin{Tool, Scope, Annotations})
‚îî‚îÄ‚îÄ package subgraph                          (e.g. "subgraph.docker")
    ‚îú‚îÄ‚îÄ phase subgraph: prepare    ‚îÄ‚îê
    ‚îú‚îÄ‚îÄ phase subgraph: install    ‚îÄ‚î§ outputs ‚Üí inputs, in canonical order
    ‚îú‚îÄ‚îÄ (provision omitted ‚Äî no script in this package)
    ‚îî‚îÄ‚îÄ phase subgraph: verify     ‚îÄ‚îò
```

## Platform-layered script selection

Phase scripts are laid out as a **platform taxonomy** under `<package>/<platform-layer>/<Action>/<phase>.star`.
A target platform selects an ordered **cascade** of layers ‚Äî most-general ‚Üí most-specific ‚Äî via
`PlatformResolutionOrder` (`internal/lorepackage/lifecycle.go`):

| Target platform | Resolution cascade |
|---|---|
| `Linux.Debian` (Ubuntu) | `Common` ‚Üí `Unix` ‚Üí `Linux` ‚Üí `Linux.Debian` |
| `Linux.Fedora` | `Common` ‚Üí `Unix` ‚Üí `Linux` ‚Üí `Linux.Fedora` |
| `Darwin` (macOS) | `Common` ‚Üí `Unix` ‚Üí `Darwin` |
| `Windows` | `Common` ‚Üí `Windows` |

For each phase, `DiscoverPhaseScripts(packageDir, platform, action, phase)` returns **every existing layer's**
`<layer>/<Action>/<phase>.star` in cascade order. A phase is **constructed in parts**: all of those scripts are
**loaded into one** planning starlark runtime, general‚Üíspecific, sharing a namespace ‚Äî the general layers
(`Common`/`Unix`/`Linux`) contribute reusable definitions, the specific layers build on them. **Exactly one**
script in the cascade defines the **`run`** entry point (the launch point): zero means the phase is absent and
is skipped; more than one is an error. Construction uses `DiscoverPhaseScripts` (the whole cascade), **not**
`GetPhaseScript` (most-specific only).

**Doubly sparse ‚Äî present-only.** Missing *phases* drop out (above) *and* missing *layers* drop out. The phase
order and the layer cascade are canonical; the actual scripts are whatever the package provides. A package may
use only leaf-platform dirs with no shared layers ‚Äî e.g. docker has only `Darwin` / `Linux.Debian` /
`Linux.Fedora` / `Windows`, so on Ubuntu each Deploy phase resolves to a single
`Linux.Debian/Deploy/<phase>.star` (`Common`/`Unix`/`Linux` absent ‚Üí skipped).

## Go ‚Üî Starlark during planning

Planning runs inside the free-standing `op.Plan(ctx, spec, fn)`: it stands up **one** shared
`RuntimeEnvironment` (and therefore one set of providers) and runs the Go closure `fn(env)` ‚Äî lore's
construction *is* that closure. The two sealed APIs meet at a single, narrow hand-off on the shared env: the
**plan provider's `InvocationRegistry`**. Starlark **writes** invocations to it (every `plan.*` call); Go
**reads** them (harvest). The bridge runtime is built over the same env, so its `plan` / `platform` / `ui`
globals *are* the shared providers. Go orchestrates with the sealed constructors; starlark authors with the
sealed `plan.*` surface; neither reaches into the other's internals.

**Go drives, starlark fills ‚Äî one phase at a time.** Inside the closure, lore loops the phases in order: load
the phase's cascade into a starlark runtime over the shared env, call its single `run(package, phase, ‚Ä¶)` (Go ‚Üí
starlark), harvest the registry into that phase's subgraph (starlark ‚Üí Go), then `Clear()` the registry before
the next phase. The graph is built up phase by phase within the one `op.Plan` scope.

**Phase isolation ‚Äî each phase adds only to its own phase subgraph.** A phase's `run` (and the cascade helpers
it calls) contributes units **exclusively to that phase's subgraph**; **no phase can add to any other
subgraph** ‚Äî not a sibling phase, not the package subgraph, not the root. Enforced mechanically: the shared
registry is harvested into the current phase's subgraph and then `Clear()`ed, and `plan.*` offers no affordance
to target another container. The **only** coupling between phases is the named-argument **data** flow (outputs ‚Üí
inputs, by value through the binding layer) ‚Äî never structural cross-contamination. The graph is therefore a
strict containment tree: package ‚Üí phase subgraphs ‚Üí units, each phase owning exactly one subtree.

## Construction

A command builds the graph bottom-up via **gather-then-construct** ‚Äî assemble children first, then call the
sealed all-args constructor. No graph is mutated after construction.

**Phase scripts ‚Üí phase subgraph.** lore creates **one per-phase** planning runtime ‚Äî with `plan`, `platform`,
and `ui` injected as globals (the `platform` provider is `access=both` and exposes `arch` / `os` / `distro` /
`version` / `hostname` for the targeted OS; lore injects it when it initializes the planning
`RuntimeEnvironment`) ‚Äî and **loads the whole platform cascade** (above) into it. It then calls the cascade's
single **`run(package, phase)`** entry point (`package`: metadata / features / settings; `phase`: identity /
`Action` / retry). The `run` body, and the cascade helpers it calls, issue `plan.*` operations ‚Äî each
registering a detached `*op.Invocation` in the runtime's `InvocationRegistry`. The command then **harvests** the
registry: the phase's top-level units are the invocations whose `Target.ParentID()` is empty (units nested under
a combinator carry a non-empty parent). Those become the children of the phase subgraph, built with
`NewSubgraph(id, action, annotations, children, slots, errorAction, retryPolicy)`. The `RuntimeEnvironment` is
shared for the whole `op.Plan` scope; phase isolation comes from `Clear()`ing the registry between phases (per
"Go ‚Üî Starlark during planning" above), with each phase's units landing only in its own subgraph.

**Native package-manager actions ‚Üí invocations.** A phase may also declare native-PM operations (install /
remove / upgrade). These are built as invocations through `Method.Planner().Plan(provider, receiverType,
method, args, slots, annotations, errorAction, retryPolicy)` and registered into the same per-phase provider,
so they assemble alongside the script's units.

**Gather up.** Phase subgraphs are gathered into the package subgraph (`NewSubgraph`), package subgraphs into
the graph (`NewGraph(origin, units, slots, catalog, rollback, retryPolicy, sopsClient)`).

## `run` arguments ‚Äî inter-phase flow and external binding

`run`'s signature is `run(package, phase, ‚Ä¶)`. `package` and `phase` are **framework-controlled** (lore sets
their values) and **reserved** ‚Äî no additional argument may be named `package` or `phase`. Beyond them a package
author may declare **additional arguments**, which the author documents. starlark is untyped, so the framework
cannot verify them: it **passes what it finds** (best-effort), with `envValue` (`pkg/op/env_value.go`) doing
JSON-based conversion where a typed value is needed.

Additional arguments are **bound by name** from three sources, through the binding layer
([2.1 Typed Slots](2.1-typed-slots.md), `VariableResolver`):

- **Inter-phase flow (stage ‚Üí stage).** A phase's outputs flow to the next phase's `run` by matching argument
  name ‚Äî this is how "the outputs of one phase are the inputs to the next" is realized concretely.
- **Command line.** Arguments on `lore deploy ‚Ä¶` are passed as **variables to the graph root**
  (`VariableResolver`, Flags source) and flow through to a `run` argument of the matching name.
- **Environment variables.** Arrive as strings, JSON-decoded to typed values
  (`envValue.ConvertTo` ‚Üí `json.Unmarshal`), bound by name.

Resolution is the framework's variable model: the root carries the variables; each phase's `run` arguments are
parameters resolved against prior-phase outputs first, then the root variables (CLI / env / config), by name, at
preflight. There is no type checking ‚Äî the package author owns and documents the contract.

## Origin and provenance

The graph carries an `Origin{Tool, Scope, Annotations}` (see
[D15](../plans/extract-starlark-from-op/phase-8.md)): `Tool` is the producing program (framework-stamped from
the runtime program name), `Scope` is per-graph plan data the tool supplies (e.g. the package set), and
`Annotations` is an opaque tool bag. **Per-unit provenance is a unit `Annotation`, not an `Origin`** ‚Äî an
`ExecutableUnit` has `Annotations` only. lore records the package a unit belongs to as
`annotations["package"] = <name>`; the framework is key-agnostic (the receipt carries the whole annotation map
opaquely), so tools own their keys.

## Portability ‚Äî the core promise

Because the graph is immutable and fully serializable, construction is **host-independent**: the graph for any
target OS can be **built on any host**, then **saved**, **loaded** faithfully (Origin, unit annotations, slots,
edges all round-trip), and **run on the target OS**. A package's per-OS phase scripts select the right
construction per target; the resulting graph is a portable plan, not a host-bound script.

```
build (any host) ‚îÄ‚îÄ‚ñ∫ save (wire form) ‚îÄ‚îÄ‚ñ∫ load (faithful) ‚îÄ‚îÄ‚ñ∫ run (target OS)
```

## The `plan.*` construction surface

Phase scripts author graphs through the projected `plan.*` API: provider sub-namespaces (`plan.package`,
`plan.shell`, `plan.file`, ‚Ä¶) whose method calls register invocations; combinators (`plan.choose`,
`plan.gather`, ‚Ä¶) for runtime branching/iteration; `plan.variable` / `plan.origin` for binding-layer values;
and `plan.assemble` for explicit assembly where a script drives it directly. The exact surface is the proving
ground for the API: building real packages against it is how gaps and ergonomics are found and fixed.
