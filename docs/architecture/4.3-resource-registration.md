# Resource Registration Lifecycle

How resource types announce themselves, register constructors, and integrate with
the marshaler. This document describes the two-phase registration model that
parallels the provider loading pattern documented in
[Provider Loading](3.1-provider-loading.md).

## 1. Motivation

Resource types register constructors with the marshaler so that Starlark values
can be coerced into typed Go resources. Previously, every `resource.go` file
contained a hand-written `init()` function that eagerly called
`op.RegisterConstructor`, and the `mem` package used a separate
`RegisterCallableExtractor` mechanism. The two-phase lifecycle below eliminates
both patterns — constructors are now generated and lazy, and callable extraction
goes through the same `AnnounceResource` / `loadConstructor` path as all other
resource types.

## 2. Two-Phase Resource Lifecycle

Resources adopt the same lifecycle that providers use:

```text
Phase 1: Announcement (init-time, lightweight)
  gen/resource.gen.go init()
    → op.AnnounceResource(&fileResource{})
    → Stores descriptor in registry — no constructor logic runs

Phase 2: Initialization (first-use, lazy)
  Marshaler encounters file.Resource for the first time
    → Looks up descriptor by reflect.Type
    → Calls descriptor.Init()
    → Init() registers the constructor via op.RegisterConstructor
    → Subsequent coercions proceed normally
```

### 2.1 Announcement

Generated code calls `AnnounceResource()` during `init()`. This is cheap — it
stores a type descriptor with no side effects:

```go
// Generated in gen/resource.gen.go
type fileResource struct{}

func (d *fileResource) Name() string { return "file.Resource" }

func (d *fileResource) Init() error {
    op.RegisterConstructor(provider.ResourceFromValue)
    return nil
}

func init() {
    op.AnnounceResource(&fileResource{})
}
```

The `ResourceDescriptor` interface:

```go
type ResourceDescriptor interface {
    Name() string
    Type() reflect.Type
    Init() error
}
```

### 2.2 Lazy Initialization

Constructor lookups happen at three points in the codebase:

| Call site          | File                    | When                              |
| ------------------ | ----------------------- | --------------------------------- |
| `unmarshalValue`   | `starvalue_marshal.go`  | Plan-time slot filling            |
| `coerceSlotValue`  | `action_reflect.go`     | Execution-time arg coercion       |
| `constructResource`| `starvalue_marshal.go`  | Resource construction helper      |

Additionally, `validateSlotType` (`action_reflect.go`) checks constructor
existence for plan-time validation without invoking.

All lookups go through a single `loadConstructor` helper that wraps
`constructorRegistry.Load` with lazy init from the resource announcement
registry:

```text
loadConstructor(targetType)
  → Is there a constructor for targetType? Yes → return it
  → No constructor yet → is there an announced descriptor? Yes → call Init()
  → Init() registers the constructor via RegisterConstructor
  → Re-check registry → return constructor or error
```

Each descriptor's `Init()` is protected by `sync.Once` — exactly-once
guarantee even under concurrent access. Failed `Init()` caches the error
and returns it on all subsequent attempts.

This ensures:
- Resources that are never used pay zero initialization cost
- The first lookup (any of the three paths) triggers lazy init transparently
- After init, the fast path (direct registry lookup) applies

## 3. Constructor Convention

The constructor function is hand-written by the developer in `resource.go`.
Codegen detects it by matching the signature:

```go
func(any) (Resource, error)
```

Where `Resource` is the type in the same package that embeds `op.ResourceBase`.
The function name is chosen by the developer. The recommended name is
`ResourceFromValue`:

```go
// Hand-written in resource.go
func ResourceFromValue(v any) (Resource, error) {
    s, ok := v.(string)
    if !ok {
        return Resource{}, fmt.Errorf("file.Resource: expected string, got %T", v)
    }
    return NewResource(s), nil
}
```

Rules:
- One constructor per resource type. Codegen errors if multiple functions match.
- Packages without a matching function (e.g., tombstone-only packages like
  `archive` and `encryption`) produce no resource descriptor — codegen simply
  has nothing to generate.
- When a resource is added later, the developer writes the struct and
  constructor; codegen picks it up automatically on the next `make generate`.

### 3.1 Source-type declaration (typed constructors)

The default constructor takes `any` — a poor man's union, since Go has no sum
type for a *value* parameter. An `any` constructor is **target-driven only**:
`op.Convert` reaches it when a slot already names the resource type (e.g. a
string URI coerced to a `*function.Resource`).

A resource born from a specific Go value — notably a Starlark builtin like
`*starlark.Function` — declares that **source type** by constraining the
constructor with a type set, the one form of union Go *does* have (in constraint
position) and the only one codegen can read off a signature:

```go
func NewResource[T *starlark.Function | string](
    env *op.RuntimeEnvironment, unit op.ExecutableUnit, identity T,
) (*Resource, error)
```

Codegen reads the type-set members — `goast` surfaces the type-parameter
constraint — and keeps only **Starlark builtins** (an allowlist: the exported
`go.starlark.net/starlark` data types plus `starlark.Value`), registering
`byType[sourceType] = the receiver type` for each through an `AnnounceResource`
source-types argument, reusing the existing type map (**no new registry**). A
member outside the allowlist — a Go primitive like the URI `string` — is **not**
keyed and stays target-driven. So `*starlark.Function → function.Resource`
becomes a source key while `string` does not. (In practice `*starlark.Function`
is the only one used; the wider allowlist is near-free and forward-looking.)

The read side is `ReceiverRegistry.ConstructorForSource(reflect.TypeOf(value))`,
called by the planner: when an immediate value is a Go type with a registered
source constructor, the planner builds the resource before its addressing
switch — naming only `op` and `reflect`, never the provider. This is how a
Starlark function reaches its content-addressable `function.Resource` without
`op` or `starlarkbridge` depending on `function`.

This relaxes the one-constructor rule above: a resource may pair a **typed
source constructor** (the declaration, e.g. `NewResource`) with an **`any`
announced constructor** (the target-driven path, e.g. `DiscoverResource`). The
generic shell stays thin — it erases to `any` into a shared unexported
`buildCandidate` — so the only monomorphized code is the constructor's few
lines, once per instantiated GC shape.

## 4. Callable Extraction

Callable extraction is a special case of value coercion:
`*starlark.Function → func(...)`. It uses the same `AnnounceResource` /
`loadConstructor` path as all other resource types — no special registry or
function pointer indirection.

### Coercion Flow

Coercion is based on the Starlark type and the Go parameter type:

```text
Plan time (planned_reflect.go):
  Starlark value: *starlark.Function
  Go param type:  func(...)  (e.g., file.Reducer)
    → extractCallable(fn, funcType)
    → loadConstructor(callableResourceType)  — lazy init via AnnounceResource
    → mem.Extract(fn, funcType) + Compile()
    → Stores CallableResource in slot

Execution time (action_reflect.go):
  Slot contains: CallableResource (concrete *mem.Callable)
  Go param type:  func(...)
    → initCallableSlots() → Init() → buildCallableFunc()
    → Adapted Go func replaces slot value before coerceArgs
```

### Registration

The `mem` package announces a `callableDesc` resource descriptor in `init()`.
The descriptor's `Init()` — protected by `sync.Once` — registers a constructor
for the `CallableResource` interface type via `RegisterConstructor`:

```go
// mem/resource.go
type callableDesc struct{}

func (d *callableDesc) Name() string       { return "mem.Callable" }
func (d *callableDesc) Type() reflect.Type { return reflect.TypeOf((*op.CallableResource)(nil)).Elem() }

func (d *callableDesc) Init() error {
    op.RegisterConstructor[op.CallableResource](func(v any) (op.CallableResource, error) {
        input := v.(op.CallableInput)
        c, err := Extract(input.Fn, input.FuncType)
        // ... Compile ...
        return c, nil
    })
    return nil
}

func init() { op.AnnounceResource(&callableDesc{}) }
```

The `CallableResource` interface, `buildCallableFunc`, `initCallableSlots`, and
adapter helpers all live in `starvalue_marshal.go`. The `Extract` and `Compile`
logic stays in `mem` as domain code.

## 5. What Developers Write vs. What Codegen Produces

### Developer writes (`resource.go`)

```go
// Resource struct with domain fields
type Resource struct {
    op.ResourceBase
    SourcePath SourcePath
    // ...
}

// Constructor — detected by signature
func ResourceFromValue(v any) (Resource, error) { ... }

// Domain methods
func NewResource(path string) Resource { ... }
func (r *Resource) Resolve() error { ... }
```

### Codegen produces (`gen/resource.gen.go`)

```go
type fileResource struct{}

func (d *fileResource) Name() string { return "file.Resource" }

func (d *fileResource) Init() error {
    op.RegisterConstructor(provider.ResourceFromValue)
    return nil
}

func init() {
    op.AnnounceResource(&fileResource{})
}
```

The developer never writes `init()`, never calls `RegisterConstructor`, never
touches the announcement registry. Codegen handles all bookkeeping. The
generated files live in `gen/` and are untracked.

## 6. Registration Sequence (Complete)

After this change, the full initialization sequence for a provider package
(using `file` as an example):

```text
Import Phase:
  register.go blank-imports all gen packages

init() Phase (lightweight):
  gen/provider.gen.go  → op.Announce(&fileProvider{})
  gen/resource.gen.go  → op.AnnounceResource(&fileResource{})

Framework Boot:
  op.InitAll(reg, ctx)
    → For each announced provider:
      → Create provider instance
      → InitProvider(provider, ctx)
      → RegisterReflectedActions(reg, name, provider, Params)

First Use (lazy):
  Marshaler encounters file.Resource
    → Descriptor.Init() → RegisterConstructor(ResourceFromValue)
    → Constructor called: string → file.Resource
```

No hand-written `init()` exists in any `resource.go`. All registration is
either generated (announcement) or triggered lazily (constructor).
