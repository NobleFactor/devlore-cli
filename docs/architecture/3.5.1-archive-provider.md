## Thesis

The archive provider turns a compressed archive into a set of **tracked, reversible file mutations**. Two ideas carry
the whole design:

1. **Content over extension.** The archive's format is discovered by reading its leading bytes — the compression and
   container magic numbers — not by trusting the filename. A `.tar.gz` that is really a zip, or a release asset named
   `download` with no extension, is handled by what it *is*, not what it is *called*.
2. **Extraction is file mutation.** Writing an extracted entry to disk is the *same* operation as `file.write_bytes`:
   stream bytes to a target path, displace any prior content to the recovery site, and emit a `file.Receipt` that undoes
   it. Extraction is therefore a **loop over the unified file-mutation core** (see
   [file-mutation-receipts.md](../plans/extract-starlark-from-op/phase-8/file-mutation-receipts.md)), not a bespoke I/O
   path with its own rollback logic. The provider owns *format* and *iteration*; the file provider owns *mutation* and
   *undo*.

These two ideas keep the provider small: a format-detection front-end and a decompressor pipeline feeding a single
container reader, with each entry handed to `file.Provider` for the actual write and its self-describing receipt.

## 1. Role in the execution graph

`archive.extract` is a **Compensable** action (`pkg/op/provider/archive/provider.go`, `+devlore:access=planned`). At
dispatch it receives the archive as a `*file.Resource` and an existing destination directory, and returns:

- `[]*file.Resource` — one canonical catalog entry per file entry the extraction created or replaced, in extraction
  order;
- `*op.RecoveryStack` — one `file.Receipt` per entry (file *or* directory), so a failed run unwinds the whole
  extraction in reverse;
- `error` — from detection, decompression, extraction, archive-on-displace, or receipt construction.

`CompensateExtract` is the complement: it unwinds the returned stack. Because every entry's receipt is a
self-describing `*file.Receipt` (it names `file.CompensateFileMutation` as its undo), `CompensateExtract` collapses to a
single `stack.Unwind()` — there is no archive-specific rollback. The destination directory must already exist; the
provider does not create the prefix (it mirrors `tar -C`, which fails on a missing target).

## 2. The two-layer model

A compressed tarball is two independent layers — a **compression wrapper** around a **container stream**. The provider
separates them, which is what lets new formats cost a single table entry rather than a new extractor:

```
file header ──► [Layer A: outer sniff]
                   │
                   ├─ zip magic  ──────────────► zip branch (random access; its own container + compression)
                   │
                   └─ pick decompressor by magic ──► io.Reader (decompressed bytes)
                         gzip / bzip2 / xz / zstd / identity        │
                                                                    ▼
                                                       [Layer B: tar container]
                                                          tar.NewReader(stream)
```

- **Layer A — outer container / compression.** Sniff the file header. If it is a **zip**, take the zip branch: zip is a
  container *and* its own per-entry compression, and `archive/zip` reads the central directory from the **end** of the
  file, so it needs random access (`io.ReaderAt` + size, or `zip.OpenReader(path)`) and cannot be streamed through the
  decompressor pipeline. Otherwise, select a **decompressor** by magic — gzip, bzip2, xz, zstd, or the **identity**
  reader (a plain, uncompressed `.tar`) — and wrap the file reader in it.
- **Layer B — tar container.** Every non-zip format funnels into **one** `tar.NewReader` over the decompressed stream.
  `.tar`, `.tar.gz`, `.tar.bz2`, `.tar.xz`, and `.tar.zst` differ only in which decompressor Layer A chose; the
  iteration, entry handling, and receipt emission are identical.

Adding a sixth tar compressor later is a one-line registration in Layer A's decompressor table, not a new code path.

## 3. Content-based format detection

Detection reads the archive's leading bytes and matches known magic numbers. It is the front-end to Layer A and the
sole replacement for today's extension switch (`provider.go:109-117`).

| Format        | Sniffed at | Magic (hex)               | Notes                                          |
|---------------|-----------:|---------------------------|------------------------------------------------|
| gzip          | offset 0   | `1F 8B`                   | compression wrapper; container confirmed by Layer B |
| bzip2         | offset 0   | `42 5A 68` (`BZh`)        | compression wrapper                            |
| xz            | offset 0   | `FD 37 7A 58 5A 00`       | compression wrapper                            |
| zstd          | offset 0   | `28 B5 2F FD`             | compression wrapper                            |
| zip           | offset 0   | `50 4B 03 04` (also `05 06` empty, `07 08` spanned) | container; random-access branch |
| tar (`ustar`) | offset 257 | `75 73 74 61 72`          | identity decompressor → tar                    |

**Robustness rules:**

- **Read with `io.ReadFull` / `io.ReadAtLeast`, never a bare `Read`.** A single `Read` may return fewer than the
  requested bytes on a valid, non-EOF stream; the `ustar` probe needs the full 262-byte prefix, so a short read could
  miss it. Read up to 262 bytes and tolerate `io.EOF` / `io.ErrUnexpectedEOF` for files shorter than that (a small file
  can still be a valid gzip or zip — the compression sniffs need only the first 2–6 bytes).
- **Non-destructive sniff.** Detection must not consume the stream the extractor then reads. Because the source is a
  file path, the provider reads the header from an `*os.File`, then `Seek(0, 0)` before handing the file to Layer A (zip
  reopens by path / `ReaderAt`; the tar path wraps the rewound file in the chosen decompressor).
- **Identity fallback and the V7 caveat.** When no compression or zip magic matches, the `ustar` magic at offset 257
  selects the plain-tar (identity) path. Pre-POSIX V7 tar carries **no** magic and is therefore not content-detectable;
  such inputs resolve to *unsupported*. (The tar header checksum is the only fully general validator and is out of scope
  for a header sniff.)

**Why content, not extension.** Extension trust silently mis-routes mislabeled files and rejects unlabeled ones, and the
container claim it makes (`.tar.gz` ⟹ tar) is unverifiable until the bytes are read anyway. Content detection both
routes correctly and lets Layer B *confirm* the container: a gzip stream whose payload is not a tar fails at
`tar.NewReader`, surfacing a precise error instead of a corrupt extraction.

## 4. Supported formats and roadmap

| Format            | Outer (Layer A) | Container | Decompressor                              | Dependency        |
|-------------------|-----------------|-----------|-------------------------------------------|-------------------|
| `.tar.gz` / `.tgz`| gzip            | tar       | `compress/gzip`                           | stdlib (have)     |
| `.zip`            | zip             | —         | `archive/zip`                             | stdlib (have)     |
| `.tar.bz2`/`.tbz2`| bzip2           | tar       | `compress/bzip2` (decompress-only)        | stdlib            |
| `.tar.xz`/`.txz`  | xz              | tar       | `github.com/ulikunitz/xz`                 | **new** (pure Go) |
| `.tar.zst`/`.tzst`| zstd            | tar       | `github.com/klauspost/compress/zstd`      | **new** (pure Go) |
| `.tar`            | none (identity) | tar       | —                                         | stdlib            |

This set is the 80% case for software distribution: tar — bare or wrapped in one of four compressors — plus zip. Explicitly **out
of scope**: `.7z` and `.rar` (no stdlib support, heavier/less-mature Go libraries, complex or proprietary formats);
bare single-file `.gz`/`.bz2` that are not tarballs (a different shape — one output file, no container); and high-level
detect-and-extract libraries (they cannot integrate with the archive-on-displace receipt model — the provider must drive
the tar/zip iteration itself).

## 5. The entry iterator

Layers A and B are presented to the extract loop through one internal iterator, so the loop is identical for tar and
zip:

```go
// archiveReader yields one entry at a time; Close releases the underlying file/decompressor.
type archiveReader interface {
	Next() (*archiveEntry, error) // io.EOF when exhausted
	io.Closer
}

// archiveEntry is one member of the archive, positioned at its bytes.
type archiveEntry struct {
	Name   string      // archive-relative path
	Mode   os.FileMode // entry permissions
	IsDir  bool        // directory entry vs. regular file
	Reader io.Reader   // the entry's content; valid until the next Next()
}
```

`openArchive(source)` performs detection, builds the Layer-A decompressor (or the zip reader), and returns the
`archiveReader`. The tar implementation wraps `tar.Reader.Next`; the zip implementation iterates `zip.ReadCloser.File`,
opening each member. The extract loop consumes this iterator and is unaware of format.

## 6. Extraction as file-mutation receipts

Each iterated entry becomes a call into the **unified file-mutation core** (see
[file-mutation-receipts.md](../plans/extract-starlark-from-op/phase-8/file-mutation-receipts.md)). The provider obtains
the file provider via `provider.Instance[file.Provider](runtimeEnvironment)` and, per entry:

- a **directory** entry → the existing `fileProvider.Mkdir(activationRecord, target.SourcePath.Abs(), mode, "")` (there
  is no `MakeDir` — `Mkdir` already creates a directory and returns a self-describing receipt);
- a **regular file** entry → `fileProvider.WriteFile(target, entry.Reader, mode)` — streamed with `io.Copy` in constant
  memory, so a multi-gigabyte member never buffers in full. (The decompressed entry is a stream, not an `*os.File`, so
  the kernel `copy_file_range`/`sendfile` fast path `WriteFile` uses for file-to-file copies does not apply here — the
  win on the extraction path is constant-memory streaming.)

Both return a self-describing `*file.Receipt`. When the target path already holds content, the mutation core archives
the prior bytes to `op.RecoverySite` (`runtimeEnvironment.RecoverySite.ArchiveFile`) before overwriting, recording the
recovery id on the receipt; brand-new targets just get written. Each receipt is pushed onto the run's
`*op.RecoveryStack`.

**Compensation.** Because every pushed receipt is a `*file.Receipt` naming `file.CompensateFileMutation` as its undo,
`CompensateExtract(stack)` is exactly `stack.Unwind()`: in reverse order each entry is removed, any displaced prior
content is restored from recovery, and directories the extraction created are pruned via the receipt boundary chain. On
a **mid-extract failure** the provider returns the partial stack alongside the error, so the saga boundary unwinds it
**before** any retry — the undo-before-redo guarantee, with no archive-specific logic.

**Resume.** A receipt built by `archive.extract` resumes like any file receipt: it reconstructs as a `*file.Receipt`
from its **compensation** identity (`file.CompensateFileMutation`) even though its dispatch action was `archive.extract`,
so `stack.Unwind()` works after a serialized reload, not only in process.

## 7. Security and resource safety

- **Path traversal (zip-slip).** Every entry's resolved target must remain inside the destination prefix. The current
  guard joins `prefix` with the cleaned entry name and rejects any result that escapes
  `filepath.Clean(prefix)+separator` (`provider.go:222-225`, `:268-271`). `github.com/cyphar/filepath-securejoin`
  (already in the module) is the hardened primitive to adopt here, closing symlink-assisted escapes the prefix check
  alone can miss.
- **Decompression bombs.** Each entry's content is bounded by `io.LimitReader` (today `1<<30` per entry). A
  **total-extraction cap** across all entries is an open question (§10) — a per-entry limit does not bound an archive
  with millions of small entries.
- **Entry-type policy.** The tar walk currently handles only `TypeDir` and `TypeReg`, silently skipping symlinks
  (`TypeSymlink`/`TypeLink`), devices, and FIFOs. Symlink support interacts with recovery (a symlink's "content" is its
  target — recovery must archive the link itself, mirroring `file.unlink`'s caveat); the policy for the remaining
  special types (skip-with-record vs. error) is an open question (§10).
- **Confinement.** All filesystem effects go through `op.Root` / `op.Path`, so extraction cannot write outside the
  authority boundary even if an entry name or symlink attempts it (see [4.4-root-path-triad.md](4.4-root-path-triad.md)).

## 8. Dependencies

- **stdlib:** `archive/tar`, `archive/zip`, `compress/gzip`, `compress/bzip2` (`bzip2` is decompress-only, which is all
  extraction needs).
- **`github.com/ulikunitz/xz`** — pure-Go xz; the established choice in the Go ecosystem.
- **`github.com/klauspost/compress/zstd`** — pure-Go zstd; the de-facto standard, widely depended upon and well
  maintained.

Both new modules are pure Go (no cgo) and additive — they appear only on the extraction path.

## 9. API surface

```go
// Extract extracts the archive at source into the existing directory prefixPath, producing one tracked file.Resource
// per entry and a recovery stack that unwinds the extraction in reverse.
func (p *Provider) Extract(
	activationRecord *op.ActivationRecord, source *file.Resource, prefixPath string,
) (products []*file.Resource, stack *op.RecoveryStack, err error)

// CompensateExtract undoes Extract by unwinding its recovery stack; each entry's file.Receipt self-routes to
// file.CompensateFileMutation.
func (p *Provider) CompensateExtract(stack *op.RecoveryStack) error

// openArchive detects source's format and returns a format-agnostic entry iterator (Layer A + Layer B).
func (p *Provider) openArchive(source *file.Resource) (archiveReader, error)
```

`Extract`/`CompensateExtract` are the dispatchable, Compensable surface; `openArchive` and the decompressor table are
internal mechanism.

## 10. Open questions

1. **Special entry types.** Policy for tar symlinks, hardlinks, devices, and FIFOs — skip-with-record, error, or
   support. Symlinks specifically require recovery to archive the link rather than its referent.
2. **Total-extraction cap.** Whether to bound aggregate extracted bytes / entry count in addition to the per-entry
   `io.LimitReader`, and where that budget is configured.
3. **`filepath-securejoin` adoption.** Replace the hand-rolled prefix check with `securejoin.SecureJoin` for
   symlink-aware containment.
4. **Format-mismatch diagnostics.** A gzip/xz/zstd stream whose payload is not a tar fails at `tar.NewReader`; settle
   the error wording so it names the detected outer format and the missing container.
5. **Streaming zip.** Zip's end-of-file central directory forces random access today; whether a streaming
   (`PK\x03\x04`-walking) reader is ever worth it for forward-only sources.
