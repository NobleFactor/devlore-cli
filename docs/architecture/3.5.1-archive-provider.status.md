## Document Discrepancies (design vs. current code)

- **Detection.** `Extract` dispatches on the lower-cased filename suffix (`provider.go:109-117`); the design specifies
  magic-byte content detection independent of extension.
- **Stale contract.** `Extract`'s doc comment states the format "is detected from the source file's extension"
  (`provider.go:55`) — contradicted by the design; corrected in the rewrite.
- **Receipt shape.** `Extract` pushes `file.NewReceiptWithBoundary(product, destination)` (`provider.go:146`); the
  design routes file entries through `file.WriteFile` and dir entries through the existing `file.Mkdir`, which return
  self-describing receipts naming `file.CompensateFileMutation`.
- **Formats.** Only `tar.gz`/`tgz` and `zip` are implemented; the design commits to the full tar family plus plain
  `tar`.

## Outstanding / open questions

Tracked in the design doc's §10:

1. Policy for tar symlinks, hardlinks, devices, and FIFOs (currently silently skipped).
2. A total-extraction byte/entry cap beyond the per-entry `io.LimitReader`.
3. Adopting `github.com/cyphar/filepath-securejoin` (already in-module) for symlink-aware containment.
4. Format-mismatch diagnostics when a decompressed payload is not a tar.
5. Whether a streaming zip reader is ever warranted.
