## Thesis

Privilege elevation is a **multi-party, just-in-time** mechanism, deliberately split across three owners — and that
split is the whole point: it keeps the graph an immutable, environment-agnostic artifact while the *how* of elevation
lives in the environment, owned by the platform/security team, not the pipeline author.

The **`elevation`** package (op-free) holds the *what* — the policy/contract (`elevation.Policy`, the `Elevator`
interface; see §6). A dedicated **`elevator` provider** holds the *how* — the mechanism for both strategies:
**`ProcessSpawn`** (sudo / runas / UAC privileged worker) and **`IdentityAssumption`** (JIT security-token minting:
AWS STS, Kubernetes ServiceAccounts, Vault, SPIFFE/SPIRE, OIDC). `elevation` is the data/contract; `elevator` is the
provider that implements it — two packages, sharp roles, *not* synonyms.

## The three-way split (graph / configuration / runtime environment)

| Owner | Carries | When | devlore home |
|---|---|---|---|
| **Graph** | the *requirement* — a **named offer** (`offer_reference_id`) + `requested_ttl`, context assertions | design-time; saved into the immutable, signed graph | the plan |
| **Configuration** | the *provisioning* — per-environment realization of named offers → token providers | per-environment (dev / test / stage / prod) | `ElevatorConfig`, a devconfig section on `Application.Config` |
| **RuntimeEnvironment** | the *instantiated orchestrator* + token minting at the elevate node | config resolution + execution | held on `RuntimeEnvironment`, built from `Application.Config` |

**Separation of duties.** Developers author graphs (the *what*); platform/security teams own the runtime config (the
*how*). *"The graph only knows what it needs, not how to get it."* A compiled graph promotes dev → stage → prod with
**zero graph edits** — the same signed artifact, resolved against a different environment's config.

This is the resolution of the open question (per-unit Policy vs offer/request/provision): **adopt the split.** §6's
everything-on-the-unit model holds for `ProcessSpawn` (the OS-escalation case is host-resolved, not credential-bearing),
but for `IdentityAssumption` the unit/graph carries only a **named offer**, and configuration carries the realization.

## The two strategies (modalities)

Both are fulfilled by the `elevator` provider, selected by `elevation.Policy.Strategy`:

- **`ProcessSpawn`** (host escalation) — a persistent **privileged child-process worker** (sudo / runas / UAC) with
  `{Start, Exec, WriteFile, Stop}`; one elevated child per phase, IPC over a pipe. The OS-escalation case
  (`pkg.install` on apt, `NeedsSudo()`). Designed in §6.
- **`IdentityAssumption`** — **JIT token minting**: an `elevate_action` node mints a short-lived `SecurityToken`
  (AWS STS, K8s SA, Vault, SPIFFE/SPIRE, OIDC) bridged by-value to the units that need it. Designed below.
- **`InteractiveChallenge`** / **`MandatedApproval`** — blocking strategies (prompt / await a gatekeeper) that **ride
  the existing pause/resume machinery**: the unit pauses (`ErrPaused`) until granted, then resumes.

## The token bridge — native typed slots

The PDF's `token_bridge` is not new machinery: it is devlore's existing **typed slots / edges**
([`2.1-typed-slots.md`](2.1-typed-slots.md)). A `SecurityToken` minted at an elevate node flows along an edge into the
consumer unit's input slot — **pass-by-value, node-owned, memory-zeroed** after use, with `MaskedVars` injected into the
target node's environment. The security property: **zero shared state** — a crashed parallel branch cannot leak a token
it was never explicitly bridged.

## The token-provider mechanism (`IdentityAssumption`)

Adapted from the elevation-engine design, with the name collision resolved (§6 already has an `ElevationProvider` — the
privileged-process executor; this token orchestrator is a **different type**, namespaced under `elevator`):

```go
// pkg/op/provider/elevator  (working names)

// SecurityToken — the unified, pass-by-value token bridged down graph edges.
type SecurityToken struct {
    Value      string            // raw secret or signed JWT payload
    Mechanism  string            // "BEARER" / "AWS_CREDS" / "KUBECONFIG"
    ExpiresAt  time.Time         // strict clock limit, for fail-fast node evaluation
    MaskedVars map[string]string // env vars to inject into the target node, masked in logs
}
func (t *SecurityToken) IsExpired() bool

// TokenProvider — the pluggable minting-driver contract (aws_sts, k8s, vault, ...).
type TokenProvider interface {
    Configure(properties map[string]any) error
    MintToken(ctx context.Context, ttl time.Duration) (*SecurityToken, error)
    Type() string
}

// The orchestrator (name TBD — NOT "ElevationProvider"; that name is taken by §6's process executor).
//   RegisterDriver(type, factory)        — announce a driver by its type key
//   InitializeFromConfig(...)            — instantiate enabled providers; FAIL FAST at bootstrap on an unknown driver
//   RequestElevation(ctx, name, ttl)     — runtime lookup-by-name + mint, called by the elevate node
```

Drivers register by type key (`aws_sts_assume_role`, `k8s_token_request`, `hashicorp_vault`, …) the same way
providers/resources announce themselves elsewhere in the framework. Concrete drivers (the AWS SDK, etc.) are **decoupled
from graph parsing** — and an unsupported/undefined driver fails at **bootstrap**, long before any node runs.

## The config shape (outline — to be settled)

The full shape is the next design step; this is the **outline** of its three concerns.

**(a) Plan-time — declared on graph nodes, saved into the signed graph (not in config):**

```yaml
# on the consuming unit / the elevate_action node
requirements:
  offer_reference_id: "off_k8s_prod_deployer"      # the NAMED offer (abstract handle)
  requested_ttl_seconds: 900
  required_context_assertions: ["target_environment == 'prod'"]
```

**(b) Environment provisioning — `ElevatorConfig`, a flat section resolved per environment:**

`ElevatorConfig` is **not** environment-keyed itself. It is a flat section — `offers` (a named offer → its realization)
plus `brokers` (the pluggable backends offers reference) — and the config model's **environment overlay axis**
([`configuration.md`](configuration.md) → "Environment overlays") supplies the per-environment values. The graph names an
abstract offer; the section, resolved for the active environment, realizes it.

```yaml
# defaults: -> elevator   (the base; per-environment variants live under defaults: -> environments: -> <env> -> elevator)
elevator:
  offers:                          # a named offer -> its realization
    off_k8s_prod_deployer:
      strategy: identity_assumption
      broker: aws_sts              # references a broker below (empty for host_escalation)
  brokers:                         # the pluggable backends -- NOT raw secrets
    sudo:
      type: host_escalation
    aws_sts:
      type: aws_sts_assume_role
      is_enabled: true
      connection_timeout_seconds: 10
      config_properties:
        role_arn: "arn:aws:iam::123456789012:role/PipelineDeployer"
        region: us-west-2
```

So: **plan-time** specs named offers and saves them in the graph; **run-time** reads the elevator section **already
resolved for the active environment** (dev / test / stage / prod) — dev supplies a mock broker, prod the real one — and
looks up the named offer's realization; and because offers reference **brokers** (`<package>.Broker`, the settled role
name), the section configures the brokers its offers use. (`host_escalation` / `ProcessSpawn` offers resolve to the
privileged-worker broker — no token minted.)

## Secretless, signed, tamper-evident

The graph artifact is **signed** (provenance/integrity; see [`graph-signing.md`](../plans/extract-starlark-from-op/phase-8/graph-signing.md))
and **secretless** — it holds only the abstract `offer_reference_id`. `ElevatorConfig` holds provider **definitions**
(role ARNs, vault addresses, mount paths) — *not* raw long-lived credentials; the actual credentials come from the
host's credential chain (the AWS default chain, `$VAULT_TOKEN`). "Secretless" means **no long-lived creds in the
artifact**, not "no provider config."

## Execution flow

1. **Construction (plan).** Declare offer intent via `offer_reference_id`; environment-agnostic; saved in the graph.
2. **Resolution / bootstrap.** Build the orchestrator from `ElevatorConfig`; **validate** — fail fast on an unsupported
   driver type.
3. **Preflight.** Cross-check **every** `offer_reference_id` in the graph against the resolved config for the active
   environment; any unresolved reference **refuses the run** before a node fires.
4. **Runtime node execution.** At the `elevate_action` node, look up the named offer for the active environment and mint
   the token (`IdentityAssumption`) or acquire the privileged context (`ProcessSpawn`).
5. **Token bridge.** Pass the token by-value down the edges into consumer input slots; consumers zero the token's memory
   on completion. The receipt records acquisition + release (the Do/Undo bracket — preserved from `flow.elevate`).

## Failure routing

Three distinct failures at three distinct times — the undefined-reference case never reaches execution:

1. **Undefined / unsupported reference → preflight refusal** (static / config error). Caught at graph-load/preflight by
   cross-checking references against the resolved config; the graph *cannot run here*, refused deterministically — never
   a mid-run surprise.
2. **Reachable-but-mint-fails → node `flow.Failed`** (runtime / dynamic). `RequestElevation` errors → the node fails →
   the unit's **`ErrorAction`** (the `flow.failed` floor) handles it; the branch **halts**, so downstream consumers
   never run unauthenticated.
3. **Upstream-touched + uncompensable → `flow.Degraded` / stranded.** Governed by the
   [compensation-failure contract](../plans/extract-starlark-from-op/phase-8/compensation-failure-contract.md); the
   `RecoveryStack` unwinds what it can, and what it can't becomes the explicit degraded outcome rather than a silent
   half-run.

These dynamic cases **reuse `ErrorAction` + `RecoveryStack`** rather than inventing new failure plumbing.

## The provider move

The mechanism leaves `flow`. Today the only elevation surface is the empty no-op `flow.Provider.Elevate()`
(`pkg/op/provider/flow/provider.go:507`). It moves into the `elevator` provider, which **preserves the Do/Undo saga
bracket** §6 specifies — `Start` is `Do` (acquire the elevated context / mint), `Stop` / token-release is `Undo` — so
privilege acquisition stays visible and releasable in the plan and the receipt. (§6's reference to
`internal/execution/flow/elevate.go` is stale.)

## Audit

Each granted offer — a token minted, a privileged session entered — is an audit event, threaded through the
receipt / `RecoveryStack` audit trail (a token's JTI is its handle).

## Prior art

This is the **credential-broker pattern** (SPIFFE/SPIRE + Vault + cloud OIDC federation) realized as a first-class graph
primitive — recent work names the shape directly ("Credential Broker Patterns for Secure CI/CD", arXiv 2504.14761). It
fuses three established lineages:

- **The broker core — Vault + SPIFFE/SPIRE.** Vault's *dynamic secrets* (mint-per-request, lease + TTL, pluggable secret
  *engines*) map onto the `TokenProvider` drivers + `requested_ttl`; SPIFFE/SPIRE's short-lived, auto-rotated,
  never-on-disk SVIDs (OIDC-exchanged for cloud tokens) are the `SecurityToken`. Vault 1.21 added SPIFFE auth — the two
  are converging.
- **The workflow shape — GitHub Actions OIDC / Airflow Connections / Concourse `((vars))`.** GitHub's secretless
  OIDC→STS (`AssumeRoleWithWebIdentity`, short-lived creds, no static keys) is the closest pipeline analog; Airflow
  **Connections** (named references resolved at runtime *on the worker*, never in the control plane, env-specific) is the
  `offer_reference_id` → env-config resolution and the separation of duties.
- **The posture — JIT PAM (Teleport / Boundary / StrongDM).** Zero standing privilege, approval workflows (Teleport's
  Access Requests ≈ `MandatedApproval`), and tamper-evident session audit.

The OS modality (`ProcessSpawn`) is the less-novel leg — Ansible `become` / config-management sudo.

**What none of them do this way.** All of the above are *external* brokers a pipeline calls out to. Here the broker is a
**graph primitive**: the elevate node mints, the token **bridges by-value along typed edges** to consumers (node-owned,
memory-zeroed, zero shared state), and it is **compensable** (the Do/Undo bracket, `RecoveryStack` revocation,
`flow.Degraded`/stranded on failure). Token-as-graph-dataflow + saga compensation is the synthesis the prior art does
not combine.

**Lessons taken:**

- **Don't reinvent authorization** (OIDC-CI) — the real grant decision lives in the provider's trust policy
  (claims → role); `ElevatorConfig` only *selects* the offer. Map graph claims (`graph_id`, environment, node) to that
  policy.
- **Model leases, not just TTL** (Vault) — track active leases so the token is **revoked on compensation/undo**; favor
  dynamic (unique-per-request) creds.
- **Resolve only on the execution side** (Airflow / Concourse) — mint on the runtime environment, never at
  plan/control time; the control plane never sees a credential.
- **Tamper-evident audit** (Teleport) — who / what / when / which-offer / JTI into the *signed receipt*, not a log line;
  keep the elevated window minimal (`Ephemeral`).
- **Fail closed; never log tokens** — the broker is the high-value target (preflight refusal aligns; `MaskedVars`).

**Competitive positioning.** We *consume* the brokers as drivers (Vault, SPIRE, cloud STS/OIDC), not compete with them.
The closest direct commercial competitor to the `IdentityAssumption` modality is **Aembit** (workload identity
federation for CI/CD). We compete with the *orchestrators* that bolt elevation onto pipelines (Ansible+Vault,
Argo/Tekton+SPIFFE, Spacelift/env0, GitHub Actions OIDC) — our wedge is elevation as a *compensable, signed graph
primitive with by-value token bridging*, which none of them treat as intrinsic.

## Design discussion: the broker registry as a provider trait

> **Design discussion item — not a settled design.** This generalizes beyond elevation; it is captured here because the
> elevator is the case study that surfaced it. To be worked through and given a permanent home (likely a provider-model
> doc) before it is built.

The elevator is not special in needing a *set of pluggable backends*. The package-install provider already works this
way — `pkg.Provider` installs `pkg.Resource` instances by delegating to one-or-more platform backends (apt, dnf, brew,
port) — it simply is not coded with a named `Broker` role yet. The proposal is to make this a standard, reusable
provider capability:

- **A broker registry as an embeddable trait.** A provider that hosts backends embeds a registry trait (sketch:
  `op.BrokerRegistry`) alongside `op.ProviderBase`. It is opt-in — only brokered providers embed it.
- **Ask a provider for its brokers.** The trait exposes the live set (`Brokers() []string`, `Broker(name)`), so a
  provider's backends are introspectable rather than hard-coded.
- **Each broker announces a typed config slot.** When a broker registers, it contributes both its factory and the shape
  of its configuration. The provider's `Config` is then the *aggregate* of its brokers' slots — not a fixed struct — and
  the config system routes each `…brokers.<name>` sub-document to the right typed slot.
- **Extensibility is the payoff.** A new backend — an authentication service, a different software installer — drops in
  as a broker plus a broker-config instance, with no change to the provider core or the config schema.

Questions to resolve when this is taken up: the trait's typing (a generic `BrokerRegistry[Broker]` vs a marker-interface
registry); how broker config slots are validated and defaulted; and how this relates to the existing `+devlore`
action-announcement machinery (brokers announce to the *config* system — a second channel).
