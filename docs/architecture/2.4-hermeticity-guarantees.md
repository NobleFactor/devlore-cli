# 2.4 Hermeticity Guarantees

**Status**: Problem framing — design not started

## Objective

Graph planning must be hermetic: given the same inputs, it produces the same graph, on any machine, every time. Secondary artifacts (files and configuration that travel with the code) are **inputs**, not external resources. The challenge is formalizing this boundary.

## Problem

Planning currently reads secondary artifacts through ambient filesystem access (`RootReader` delegates to live `os.*`). This violates hermeticity in two ways:

1. **No input boundary** — the planner can read anything reachable from the filesystem, not just declared inputs.
2. **No temporal guarantee** — files can change between reads during a single plan operation.

### Scoping Complexity

The input boundary differs between lore and writ, and the two interact.

#### Lore Package Scope (Solved by Construction)

Lore package planning is hermetic by construction. Each package is scoped to its cache location (`${XDG_CACHE_HOME}/devlore/`), carrying its own secondary artifacts (Starlark scripts, installation scripts, manifests). The cache location is deterministic given a package identity and version. The package plans against its cache contents and nothing else — the hermeticity boundary is the package itself.

Execution scope is a separate concern. A package installation may target:
- System paths (`/`) — most package managers (apt, dnf, port, winget) require elevation
- User paths (`$HOME`) — brew and user-level configuration
- Arbitrary configuration paths — packages inevitably tweak config files across the filesystem

The execution scope does not affect planning hermeticity. The plan is sealed against the cache; where it writes is an execution-time decision.

#### Writ Layer Scope

Writ has three layers, processed in override order (later layers win collisions):

| Layer | Purpose |
|---|---|
| **base** | Foundational configuration shared across all environments |
| **team** | Shared team configuration |
| **personal** | User-specific configuration |

Each layer's physical location can be anywhere on disk. The common reference point is `${XDG_DATA_HOME}/devlore/writ/layers/{layer}/` — each slot is either a symlink to the actual repository or a directory at that path. This gives one well-known root that links to all layer sources.

Within each layer, there are **two target scopes**:

| Source subdirectory | Target root | Purpose |
|---|---|---|
| `System/` | `/` | System-wide files (requires elevation) |
| `Home/` | `$HOME` | User home files |

#### Planning: One Tree, Six Cells

Planning builds a **single target tree** that can be queried for elements originating from any cell of the 2×3 matrix. The tree is populated by traversing the matrix in a deterministic order that ensures correct shadowing:

```
1. system/base
2. home/base
3. system/team
4. home/team
5. system/personal
6. home/personal
```

The shadowing rules:
- **Home shadows System** within the same layer (for overlapping target paths)
- **Team shadows Base**, **Personal shadows Team and Base** (layer precedence)

The traversal order guarantees that later entries override earlier ones, so personal always wins over team, team always wins over base, and home always wins over system within the same layer. Collision detection operates on this single tree with full cross-cell visibility.

#### Execution: Graph Count

From the single planning tree, writ extracts execution graphs — one per populated target scope:

**System: always 1 graph** (when system files exist). All system files (from base, team, and personal layers) target `/`. The graph is unconfined and elevated as required.

**Home: always 1 graph** (when home files exist). All home files target `$HOME`. The graph is confined to `$HOME` via `os.Root`.

| Scope | Target Root | Confinement | Elevation | Graph Count |
|---|---|---|---|---|
| System | `/` | Unconfined | Required | 0 or 1 |
| Home | `$HOME` | Confined (`os.Root`) | As needed | 0 or 1 |

Total execution graphs: **1 to 2** in practice. Each graph gets its own `Context`, `Root`, `RecoverySite`, and executor instance. Execution order is deterministic: system first, then home.

#### Confinement and Source Reachability

An executor confined to `$HOME` can read sources within `$HOME` but not outside it. For layer repos outside `$HOME`:

- **Symlinks work**: `file.link` creates a symlink file within `$HOME` pointing to the source path. The target path is just a string — `os.Symlink` does not read the source. No confinement violation.
- **Copy/template/decrypt fail**: these operations must read source file content, which is outside the confined root.

Rather than splitting Home into multiple graphs with different confinement boundaries, this is solved by **git worktree snapshots** (Phase 6): snapshots are placed within `$HOME` (e.g., `$HOME/.devlore/snapshots/`), rewriting `LayerSource.SourceRoot` to point within the confinement boundary. All sources become reachable, all operations work, and the 1-graph-per-scope model holds.

#### Layer Hermeticity Within Each Cell

Within each cell, the tree walk is self-contained — scoped to its own `SourceRoot` directory. Each layer's tree walk reads only from that layer's source directory. Collision resolution across cells is a metadata comparison (path + layer order + specificity) — no cell reads file contents from another cell.

Planning hermeticity for writ reduces to: each layer source directory is an input, and the tree walk reads only within that directory.

#### The Interaction with Lore

Writ discovers `packages-manifest.yaml` files within its layer sources and delegates to lore for package resolution. Since lore planning is already hermetic (scoped to package cache), the interaction does not compromise hermeticity. The manifest file is within the layer's source scope; the resolved packages plan against their own cache.

## Strategy: Git-Native Snapshots

Writ layer sources are git repositories. This is a foundational assumption — layers are versioned, shareable configuration that travels as code. Git provides content-addressed, immutable snapshots for free.

### Mechanism

At planning time, each layer is pinned to a specific commit hash. The planner reads from a git worktree detached at that hash, guaranteeing immutability during the planning pass.

```
base     → git worktree @ abc123
team     → git worktree @ def456
personal → git worktree @ 789fed
```

### Properties

This gives us every hermeticity property without custom snapshot infrastructure:

- **Immutability** — the worktree is pinned to a commit; files cannot change underneath the planner
- **Content addressing** — a commit hash is an immutable identity for the entire tree; same hash = same content, guaranteed
- **Reproducibility** — anyone with the repo can recreate the exact input set from the hash
- **Plan caching** — the tuple `(base_hash, team_hash, personal_hash, target_scope)` is a cache key for the graph; same tuple = same graph, skip planning
- **Auditability** — the receipt records the commit hashes that produced the graph; the provenance chain is complete

### Temporal Guarantee

The original problem identified two hermeticity violations. Git worktrees resolve both:

1. **Input boundary** — the planner reads from a detached worktree, not the live working directory. Uncommitted changes, staged files, and untracked files are invisible.
2. **Temporal guarantee** — the worktree is pinned to a single commit. No file can change between reads during planning.

### Worktree Lifecycle

The planner creates lightweight worktrees at the start of each planning pass and removes them after the graph is built. This is a fast operation — git worktrees are cheap (they share the object store with the main repo).

An alternative is to read the HEAD commit of each layer at planning time and trust that planning is fast enough that nothing changes. This avoids worktree overhead but has a theoretical temporal gap. The worktree approach is airtight.

## Feasibility: Multi-Scope with Separate Contexts

Code review confirms that the separate-graph approach requires **no changes below the command layer**. The execution engine, `Context`, `Root`, `RecoverySite`, providers, and graph structure all work unchanged when each cell of the 2×3 matrix gets its own graph.

### Why It Works

Each scope produces an independent graph with its own executor, context, and confined root. The components that appear to assume "single scope" are in fact scoped per-graph, which is exactly what we need:

| Component | Current Design | Multi-Scope Impact |
|---|---|---|
| `Context.Root` | One `Root` per context | Each graph gets its own context with its own root — no conflict |
| `RecoverySite` | Holds one `ctx`, writes to `.devlore/recovery/` within that root | Each graph's recovery is isolated within its own root (`/.devlore/recovery/` vs `$HOME/.devlore/recovery/`) |
| `GraphContext.TargetRoot` | Single string | Each graph records its own target — `/` or `$HOME` |
| Node IDs | Unscoped strings (relative target path) | No collision — same relative path in different graphs is fine |
| `ExecutorOptions.Root` | Single string | Create a separate executor per scope, or call `Run()` with different options |
| Provider hydration | All providers initialized with one `ctx` | Each graph hydrates against its own context |
| `ResourceCatalog` | One per graph | Each graph has its own catalog — no URI collision |

### Command-Layer Changes Required

The orchestration logic in `internal/writ/commands.go` needs four changes:

1. **Build the planning tree from all 6 cells.** `CollectLayerSources()` already returns all cells as a flat `[]LayerSource`. The tree builder populates a single tree with full cross-cell visibility for shadowing and collision detection.

2. **Extract and execute graphs by scope.** Currently `runDeployV2()` builds one graph and creates one executor. Change to: extract 1 system graph and 1 home graph from the unified tree. Execute in deterministic order (system first, then home), each with its own `ConfigureEngine()` call. Source reachability for confined home execution is solved by git worktree snapshots placed within `$HOME`.

3. **Cell-tagged receipts.** Receipt filenames (`writ-latest.yaml`) don't distinguish cells. Either tag receipts per cell (e.g., `writ-system-base-latest.yaml`) or embed the cell identity in the receipt's `GraphContext`. Without this, later receipts overwrite earlier ones.

4. **Cell-aware state view.** `StateViewBuilder` loads all receipts indiscriminately. If receipts are cell-tagged, the state view can filter by cell. Without this, decommission mixes state across cells.

## Resolved Decisions

- **Lore hermeticity**: solved by construction — package plans against its cache, execution scope is separate.
- **One tree, two graphs**: planning builds a single tree with full cross-cell visibility (2×3 matrix). Execution extracts 1 system graph (unconfined, elevated) and 1 home graph (confined to `$HOME`). Source reachability under confinement is solved by git worktree snapshots within `$HOME`, not by graph splitting.
- **Collision detection**: metadata-only comparison across all cells in the single planning tree — does not break per-cell hermeticity.
- **Snapshot mechanism**: git commit hashes as content-addressed input identities. Worktrees pinned to specific commits provide immutable planning inputs.
- **Execution engine compatibility**: no changes needed below the command layer. `Context`, `Root`, `RecoverySite`, providers, and graph structure all support multi-scope by construction.

### Failure Coordination and Compensation

The current policy is **fail-forward**: if one scope's graph fails, the other scope continues. Each scope is independent — system and home target different roots with different confinement boundaries. The `RecoverySite` compensates within a single graph; cross-graph compensation is not modeled. A failed system graph does not block home execution.

### Coupled Design Decisions

Confinement reachability, receipt identity, and failure coordination must be designed together — they are coupled:

- A failure policy that compensates across graphs depends on receipts knowing which graph they belong to.
- Receipt identity depends on cell tagging.
- Cell tagging depends on how many graphs were produced, which is driven by confinement reachability.

These three cannot be designed in isolation.

### Combinatorial Test Surface

The 2×3 matrix with variable graph counts (2–4) creates a combinatorial test surface. Fixtures are needed for all reachability scenarios:

- All layer repos within `$HOME` (1 home graph)
- One layer repo outside `$HOME` (2 home graphs)
- All layer repos outside `$HOME` (3 home graphs)
- Layers with no System content (0 system graphs)

### Failure Coordination and Compensation

If the system graph fails, what happens to home graphs that haven't started? The `RecoverySite` compensates within a single graph, but cross-graph compensation is not currently modeled. The failure policy needs to answer: does a failed system graph block home execution, and if home graphs have already completed, should they be rolled back?
