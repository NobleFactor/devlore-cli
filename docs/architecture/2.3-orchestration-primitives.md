# Orchestration Primitives

Flow actions — Gather, Choose, WaitUntil, Complete, Degraded, Fatal, Elevate —
are visible runtime operations, not engine internals. Each appears as one or
more nodes in the graph, is recorded in dry-run output, and preserved in the
receipt. The executor does not distinguish between "resource actions" and "flow
actions" at the dispatch level — both implement the Action interface and receive
resolved slots.

## Design Principle

**The graph is self-contained.** No hidden runtime queries. Every decision the
executor makes is rooted in a graph node whose inputs and outputs are traceable.
The plan shows what will happen and why. The receipt shows what happened and why.

| Runtime behavior | Flow action |
|---|---|
| Iterate over items | **Gather** |
| Select among alternatives | **Choose** |
| Wait for a condition | **WaitUntil** |
| Signal success | **Complete** |
| Signal partial success | **Degraded** |
| Halt execution | **Fatal** |
| Privilege transition | **Elevate** (stub) |

## Subgraph Execution & Recovery-Stack Ownership (current model — 2026-06-20)

> **This section is the current, authoritative model.** The Phase-based prose
> below it (`Phase as Universal Boundary`, `ExecutePhaseInner`, `Graph.Phases`,
> `internal/execution/*`) describes the superseded pre-`pkg/op` design and is
> retained only as history pending a full rewrite of this document.

**Every subgraph executes via its own executor, and that executor owns the
subgraph's recovery stack** — together with the subgraph's variable scope, pause
state, trace, and catalog scope. There is one rule, applied recursively down the
subgraph tree.

- **There are no special "combinators."** `Gather`, `Choose`, and `Subgraph` are
  not a distinct kind of thing — each *is* a subgraph, and each gets its own
  executor. `Gather` calls `Subgraph` once per item: N iterations, each running
  the body under its own executor with its own stack (created in the iteration's
  goroutine — never shared, so no race). Gather returns those as a **slice of
  stacks** (`[]*op.RecoveryStack`, one per iteration); `CompensateGather` undoes
  the slice. A "per-iteration saga boundary" is not a bespoke API; it is just
  "each iteration runs `Subgraph`," the universal rule applied N times.

- **`Choose` does NOT select a branch.** `ChoosePlanner` constructs `Choose` as
  a single subgraph with the branches built **into the graph** at plan time. At
  runtime the **graph** selects which branch executes; `Choose` runs that one
  planner-built subgraph (via `flow.Provider.Subgraph`) and merely **receives the
  result**. There is no runtime case-selection in `Choose` — it is a `Subgraph`
  with plan-time branch structure, not a runtime chooser.

- **A flow action's `Do()` must never mint its own recovery stack — but it
  keeps its compensate companion.** Today the combinator's `Do()` calls
  `op.NewRecoveryStack()`; that *minting* is the deviation. The per-subgraph
  executor owns and creates the stack instead. The combinator's forward action
  still returns that stack as its **complement**, and its **compensate companion
  still undoes it** — every combinator keeps both. `Subgraph`, `Choose`, and
  `WaitUntil` return a single `*op.RecoveryStack` (undone by
  `CompensateSubgraph` / `CompensateChoose` / `CompensateWaitUntil`); `Gather`
  returns a `[]*op.RecoveryStack` slice (undone by `CompensateGather`, which
  unwinds each iteration's stack). The fix moves *who creates the stack* (the
  executor, not `Do()`), not *whether a companion exists* — it always does. The
  full action/compensation signature table lives in the
  [step-28 plan doc](../plans/extract-starlark-from-op/phase-8/steps/28-subgraph-executor-ownership.md#combinator-signatures-confirmed-in-review--2026-06-20).

- **Stack count is determined, not chosen.** "One stack or many?" for `Gather`
  is settled — *many*, one per dispatch — and that count falls out of "subgraph
  = executor owns a stack" applied to N body dispatches. It is never a
  `Gather`-level decision. The open defect is *ownership* (the per-dispatch
  executor, not the combinator's `Do()`), not the count.

- **The per-subgraph stacks chain — up for resolution, down for unwind.** The
  stacks form one tree linked two ways: a child stack points **up** to its
  parent, and `ResultByUnitID` walks that chain up so a promise resolves against
  whatever ancestor stack holds the producer's receipt; and a child stack is
  nested **down** into its parent (`PushNested`) so `Unwind` cascades
  compensation through the tree. The chain *is* the nested tree, so a `Trace`
  carries it: **the nesting is durable (serialized), the parent pointer is
  transient (re-derived on load)** — save serializes the tree, load rebuilds it
  and re-derives the chain, which is what makes resume's promise resolution and
  skip-completed work. (Detail in the
  [step-28 plan doc](../plans/extract-starlark-from-op/phase-8/steps/28-subgraph-executor-ownership.md#chained-recovery-stacks--up-for-resolution-down-for-unwind).)

- **The saga boundary is per-subgraph and retry-gated.** Each subgraph executor
  is a saga boundary, and the boundary is *maintained* — rollback is a
  per-boundary unwind that propagates outward, never a single root-level sweep.
  A boundary respects its retry policy: on failure it exhausts its retries (one
  attempt when none; all N when the retry count is N) **before** rollback
  continues up the stack. Rollback continuing up the stack means the failure
  reaches the next outer saga boundary, which applies *its own* retry policy
  before unwinding its own stack and propagating further. Each executor unwinds
  its own stack recursively. Because the boundary is atomic, each failed attempt
  first unwinds to the boundary's entry precondition before the next retry —
  re-running completed children without undoing them would double-apply
  non-idempotent work, so there is no no-undo retry.

- **Retry policy is a tri-state with per-type defaults.** A unit's retry policy
  is *no retry* (explicit `MaxAttempts:0`, fail immediately), *default retry*
  (nil — resolves to the graph's default policy), or *non-default retry* (an
  explicit count). The unset-default differs by unit kind: a **node** (a leaf
  provider call) carries an explicit `MaxAttempts:0` — fail-fast, the boundary
  decides; a **subgraph** (the saga boundary) carries nil and inherits the
  **graph default retry policy — `MaxAttempts:3`, exponential backoff, with
  jitter** — so the retry-gated rollback above is the boundary's default
  behavior rather than inert. (As of 2026-06-20 the code does not yet encode
  this — nil is conflated with no-retry and `RetryPolicy` has no jitter;
  realizing it is phase-8 step 32.)

**Why this is the model.** With per-subgraph executors owning their stacks,
pause/resume, skip-already-completed, and catalog capture/restore become one
recursive rule — the serialized `Trace` is a tree of sub-traces mirroring the
subgraph tree — instead of per-combinator special handling. Resume is impossible
while a shared executor's combinators re-mint empty stacks on every dispatch.

**Current deviation (the work).** Today there is a single shared
`op.GraphExecutor`; `subgraph.Execute` hands that same executor to children via
the `dispatchChild` closure while the `flow` combinators
(`pkg/op/provider/flow/provider.go`) hand-roll their own stacks
(`op.NewRecoveryStack()` in `Subgraph`, per-iteration `iterStack` in `Gather`,
an empty stack in `Choose`). Making subgraph dispatch go through a per-subgraph
executor that owns its stack is the prerequisite for resume — tracked in the
phase-8 plan, step 28.

## Vocabulary

The execution model uses three layers, each with distinct lifecycle and
mutability characteristics.

### Three-Layer Model

| Layer | Types | Lifecycle |
|---|---|---|
| **Definition** | Node, Phase, Edge, SlotValue | Plan-time, immutable during execution |
| **Activation** | ActivationState | Per-execution, mutable, transient — discarded after results and complement are captured |
| **Recovery** | Complement, RecoveryEntry, RecoveryStack | Durable until rollback completes or graph succeeds |

**Definition layer.** The graph structure — nodes, phases, edges, and slot
values — is built at plan time by Starlark scripts and the graph builder. Once
execution begins, the definition layer is immutable. Nodes are never cloned or
mutated during execution.

**Activation layer.** Per-execution mutable state that tracks node progress
during a single execution pass. For non-gather execution, activation state is
inlined on the Node struct (the existing `Status`, `Timestamp`, `Error` fields).
For gather's concurrent execution, ActivationState lives in a per-iteration
map — nodes are shared and never mutated.

```go
// internal/execution/activation.go

type ActivationState struct {
    Status    op.NodeStatus
    Timestamp string
    Error     string
}
```

**Recovery layer.** State that persists beyond a single execution pass for the
purpose of rollback. Complement is captured by each action's `Do` method and
stored in RecoveryEntry on the RecoveryStack. Recovery state is durable — it
survives until rollback completes or the graph succeeds.

## Phase as Universal Boundary

There is no Subtree type. **Phase is the universal boundary** — a single type
used for saga steps, compensation, gather bodies, and choose branches.
Distinguished by how they're referenced, not by structure.

| Usage | Referenced by | Executed by |
|---|---|---|
| Saga step | `Graph.Phases` order | `RunPhased` forward walk |
| Compensation | Another phase's `Compensate` field | `RunPhased` unwind |
| Gather body | Gather node's `do` slot (phase ID) | Gather's per-item loop |
| Choose branch | Choose node's `then`/`else` slots (phase IDs) | Choose after condition eval |

All four usages execute via `ExecutePhaseInner`. All four get retry policy.

Gather-body and choose-branch phases appear in `Graph.Phases` but are skipped
during the forward saga walk — same pattern as compensating phases. The executor
identifies them by checking whether they're referenced by a gather or choose
node rather than by position in the phase list.

### Phase Execution Contract

Regardless of usage, every phase execution follows the same contract:

1. Collect the phase's nodes from the graph (via `Phase.NodeIDs`)
2. Order nodes topologically within the phase
3. Execute nodes sequentially, resolving slots from the results map
4. Push each completed node onto the recovery stack
5. On failure, unwind the phase's recovery stack entries

### Phase Structure

```go
// pkg/op/phase.go

type Phase struct {
    ID         string         `json:"id" yaml:"id"`
    Name       string         `json:"name" yaml:"name"`
    Status     PhaseStatus    `json:"status" yaml:"status"`
    Retry      *RetryPolicy   `json:"retry,omitempty" yaml:"retry,omitempty"`
    NodeIDs    []string       `json:"nodes,omitempty" yaml:"nodes,omitempty"`
    Compensate string         `json:"compensate,omitempty" yaml:"compensate,omitempty"`
    Attempts   []Attempt      `json:"attempts,omitempty" yaml:"attempts,omitempty"`
    State      map[string]any `json:"state,omitempty" yaml:"state,omitempty"`
}
```

## SlotProxy

SlotProxy is the third SlotValue variant, enabling N concurrent instances with
isolated slot resolution. It is the mechanism by which gather body phases
reference per-iteration values without mutating shared nodes.

### Go Types

```go
// pkg/op/graph.go

type SlotValue struct {
    Immediate any    `json:"immediate,omitempty" yaml:"immediate,omitempty"`
    NodeRef   string `json:"node_ref,omitempty"  yaml:"node_ref,omitempty"`
    Slot      string `json:"slot,omitempty"      yaml:"slot,omitempty"`
    GatherRef string `json:"gather_ref,omitempty" yaml:"gather_ref,omitempty"`
    Field     string `json:"field,omitempty"      yaml:"field,omitempty"`
}

func (s SlotValue) IsImmediate() bool { return !s.IsPromise() && !s.IsProxy() }
func (s SlotValue) IsPromise() bool   { return s.NodeRef != "" }
func (s SlotValue) IsProxy() bool     { return s.GatherRef != "" }
```

Three variants, mutually exclusive:

| Variant | Fields set | Resolution |
|---|---|---|
| Immediate | `Immediate` | Value used directly |
| Promise | `NodeRef` (+ optional `Slot`) | Resolved from upstream node's Result at execution time |
| Proxy | `GatherRef` + `Field` | Resolved from per-iteration proxy context at gather execution time |

### Plan-Time Recording

The `do` lambda receives a proxy object (`starlark.HasAttrs`). Field access
records the path and returns a `FieldRef` marker. The planned receiver stores it
as `SlotValue{GatherRef, Field}`. The lambda builds nodes — it does not compute
values.

```python
plan.gather(
    items = servers,
    do = lambda item: plan.sequence(item, [
        plan.service.stop(item.name),     # → {gather_ref: "g1", field: "name"}
        plan.service.start(item.name),
    ]),
    policy = plan.concurrency(limit=10)
)
```

When `item.name` is evaluated, the proxy object records the access path and
returns a `FieldRef` marker. The planned receiver creates the node with:

```go
node.SetSlotProxy("name", gatherID, "name")
```

### Execution-Time Resolution

`ResolvedSlots` accepts an optional proxy context parameter. The proxy context
maps gather IDs to the current iteration's item value:

```go
func (n *Node) ResolvedSlots(results map[string]any, proxyCtx ...map[string]any) map[string]any {
    slots := make(map[string]any, len(n.Slots))
    for name, sv := range n.Slots {
        switch {
        case sv.IsProxy():
            if len(proxyCtx) > 0 && proxyCtx[0] != nil {
                if item, ok := proxyCtx[0][sv.GatherRef]; ok {
                    slots[name] = fieldAccess(item, sv.Field)
                }
            }
        case sv.IsPromise():
            if results != nil {
                if val, ok := results[sv.NodeRef]; ok {
                    slots[name] = val
                }
            }
        case sv.IsImmediate():
            slots[name] = sv.Immediate
        }
    }
    return slots
}
```

For non-gather execution, `proxyCtx` is not passed and proxy slots resolve to
nil (a validation error if encountered). This ensures proxy slots only appear
in gather body phases.

### Serialized Form

Proxy slots serialize naturally to YAML/JSON:

```yaml
- id: stop-svc
  action: service.stop
  slots:
    name: {gather_ref: patch-servers, field: name}
```

## Flow Actions

### Gather — Parallel Comprehension

Gather is a parallel comprehension — an iterator that executes a phase body
once per item, with configurable concurrency.

**Slots:**

| Slot | Type | Description |
|---|---|---|
| `items` | `[]any` | List of items to iterate |
| `do` | `string` | Phase ID — the body to execute per item |
| `limit` | `int` | Concurrency limit (default 1 = sequential) |

**Execution:**

1. Resolve `items` slot (immediate or promise)
2. Look up the `do` phase by ID from `ctx.Graph`
3. Execute the phase once per item, with concurrency bounded by `limit`
4. Each iteration gets isolated state: proxy context, results map, recovery stack
5. Collect results from terminal node of each iteration
6. Return collected results and `gatherComplement` for undo

**Concurrent execution** uses a semaphore channel and `sync.WaitGroup`. Each
goroutine has its own `ActivationState` map, `nodeResults` map, `RecoveryStack`,
and `proxyCtx`. Shared node objects are read-only — never cloned, never mutated.

```go
// internal/execution/flow/gather.go

type gatherComplement struct {
    Iterations []iterationUndo
}

type iterationUndo struct {
    ProxyCtx map[string]any
    Results  map[string]any
    Stack    *op.RecoveryStack
}
```

**Undo** walks iterations in reverse, unwinding each iteration's recovery stack
in LIFO order.

### Choose — Conditional Branching

> **Superseded — `Choose` does NOT select.** The branches are built into the graph by `ChoosePlanner` at plan time; the
> graph selects the branch at runtime and `Choose` only receives the result. See the current-model section above. The
> runtime "evaluates a condition and executes one of two branches" description below is historical.

Choose evaluates a boolean condition and executes one of two branch phases.

**Slots:**

| Slot | Type | Description |
|---|---|---|
| `when` | `bool` | Condition (resolved from immediate or promise) |
| `then` | `string` | Phase ID to execute if true |
| `else` | `string` | Phase ID to execute if false (optional) |

**Execution:**

1. Resolve the `when` slot to a boolean
2. Select the `then` or `else` phase ID
3. If the selected phase ID is empty, return nil (no-op)
4. Execute the selected phase's nodes in order
5. Return the terminal node's Result as the choose node's output
6. Downstream nodes reference the choose node via promise slots

```go
// internal/execution/flow/choose.go

type chooseComplement struct {
    Results map[string]any
    Stack   *op.RecoveryStack
}
```

**Undo** delegates to the selected branch's recovery stack. Only the executed
branch has recovery entries — the unmatched branch was never executed and has
nothing to undo.

### WaitUntil — Polling Sensor

WaitUntil is an event-driven sensor — a synchronization primitive that pauses
execution until an external condition is met.

**Slots:**

| Slot | Type | Description |
|---|---|---|
| `target` | Promise (node ref) | The node whose Result is polled |
| `predicate` | `PredicateFunc` | `func(any) (bool, error)` — condition to evaluate |
| `timeout` | `string` | Maximum wait time (e.g., `"5m"`) |
| `interval` | `string` | Poll interval (e.g., `"10s"`, default `"5s"`) |

**Execution:**

1. Resolve the `target` slot (promise)
2. Start a polling loop with the configured `interval`
3. On each tick, evaluate the `predicate` against the target's current value
4. If the predicate returns true, the node completes with the target value
5. If `timeout` expires, the node fails with a timeout error

WaitUntil is not compensable — it observes state but does not modify it.

### Complete — Terminal Success

Complete signals that the graph (or a branch) has reached successful completion.
Returns the `output` slot value as its Result.

```go
// internal/execution/flow/complete.go
func (a *Complete) Do(_ *op.Context, slots map[string]any) (op.Result, op.Complement, error) {
    return slots["output"], nil, nil
}
```

### Degraded — Terminal with Warning

Degraded signals non-optimal completion. Formats a warning template from
variadic args/kwargs and writes it to stderr. The graph continues but the
receipt records the degraded state.

Placed inside a unit's **error action** (a `*Subgraph` the executor **MUST** run
on failure), a `flow.Degraded` node transitions the run to the `Degraded`
terminal and **continues execution** — successes kept, failure recorded, no
unwind. This is `$ErrorActionPreference = 'Continue'` semantics: dependents are
not pre-pruned; a downstream unit that needs a failed output fails on its own
when it runs, then its *own* error action degrades or escalates it. Choosing
`flow.Degraded` (continue) rather than letting the failure escalate to an unwind
is the author's continue-vs-halt call — so a `Degraded` run preserves forward
movement (exit `0`) while surfacing the degradation loudly. The four run
terminals (`Completed`/`Degraded`/`Failed`/`Stranded`) and the
error-actions-MUST-run contract are in
[§2.2 Phase Execution](2.2-phase-execution.md) and
[compensation-failure-contract.md](../plans/extract-starlark-from-op/phase-8/compensation-failure-contract.md).

### Fatal — Immediate Halt

Fatal returns a `FatalError` that the executor treats as an unrecoverable
failure. Execution halts immediately — no retry, no compensation.

### Elevate — Privilege Boundary (Stub)

Elevate makes privilege transitions visible in the graph. Currently a
passthrough stub that returns `nil, nil, nil`. The full privilege provider
design is in [Execution Topology](6-execution-topology.md).

**Design intent:**

```
  ┌──────────┐     ┌──────────┐     ┌──────────┐     ┌──────────┐
  │ download │ →   │ elevate  │ →   │ install  │ →   │ drop     │
  │ (user)   │     │ (→ root) │     │ (root)   │     │ (→ user) │
  └──────────┘     └──────────┘     └──────────┘     └──────────┘
```

Use cases:

- **Package installation**: Download as user, elevate for install, drop for config
- **Service management**: Elevate for systemd/launchd operations
- **Audit**: Receipt shows exactly which operations ran with elevated privileges
- **Dry-run clarity**: Plan shows "this step requires root" before execution

## Retry

Declarative retry policy attached to a node or phase. If the node fails,
re-execute according to the policy. The plan declares the policy and the receipt
records every attempt.

### RetryPolicy

```go
// pkg/op/phase.go

type BackoffStrategy string

const (
    BackoffNone        BackoffStrategy = "none"
    BackoffLinear      BackoffStrategy = "linear"
    BackoffExponential BackoffStrategy = "exponential"
)

type RetryPolicy struct {
    MaxAttempts  int             `json:"max_attempts" yaml:"max_attempts"`
    Backoff      BackoffStrategy `json:"backoff" yaml:"backoff"`
    InitialDelay string          `json:"initial_delay,omitempty" yaml:"initial_delay,omitempty"`
    MaxDelay     string          `json:"max_delay,omitempty" yaml:"max_delay,omitempty"`
}
```

`ComputeDelay(attempt)` applies the backoff strategy: linear multiplies by
`(attempt+1)`, exponential doubles per attempt, both capped by `MaxDelay`.

### Use Cases

- **Package manager failures**: Mirror timeout, lock contention, index refresh race
- **Network operations**: Registry unreachable, download interrupted
- **Service startup**: Service takes time to become healthy after start

### Receipt Output

```yaml
- id: install-docker
  status: completed
  attempts:
    - { attempt: 1, status: failed, error: "mirror timeout", timestamp: "..." }
    - { attempt: 2, status: failed, error: "lock held by apt", timestamp: "..." }
    - { attempt: 3, status: completed, timestamp: "..." }
```

## Compensation

State-based compensating actions. Each action's `Do` method returns a
`Complement` (the undo state) that the executor stores on the recovery stack.
If a later node fails, the executor unwinds the stack in reverse order, passing
the stored complement to each action's `Undo` method.

No separate rollback nodes. The action that performed the work provides both
`Do` and `Undo`, and `Do` captures exactly the state needed for compensation.

### Action Interfaces

```go
// pkg/op/action.go

type Result = any      // data flowing to downstream nodes
type Complement = any  // state captured by Do, passed to Undo during rollback

type Action interface {
    Name() string
    Params() []ParamInfo
    Do(ctx *Context, slots map[string]any) (Result, Complement, error)
}

type CompensableAction interface {
    Action
    Undo(ctx *Context, complement Complement) error
}
```

### Recovery Stack

```go
// pkg/op/recovery.go

type RecoveryStack struct {
    entries []RecoveryEntry
}

type RecoveryEntry struct {
    compensate     func(any) error
    reconcile      func(any) (bool, error)
    undoState      any
    reconcileState any
}
```

Key methods:

- `Do(invoke, compensate, reconcile)` — invoke closure, push recovery entry on success
- `PushAction(ctx, action, complement)` — push CompensableAction undo handler
- `Unwind()` — compensate all entries in LIFO order; reconcile checked first
- `Discard()` — drop all entries without unwinding

The optional `reconcile` function enables drift detection — before compensating,
the stack checks whether the state has already changed (e.g., a file was
modified externally). If reconcile returns false, the compensation is skipped
for that entry.

### Example

```
  Do:   install-docker → returns complement {packages: [...]}
  Do:   configure-docker → returns complement {config_backup: "..."}
  FAIL: start-docker
  Undo: configure-docker(complement) → restores config
  Undo: install-docker(complement) → removes packages
```

## Lifecycle Hooks

Sidecar provides observability hooks — non-intrusive telemetry that attaches
to node and phase boundaries without altering core logic.

### LifecycleHook Interface

```go
// internal/execution/hooks.go

type LifecycleHook interface {
    OnNodeStart(ctx *op.Context, nodeID string, slots map[string]any)
    OnNodeComplete(ctx *op.Context, nodeID string, result op.Result, err error)
    OnPhaseStart(ctx *op.Context, phaseID string)
    OnPhaseComplete(ctx *op.Context, phaseID string, err error)
}
```

### HookRegistry

```go
type HookRegistry struct {
    hooks []LifecycleHook
}

func (r *HookRegistry) Register(hook LifecycleHook)
func (r *HookRegistry) FireNodeStart(ctx, nodeID, slots)
func (r *HookRegistry) FireNodeComplete(ctx, nodeID, result, err)
func (r *HookRegistry) FirePhaseStart(ctx, phaseID)
func (r *HookRegistry) FirePhaseComplete(ctx, phaseID, err)
```

Hooks are fire-and-forget. A hook panic is recovered and logged but does not
fail the node or phase. Hooks run synchronously within the executor's
goroutine — they must not block.

### What Serializes

| Type | Serializes | Format |
|---|---|---|
| SlotValue (Immediate) | Yes | `{immediate: value}` |
| SlotValue (Promise) | Yes | `{node_ref: id, slot: name}` |
| SlotValue (Proxy) | Yes | `{gather_ref: id, field: name}` |

| chooseComplement | Yes (in recovery) | Results map and recovery stack |
| RecoveryEntry | Yes (in recovery) | Node ref + complement |
| Graph | Yes | JSON and YAML, format version 6 |

### What Doesn't Serialize

| Type | Reason |
|---|---|
| PredicateFunc | Go function value — not serializable. Must be re-supplied at execution time. |
| ActivationState | Transient per-execution state. Reconstructed on each execution pass. |
| LifecycleHook | Runtime callback — registered at execution time, not plan time. |
