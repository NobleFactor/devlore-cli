# Receipt Integrity Model

This document describes the two-tier integrity and authenticity model for deployment receipts.

## Overview

Receipts record what writ and lore deployed to a workstation. They serve two purposes:

1. **Integrity**: Detect if receipt contents have been modified (accidental or intentional)
2. **Authenticity**: Verify that a receipt was generated by a trusted source (for sharing)

## Two-Tier Model

### Tier 1: Checksum (Integrity)

Every receipt includes a checksum that detects any modification to the receipt contents.

**Algorithm**: Git-style content-addressed hash

```
SHA256("receipt <basename> <length>\0<content>")
```

Where:
- `basename`: The receipt filename (e.g., `writ-2025-01-29T10-30-00.yaml`)
- `length`: Byte length of the canonical content
- `content`: Canonical YAML representation (excludes checksum and signature fields)

**Format**: `sha256:<hex>`

**Example**:
```yaml
checksum: "sha256:a7b9c3d4e5f6..."
```

The checksum is computed on the canonical content, which excludes the `checksum` and `signature` fields themselves. This prevents circular dependency issues.

### Tier 2: Signature (Authenticity)

Signatures are optional and provide cryptographic proof of origin. This is useful when sharing receipts between team members or systems.

**Current Implementation**: Age encryption

The current implementation uses age encryption as a "claim" mechanism:
- Hash of canonical content is encrypted with an age identity
- Verification requires the corresponding age identity
- Provides proof that the signer possessed the private key

**Future Enhancement**: SOPS-based signing

When SOPS configuration includes a signing-capable key type, proper digital signatures can be used:

| Key Type | Signing Capability |
|----------|-------------------|
| PGP | Full signing support |
| AWS KMS | Full signing support |
| GCP KMS | Full signing support |
| Azure Key Vault | Full signing support |
| HashiCorp Vault | Full signing support |
| age | Claim only (encryption-based) |

## Canonical Content

The canonical representation used for checksum and signature computation includes:

```yaml
version: "4"
format: graph
timestamp: "2025-01-29T10:30:00Z"
tool: writ
platform:
  os: darwin
  arch: arm64
context:
  source_root: ~/.local/share/devlore/repos
  target_root: ~
  projects: [project1, project2]
  segments: {os: darwin, arch: arm64}
roots: [project1, project2]
nodes:
  - id: .config/app/config.yaml
    action: file.link
    status: completed
    ...
edges: []
```

Excluded from canonical content:
- `checksum` field
- `signature` field
- `summary` field (computed from nodes)

## Receipt Storage

### Directory Structure

All tools share a unified receipts directory:

```
~/.local/state/devlore/receipts/
├── writ-2025-01-29T10-30-00.yaml
├── writ-2025-01-28T14-22-15.yaml
├── lore-2025-01-29T11-00-00.yaml
├── writ-latest.yaml -> writ-2025-01-29T10-30-00.yaml
└── lore-latest.yaml -> lore-2025-01-29T11-00-00.yaml
```

### Filename Convention

```
<tool>-<timestamp>.yaml
```

Where:
- `tool`: `writ` or `lore`
- `timestamp`: `YYYY-MM-DDTHH-MM-SS` (ISO 8601, dashes for filesystem safety)

Each tool maintains its own `<tool>-latest.yaml` symlink pointing to the most recent receipt.

## Verification

### Checksum Verification

```go
func (r *Receipt) VerifyChecksum(filename string) error {
    canonical, _ := r.canonicalContent()
    expected := GitStyleChecksum(filename, canonical)
    if r.Checksum != expected {
        return fmt.Errorf("checksum mismatch")
    }
    return nil
}
```

## Security Considerations

1. **Checksum is not cryptographic authentication**: The checksum detects accidental modifications but does not prove origin. Anyone can compute a valid checksum.

2. **Age signatures are claims, not proofs**: The current age-based signature proves the signer possessed the private key at signing time, but age keys are not tied to verified identities.

3. **Trust model**: For shared receipts, establish trust through:
   - Key exchange (age public keys)
   - SOPS configuration (shared `.sops.yaml`)
   - PGP web of trust (when using PGP keys)
