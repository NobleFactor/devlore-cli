# Recovery Site

The recovery site is a shared service that manages archival and restoration
of resources during compensation. It lives on `op.Context` and provides
two pairs of operations: file archival (zero-copy rename) and data archival
(byte serialization). All operations are scoped to the authority boundary
via `op.Root` (the interface wrapping `*os.Root`).

## 1. Problem

The recovery system was originally embedded in the file provider as methods
on `*file.Provider`: `moveToRecovery`, `restoreFromRecovery`, and
platform-specific `getRecoveryBase` implementations. This created three
problems:

1. **Tight coupling.** Recovery logic was bound to `file.Resource` and
   `file.Tombstone`. Other providers that need recovery (e.g., mem) could not
   reuse it without importing the file package.

2. **Platform complexity.** `getRecoveryBase` discovered a same-device
   directory for zero-copy renames — mount-point traversal on Unix,
   volume detection on Windows. This complexity existed solely because
   recovery lived outside the authority boundary.

3. **No data archival.** The original system only handled files (rename).
   Memory resources (`mem.Resource`) hold `Data []byte` in the process
   heap. There was no mechanism to archive and restore byte data during
   compensation.

## 2. Design

### 2.1 Recovery Directory

The recovery directory is `.devlore/recovery/` within the `op.Root`
authority boundary. This eliminates platform-specific discovery — same-
partition is guaranteed because recovery is inside root.

```
<authority-boundary>/   ← op.Root wraps this directory
  .devlore/
    recovery/
      <uuid>/     ← one entry per ArchiveFile/ArchiveData call
    tmp/           ← test runner scratch space
  ...              ← user content
```

Each archive operation generates a UUID and stores the archived entity
at `.devlore/recovery/<uuid>`. The UUID is returned to the caller as
the recovery path, which the caller stores in its tombstone.

### 2.2 RecoverySite

`op.RecoverySite` holds `Context` — the same pattern as `ProviderBase`.
It is a context-aware service, not a standalone utility. It lives in
`pkg/op/` to avoid import cycles (`op.Context` holds `*RecoverySite`).

```go
package op

// RecoverySite manages archival and restoration of resources within the
// op.Root authority boundary. File locations use op.Path. Recovery IDs
// are opaque strings. All I/O goes through Context.Root (the op.Root
// interface — confinedRoot in execution, RootReaderWriter in tests).
type RecoverySite struct {
    ctx Context
}

func NewRecoverySite(ctx Context) *RecoverySite
```

The RecoverySite does not store a base path — it owns the `.devlore/recovery/`
convention internally. Callers never need to know the recovery directory
layout.

### 2.3 File Archival

File archival moves a file into the recovery directory via `ctx.Root.Rename`.
This is a zero-copy operation — no data movement occurs. The file's
identity (inode, content) is preserved; only its directory entry changes.

```go
// ArchiveFile moves a file to recovery via zero-copy rename.
// No data is copied — the file's directory entry is relocated.
// Takes op.Path for the user-facing location. Returns an opaque recovery
// ID for tombstone storage.
func (s *RecoverySite) ArchiveFile(p Path) (recoveryID string, err error)

// RestoreFile moves a file back from recovery via zero-copy rename.
// No data is copied — the directory entry is relocated back.
// Takes op.Path for the original location and the opaque recovery ID.
func (s *RecoverySite) RestoreFile(original Path, recoveryID string) error
```

`ArchiveFile` ensures the recovery directory exists (`ctx.Root.MkdirAll`),
generates a UUID, and renames the file into `.devlore/recovery/<uuid>`.
`RestoreFile` renames it back, recreating the original parent directory
if needed.

### 2.4 Data Archival

Data archival writes bytes to a file in the recovery directory. This
supports providers whose resources exist in memory (e.g., `mem.Resource`
with `Data []byte`).

```go
// ArchiveData writes bytes to a file in the recovery directory.
// Returns the fsroot-relative recovery path for tombstone storage.
func (s *RecoverySite) ArchiveData(data []byte) (recoveryPath string, err error)

// RestoreData reads bytes back from a file in the recovery directory.
func (s *RecoverySite) RestoreData(recoveryPath string) ([]byte, error)
```

`ArchiveData` writes the bytes to `.devlore/recovery/<uuid>` via
`ctx.Root.WriteFile`. `RestoreData` reads them back via `ctx.Root.ReadFile`.

### 2.5 Context Integration

The RecoverySite is a field on `op.Context`, instantiated by the executor:

```go
type Context struct {
    context.Context
    Root         Root            // op.Root interface
    RecoverySite *RecoverySite
    // ...
}
```

Providers access it through the standard context path:

```go
// file provider — zero-copy file archival (op.Path in, opaque ID out)
recoveryID, err := p.Context().RecoverySite.ArchiveFile(resource.SourcePath)

// file provider — restore from recovery (op.Path + opaque ID)
err := p.Context().RecoverySite.RestoreFile(resource.SourcePath, tombstone.RecoveryID)

// mem provider — byte serialization
recoveryID, err := p.Context().RecoverySite.ArchiveData(resource.Data)

// mem provider — restore bytes
data, err := p.Context().RecoverySite.RestoreData(tombstone.RecoveryID)
```

The file provider passes `resource.SourcePath` (an `op.Path`) directly.
No `relPath` helper needed — `op.Path` carries both `Rel()` and `Abs()`.

## 3. Provider Integration

### 3.1 File Provider

The file provider uses RecoverySite for operations that archive files to
the recovery directory: `Remove`, `RemoveAll`, `Unlink`, and `prepareWrite`
(which backs up existing files before overwriting).

**RecoverySite users** (archive to `.devlore/recovery/`):
- `Remove`, `RemoveAll`, `Unlink` — call `ArchiveFile` to move files to recovery
- `CompensateRemove`, `CompensateRemoveAll`, `CompensateUnlink` — call `RestoreFile` to move files back
- `prepareWrite` → calls `Remove` internally
- `compensateWrite` — calls `RestoreFile` to restore the original file

**Direct rename users** (NOT RecoverySite):
- `Backup` — renames file to a peer backup location (same directory, suffix appended)
- `CompensateBackup` — renames the backup back to the original path via `p.rename`
- `Move` — renames file from source to destination
- `CompensateMove` — renames the file back to source via `p.rename`

`Backup` and `Move` do not use RecoverySite because their "recovery" paths
are regular file locations (backup peer, move destination), not the
`.devlore/recovery/` directory. Their compensation uses the provider's
`rename` method directly.

The `pruneEmptyParents` call remains on the file provider — pruning is
a file-specific concern, not a recovery concern.

### 3.2 Memory Provider

The mem provider can use `ArchiveData`/`RestoreData` for compensable
operations that modify in-memory resources. The recovery path is stored
in the provider's own tombstone type.

```go
type Tombstone struct {
    op.TombstoneBase
    RecoveryID string
}
```

## 4. What Got Deleted

The following files and functions were removed from the file provider
when recovery was extracted:

| File | Content | Replacement |
| --- | --- | --- |
| `recovery.go` | `moveToRecovery`, `restoreFromRecovery` | `RecoverySite.ArchiveFile`, `RecoverySite.RestoreFile` |
| `recovery_unix.go` | `getRecoveryBase`, `findMountPoint`, `getFirstExistingAncestor`, `isSameDevice` | Eliminated — recovery is within root |
| `recovery_windows.go` | `getRecoveryBase` (volume detection) | Eliminated — recovery is within root |

The recovery logic now lives in `pkg/op/recovery_site.go`.

## 5. Guarantees

1. **Zero-copy file archival.** `ArchiveFile` and `RestoreFile` use
   `ctx.Root.Rename` — no data is read, copied, or written. The file
   system relocates the directory entry in O(1).

2. **Same-partition.** The recovery directory is inside the `op.Root`
   boundary. All renames are within the same filesystem — cross-device
   errors cannot occur.

3. **Confinement.** All recovery I/O goes through `ctx.Root` (the `op.Root`
   interface). In execution mode (`confinedRoot`), symlink escapes, `..`
   traversal, and paths outside the authority boundary are blocked by the
   kernel. In test mode (`RootReaderWriter`), I/O is unconfined but scoped
   to the test directory.

4. **Provider independence.** The RecoverySite works with paths and bytes.
   It has no knowledge of `file.Resource`, `mem.Resource`, or any provider-
   specific type. Providers wrap the recovery path into their own
   tombstone types.

5. **Idempotent restore.** `RestoreFile` recreates the parent directory
   if it was pruned after archival. The restored file appears at its
   original path regardless of intermediate directory cleanup.
