# Recovery Site

The recovery site is a shared service that manages archival and restoration
of resources during compensation. It lives on `op.Context` and provides
two pairs of operations: file archival (zero-copy rename) and data archival
(byte serialization). All operations are scoped to the authority boundary
via `os.Root`.

## 1. Problem

The recovery system is currently embedded in the file provider as methods
on `*file.Provider`: `moveToRecovery`, `restoreFromRecovery`, and
platform-specific `getRecoveryBase` implementations. This creates three
problems:

1. **Tight coupling.** Recovery logic is bound to `file.Resource` and
   `file.Tombstone`. Other providers that need recovery (e.g., mem) cannot
   reuse it without importing the file package.

2. **Platform complexity.** `getRecoveryBase` discovers a same-device
   directory for zero-copy renames — mount-point traversal on Unix,
   volume detection on Windows. This complexity exists solely because
   recovery lives outside the authority boundary.

3. **No data archival.** The current system only handles files (rename).
   Memory resources (`mem.Resource`) hold `Data []byte` in the process
   heap. There is no mechanism to archive and restore byte data during
   compensation.

## 2. Design

### 2.1 Recovery Directory

The recovery directory is `.devlore/recovery/` within the `os.Root`
authority boundary. This eliminates platform-specific discovery — same-
partition is guaranteed because recovery is inside root.

```
<BaseDir>/
  .devlore/
    recovery/
      <uuid>/     ← one entry per ArchiveFile/ArchiveData call
    tmp/           ← test runner scratch space
  ...              ← user content
```

Each archive operation generates a UUID and stores the archived entity
at `.devlore/recovery/<uuid>`. The UUID is returned to the caller as
the recovery path, which the caller stores in its tombstone.

### 2.2 Site

The `recovery.Site` struct holds a reference to the `os.Root` from
Context. It is instantiated by the executor alongside `os.Root` and
placed on `op.Context`.

```go
package recovery

// Site manages archival and restoration of resources within the os.Root
// authority boundary. All paths are root-relative.
type Site struct {
    root *os.Root
}

func NewSite(root *os.Root) *Site
```

The Site does not store a base path — it owns the `.devlore/recovery/`
convention internally. Callers never need to know the recovery directory
layout.

### 2.3 File Archival

File archival moves a file into the recovery directory via `os.Root.Rename`.
This is a zero-copy operation — no data movement occurs. The file's
identity (inode, content) is preserved; only its directory entry changes.

```go
// ArchiveFile moves a file to recovery via zero-copy rename.
// No data is copied — the file's directory entry is relocated.
// Returns the recovery-relative path for tombstone storage.
func (s *Site) ArchiveFile(path string) (recoveryPath string, err error)

// RestoreFile moves a file back from recovery via zero-copy rename.
// No data is copied — the directory entry is relocated back.
func (s *Site) RestoreFile(originalPath, recoveryPath string) error
```

`ArchiveFile` ensures the recovery directory exists (`root.MkdirAll`),
generates a UUID, and renames the file into `.devlore/recovery/<uuid>`.
`RestoreFile` renames it back, recreating the original parent directory
if needed.

### 2.4 Data Archival

Data archival writes bytes to a file in the recovery directory. This
supports providers whose resources exist in memory (e.g., `mem.Resource`
with `Data []byte`).

```go
// ArchiveData writes bytes to a file in the recovery directory.
// Returns the recovery-relative path for tombstone storage.
func (s *Site) ArchiveData(data []byte) (recoveryPath string, err error)

// RestoreData reads bytes back from a file in the recovery directory.
func (s *Site) RestoreData(recoveryPath string) ([]byte, error)
```

`ArchiveData` writes the bytes to `.devlore/recovery/<uuid>` via
`root.WriteFile`. `RestoreData` reads them back via `root.ReadFile`.

### 2.5 Context Integration

The Site is a field on `op.Context`, instantiated by the executor:

```go
type Context struct {
    context.Context
    BaseDir      string
    Root         *os.Root
    RecoverySite *recovery.Site
    // ...
}
```

Providers access it through the standard context path:

```go
// file provider — zero-copy file archival
recoveryPath, err := p.Context().RecoverySite.ArchiveFile(resource.SourcePath.Rel)

// file provider — restore from recovery
err := p.Context().RecoverySite.RestoreFile(resource.SourcePath.Rel, tombstone.RecoveryPath)

// mem provider — byte serialization
recoveryPath, err := p.Context().RecoverySite.ArchiveData(resource.Data)

// mem provider — restore bytes
data, err := p.Context().RecoverySite.RestoreData(tombstone.RecoveryPath)
```

## 3. Provider Integration

### 3.1 File Provider

The file provider replaces its embedded `moveToRecovery` and
`restoreFromRecovery` methods with calls to `RecoverySite.ArchiveFile`
and `RecoverySite.RestoreFile`. The provider continues to own its
`Tombstone` type — the recovery path returned by `ArchiveFile` is stored
in `Tombstone.RecoveryPath` as before.

Methods affected:
- `Remove`, `RemoveAll`, `Unlink` — currently call `moveToRecovery`
- `CompensateRemove`, `CompensateRemoveAll`, `CompensateUnlink` — currently call `restoreFromRecovery`
- `CompensateBackup`, `CompensateMove` — currently call `restoreFromRecovery`
- `prepareWrite` — calls `Remove` internally, which calls `moveToRecovery`
- `compensateWrite` — calls `restoreFromRecovery`

The `pruneEmptyParents` call remains on the file provider — pruning is
a file-specific concern, not a recovery concern.

### 3.2 Memory Provider

The mem provider can use `ArchiveData`/`RestoreData` for compensable
operations that modify in-memory resources. The recovery path is stored
in the provider's own tombstone type.

```go
type Tombstone struct {
    op.TombstoneBase
    RecoveryPath string
}
```

## 4. What Gets Deleted

The following files and functions are removed from the file provider
once the recovery.Site migration is complete:

| File | Content | Replacement |
| --- | --- | --- |
| `recovery.go` | `moveToRecovery`, `restoreFromRecovery` | `RecoverySite.ArchiveFile`, `RecoverySite.RestoreFile` |
| `recovery_unix.go` | `getRecoveryBase`, `findMountPoint`, `getFirstExistingAncestor`, `isSameDevice` | Eliminated — recovery is within root |
| `recovery_windows.go` | `getRecoveryBase` (volume detection) | Eliminated — recovery is within root |

## 5. Guarantees

1. **Zero-copy file archival.** `ArchiveFile` and `RestoreFile` use
   `os.Root.Rename` — no data is read, copied, or written. The file
   system relocates the directory entry in O(1).

2. **Same-partition.** The recovery directory is inside the `os.Root`
   boundary. All renames are within the same filesystem — cross-device
   errors cannot occur.

3. **Confinement.** All recovery I/O goes through `os.Root`. Symlink
   escapes, `..` traversal, and paths outside the authority boundary
   are blocked by the kernel.

4. **Provider independence.** The Site works with paths and bytes. It
   has no knowledge of `file.Resource`, `mem.Resource`, or any provider-
   specific type. Providers wrap the recovery path into their own
   tombstone types.

5. **Idempotent restore.** `RestoreFile` recreates the parent directory
   if it was pruned after archival. The restored file appears at its
   original path regardless of intermediate directory cleanup.
