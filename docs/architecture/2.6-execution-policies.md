# Execution Policies: Retry, Elevation, and Error Handling

> **Status:** design (draft). Records the per-unit policy model, the current implementation maturity, the resolution
> model we are adopting, and the open design work. Companion:
> [`2.6-execution-policies.status.md`](2.6-execution-policies.status.md).

## Thesis

Every executable unit in a graph carries a small set of **execution policies** governing how it behaves under failure
and privilege: how many times it retries, what runs when it fails, and whether it runs elevated. These are declared
together on the `ExecutableUnit` interface as a "per-unit policy triplet" — but **the three are not equals.** Retry and
error-handling are genuine per-unit policies, resolved by a *floor-plus-inheritance* model at plan time. **Elevation is
a different, multi-party mechanism**: a unit declares a *requirement*, a graph aggregates the *offers it needs*,
configuration *provisions* the specific offers, and execution *realizes* them.

This document records what the code does today — with precise maturity levels — the resolution model, and what remains
to design.

## The policy family

`pkg/op/executable_unit.go:54-57` declares the triplet on the `ExecutableUnit` interface:

```go
// Per-unit policy triplet: elevation, retry, and the error-handler subgraph (each nil-able).
ElevationOffer() *ElevationOffer
RetryPolicy()    *RetryPolicy
ErrorAction()    *Subgraph
```

Each is `nil`-able and backed by an unexported field on the shared `executableUnit` base
(`pkg/op/executable_unit.go:100-102`), which **`Node` and `Subgraph` both embed** — and since a `Graph`'s root is a
`Subgraph`, every unit kind carries the triplet. The common lifecycle is:

```
author (spec With* / starlark) → plan-time stamp (constructor seal) → unit accessor → executor consumption
```

- **Spec surface.** `ExecutableUnitSpec` holds all three fields; `NodeSpec` / `SubgraphSpec` / `GraphSpec` each
  re-declare `With{ElevationOffer,RetryPolicy,ErrorAction}` for fluent chaining, and `GraphSpec`'s three delegate to the
  **root subgraph** (`pkg/op/graph.go:731,743,786`).
- **Plan-time stamping.** Copied spec→unit in the constructors, nil-guarded (`pkg/op/node.go:55-65`,
  `pkg/op/subgraph.go:715-725`), honoring the "no post-construction mutation" seal.
- **Planner channel.** `Planner.Plan` threads `errorAction` and `retryPolicy` as explicit parameters
  (`pkg/op/planner.go:116-117`) — but **not** `elevationOffer`; elevation has no planner channel today, only the Go spec
  path.

## Maturity gradient (read this first)

The three sit at very different levels of completeness. This is the single most important fact for anyone touching
them:

| Policy | Authoring | Stamped | Serialized | Consumed at runtime? |
|---|---|---|---|---|
| **RetryPolicy** | Go spec **+ 2 starlark paths** | ✅ | ✅ (`retry:`) | ✅ **Fully** — drives the retry loop |
| **ErrorAction** | Go spec **+ starlark** (`error_action=`) | ✅ | ✅ (as a child subgraph) | ✅ **Yes, but observation-only** |
| **ElevationOffer** | **Go spec only** (no starlark) | ✅ | ❌ dropped | ❌ **Never — dormant scaffolding** |

## Retry

**Definition** (`pkg/op/retry_policy.go:11-24`): `{MaxAttempts int, Backoff (none/linear/exponential), InitialDelay,
MaxDelay string}`. `ComputeDelay(attempt)` (`pkg/op/retry_policy.go:40-69`) does the backoff math with a `MaxDelay`
clamp. **`MaxAttempts` counts *retries*, not attempts** — the executor uses `MaxAttempts + 1` total attempts; a nil
policy means exactly one attempt.

**Two starlark authoring surfaces** (unique to retry):
1. **Reserved kwarg** `plan.x.y(..., retry_policy=...)` → `splitReservedKwargs` (`pkg/op/provider/plan/helpers.go:232`)
   → planner → spec seal.
2. **Chained builtin** `node.retry(max_attempts=…, backoff=…)` → `Promise.retryBuiltin`
   (`pkg/op/promise.go:259-308`), which **mutates the already-built unit in place** via `setRetryPolicy`
   (`pkg/op/promise.go:306`).

**Consumption — the single act-on-it site: `ActivationRecord.DispatchChild`** (`pkg/op/activation_record.go:166-205`).
The attempt loop: budget = `nil ? 1 : MaxAttempts+1` (`:174-177`); backoff before each retry via
`ComputeDelay(attempt-1)`, interruptible by `ctx.Done()` (`:182-191`); first success returns; **pause/cancel never burn
budget** (`:199-201`); exhaustion returns the last error.

**fsroot-inert caveat.** Retry lives *only* in `DispatchChild`, which is *only* how a bound subgraph dispatches its
children (`pkg/op/provider/flow/helpers.go:115`). So a unit's policy is honored only when it is walked as a child — and
the **root subgraph's own policy is inert**, because the root is dispatched directly by `GraphExecutor.Run`
(`pkg/op/graph_executor.go:263`), never through `DispatchChild`. `Node.Execute` / `Subgraph.Execute` contain no retry
loop.

## Elevation

Elevation is **not a true peer** of retry and error-handling. It is a multi-party mechanism with four aspects, each
owned by a different participant:

| Aspect | Owner | Meaning | Today's reality |
|---|---|---|---|
| **Requirement** | the **ExecutableUnit** | "I need to run as admin" — a *kind* | a *boolean* `NeedsSudo()` on the **manager** |
| **Offer-need** | the **Graph** | "what offers do I need?" = aggregate of its units' requirements | — nothing; never aggregated |
| **Offer-provision** | the **Configuration** | the env-specific realization (dev creds in test, prod in prod) | — nothing; no creds/identity layer |
| **Realization** | **execution** | actually entering the elevated shell | inline `sudo -n` per command |

An *ElevationOffer* is a **guarantee** ("in this environment I offer sudo"); a **requirement** is "I need sudo"; a
**realization** is "running `apt` in a sudo shell."

**Current code.** The `ElevationOffer` type is rich — a four-axis declarative policy
`{Strategy, Scope, Lifespan, Fallback}` (`pkg/op/elevation_policy.go`) with strategies
`host_escalation / interactive_challenge / identity_assumption / mandated_approval`. But it is **plumbed and dormant**:
no code reads `unit.ElevationOffer()`, it is not serialized, the `flow.Provider.Elevate()` builtin is an empty no-op
(`pkg/op/provider/flow/provider.go:507`), and there is no starlark authoring path. The only *requirement* statement that
exists is the per-manager boolean `NeedsSudo()` (`pkg/platform/manager.go:180`; `true` for apt/dnf/pacman/systemd,
MacPorts, the Windows service manager; `false` for brew/winget/launchd). `pkg.install`
(`pkg/op/provider/pkg/provider.go:51`) declares nothing itself — it is a thin veneer that delegates to the resolved
manager. The realization is the inline `sudo -n` in `pkg/platform/helpers.go:79`, gated by that boolean, with a TODO
(`pkg/platform/helpers.go:77-78`) to route privileged execution through `ElevationOffer` someday.

**The host-resolution tension (key constraint).** The requirement is **platform-conditional and resolved at
*execution*, not planning**: `pkg.install` needs `host_escalation` on apt/dnf/pacman/MacPorts but *not* on brew/winget,
and which manager applies is not known until `platform.Detect()` runs on the host. A graph planned on a Mac (brew, no
sudo) and executed on Ubuntu (apt, sudo) has a *different* elevation requirement at execute time than anything stamped
at plan time. So a unit cannot statically declare "I require sudo"; at most it declares "I perform a
**privileged-capable** operation, and whether elevation is actually needed is resolved against the execution host's
manager." This pushes elevation's requirement→need→provision resolution at least partly to **execution time** — in
tension with the plan-time stance taken for retry and error-handling. **Elevation is the odd one out.**

**Other design forces** (for the eventual spec, not now):
- **Cross-platform granularity.** `sudo -n` is *per-command*; Windows UAC / `runas` and macOS GUI auth are *per-process
  re-launch* — you generally cannot elevate a single in-flight unit in-process, you re-launch a helper elevated. So
  elevation often spans a **session of units**, not one unit — which is what `ElevationLifespan{Ephemeral,
  CacheDuration}` and the "one credential-cached, audited session" TODO reach for. The **elevated session** wants to be
  a first-class runtime object, distinct from both the offer and the request.
- **Async strategies ride pause/resume.** `interactive_challenge` (prompt) and `mandated_approval` (await a gatekeeper)
  are blocking; a request needing approval should **pause the unit** (a pause-point, reusing `ErrPaused`) until granted.
- **Audit falls out.** Each request matched against the offer is a grant event; thread it through the receipt /
  RecoveryStack audit trail.

## Error Handling

**Definition.** An ErrorAction is a `*Subgraph` failure-handler, stamped as a **child of the guarded unit**
(`pkg/op/executable_unit.go:185-199`, `ea.stampParentID(e.ID())`) so the orphan scan covers it. It is itself a
`flow.subgraph` container of the bodies passed via `error_action=`.

**Authoring.** The `error_action=` reserved kwarg → `errorActionSubgraph` → `subgraphFromInvocations`
(`pkg/op/provider/plan/helpers.go:247,108,289`); plus a first-class `plan.assemble(error_action=…)` path
(`pkg/op/provider/plan/provider.go:160-193`).

**Consumption — observation-only.** `flow.Provider.Subgraph` passes `subgraph.ErrorAction()` into `walkSubgraphChildren`
(`pkg/op/provider/flow/helpers.go:102-133`). After a child's `DispatchChild` returns an error — i.e., **only after
retries are exhausted** — if the error-action is non-nil it is dispatched **once, best-effort**; its result and error
are **discarded** (`:122` logs to stderr) and the **original child error always surfaces** (`:126`). It receives the
same variable frame and recovery stack as the failing child but **no error context**. It is **distinct from
compensation**: compensation is the separate LIFO `stack.Unwind()` driven by `GraphExecutor.Run`
(`pkg/op/graph_executor.go:271-275`); error-action is a forward observation pass on the same stack. `Gather`
deliberately passes `nil` (does not honor it). The provider doc states it plainly: "errorAction is an observation hook,
not a recovery path" (`pkg/op/provider/flow/provider.go:323-325`).

**Floor.** The intended ground-level error action is **`flow.failed`** (`flow.Provider.Failed`; starlark `flow.failed`).
The accessor doc already anticipates a "`flow.Provider.Failed` sentinel" default (`pkg/op/executable_unit.go:181`), but
that default is **deferred/unwired** today (`pkg/op/provider/flow/provider.go:323-325`) — nil currently means "skip the
observation pass," not "run `flow.failed`." Wiring `nil → flow.failed` at resolution is small and localized.

## The resolution model — floor + inherit, at plan time

We resolve each policy from a **floor** through **inheritance** to an explicit **set** value, and we do it **at planning
time**, stamping the concrete result on the sealed unit. This deliberately supersedes the code's *pending* runtime
inheritance walk (`pkg/op/provider/flow/provider.go:316-317` flags the current per-child `RetryPolicy()` lookup as
interim, with a frame-chain `effectiveRetryPolicy` helper "pending"): if planning bakes in the effective value,
execution stays dumb, the resolution is visible in the saved document, and the runtime walk becomes unnecessary.

The two resilience policies differ in *cardinality of state*, which drives their representation:

- **ErrorAction — bi-state `{set, inherit}`.** The `*Subgraph` pointer suffices: `nil` = inherit (→ the `flow.failed`
  floor), non-nil = override. No representation problem.
- **RetryPolicy — tri-state `{nil, set, inherit}`.** A bare `*RetryPolicy` pointer is only two-state, and `nil` already
  *means* "no retry," so the third state must be encoded. The cleanest encoding without a sentinel: **`nil` = inherit**
  (walk to the floor), **`&{MaxAttempts:0}` = explicit no-retry**, **`&{MaxAttempts:N>0}` = retry** — reusing the
  existing `MaxAttempts==0 ⇒ one attempt` semantics as the "explicit none" value. The cost: this **flips nil's current
  meaning** from "no retry" to "inherit," so a policy-less unit would start inheriting the floor. The alternative is a
  wrapper/enum (`{Inherit, None, Policy{…}}`) — more explicit, heavier. **Open.**

Elevation does not fit this model: its "floor" is a graph-level *offer*, its per-unit value is a *requirement* (not an
offer), and its resolution is partly host-bound (above). It is resolved separately.

## Floors

The plan-time base values each policy starts from:

| Policy | Floor |
|---|---|
| **RetryPolicy** | `<to-be-determined>` (see the tri-state note above — interacts with whether `nil` means inherit) |
| **ErrorAction** | `flow.failed` (`flow.Provider.Failed`; starlark `flow.failed`) |
| **ElevationOffer** | the run-as-another-user host offer (sudo / `runas` / UAC — `host_escalation`) |

These floors are **plan-time defaults**, not configuration the `RuntimeEnvironment` injects — with the one exception
that elevation's *offer-provision* (the env-specific credentials) is genuinely configuration's job (see Elevation).

## Open design work

Explicitly deferred — this document is the analysis, not the build:

1. **RetryPolicy tri-state encoding** — `nil=inherit` vs a wrapper/enum; and the floor value itself.
2. **Plan-time resolution pass** — where the planner walks the frame chain and stamps the effective retry / error-action
   onto each unit (retiring the pending runtime `effectiveRetryPolicy`).
3. **ErrorAction floor wiring** — `nil → flow.failed`.
4. **The elevation spec** — the full requirement / offer-need / offer-provision / realization mechanics: moving the
   offer off the per-unit triplet to the graph, the per-unit *requirement* (a kind), the elevated-session object, the
   pause-on-approval flow, the cross-platform realization, and the audit trail. Substantial; its own effort.
