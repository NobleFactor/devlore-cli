# Plan: 13.0(n) — Variable binding infrastructure

## Summary

Rebuild the slot value model around a PowerShell-style variable binding system:
graphs, subgraphs, and flow-control containers expose a bubble-up parameter
surface; the caller supplies values at execute time via a single immutable
variable map; a new `binding.VariableResolver` assembles that map from layered
sources (override / flag / env / config / default) with explicit precedence and
origin tracking. Retire the ambient `RuntimeEnvironment.Data` / `Properties`
machinery. Test-first: write the starlark integration test that models
`writ adopt` before writing the infrastructure. `writ adopt` running through
the new binding model is the exit discriminator.

## Exit criterion

13.0(n) is complete when both of the following hold:

1. **All tests pass**, including a new starlark integration test under
   `cmd/devlore-test/testdata/binding/` that models the `writ adopt`
   workflow end-to-end (mkdir → move → link with bubbled parameter binding).
2. **`writ adopt` runs through the new infrastructure** and produces the
   same behavior the user observes today — the three nil-activation call
   sites in `cmd/writ/writ/commands.go` (`:1400` Mkdir, `:1410` Move,
   `:1422` Link) are gone, replaced by graph construction +
   `VariableResolver` + `GraphExecutor.Run`.

When both criteria hold, `make build` is clean, `make vet` is clean,
`make test` is green across the full suite.

The remaining 5 nil-activation call sites in `cmd/writ/writ/migrate_cmd.go`
and `cmd/writ/writ/migrate/execute.go` (writ migrate) and the file
provider's defensive nil-activation paths are **out of scope** for 13.0(n) —
they will be addressed in a follow-on step that picks up after 13.0 closes,
with the smaller-PR cadence.

## Background

### Today's slot model (pre-13.0(n))

Sealed three-variant `op.SlotValue` (`pkg/op/slot.go:40-46`):

| Variant | Resolved at | Source |
|---|---|---|
| `ImmediateValue{Value any}` | plan time | call-site literal or converted Go value |
| `PromiseValue{NodeRef, Slot}` | execute time | `results[NodeRef]` (sibling output) |
| `EnvironmentValue{Key string}` | execute time | `Properties.Property(Key)` reading `RuntimeEnvironment.Data` |

The `Properties` interface and `RuntimeEnvironment.Data map[string]any`
(`pkg/op/runtime_environment.go:48`, `:263`) provide an ambient property bag
populated at process startup from CLI flags, env vars, and user config files
(`docs/architecture/2.1-typed-slots.md:30-36`). `plan.env("target_root")`
created an `EnvironmentValue` at plan time; resolution deferred to execute
time via `Properties.Property(key)`.

### Why the model needs to change

Two structural problems block the bubble-up composition model that
flow-control combinators (`plan.subgraph`, `plan.choose`, `plan.gather`,
`plan.wait_until`) require:

1. **No declared parameter surface.** Today, `overrides` passed to
   `Execute(graph, overrides)` is an opaque `map[string]any`. The executor
   has no declared schema to check against. A missing required parameter
   surfaces as a nil dereference or a type-assertion panic deep inside a
   provider method, mid-execution, after side effects have partially
   committed.
2. **Ambient state is not composable.** `RuntimeEnvironment.Data` is
   process-wide; it cannot be re-scoped per subgraph. A subgraph composed
   inside a parent graph cannot supply a different value for its child's
   `EnvironmentValue` lookup — the lookup hits the shared `Data` bag.

Bubble-up requires both: a typed parameter surface declared at plan time,
and a variable-resolution layer that scopes per-call rather than per-process
and tracks where each value came from.

### The unified model

Replace `EnvironmentValue` with `Variable`. The slot's lookup target moves
from "process-wide property bag" to "this call's variable map." The tool
main becomes the assembler of the variable map — it reads CLI flags, env,
config, and produces a typed `map[string]binding.Variable` via
`VariableResolver`; the executor sees only the resolved map. The
`Properties` abstraction goes away from the slot model entirely. Subgraph
composition becomes recursive bubble-up: a subgraph's `Variable` slots are
bound by its parent graph's slot fill, which itself may be `ImmediateValue`,
`PromiseValue`, or another `Variable` (bubble continues all the way up to
the root graph's parameter surface).

## Design

### D1 — Sealed three-variant `SlotValue`

```go
type SlotValue interface {
    isSlotValue()
    Resolve(variables map[string]binding.Variable, results map[string]any) any
}

type ImmediateValue struct{ Value any }
type PromiseValue   struct{ NodeRef, Slot string }
type Variable       struct{ Name string }
```

Signature change: the old `Resolve(props Properties, results map[string]any)`
becomes `Resolve(variables map[string]binding.Variable, results map[string]any)`.
`ImmediateValue` and `PromiseValue` ignore the variables map; `Variable.Resolve`
returns `variables[v.Name].Value`.

`Variable.Resolve` is reachable only after preflight has confirmed every
required `Variable` has a matching entry in the variable map. If
`variables[Name]` is missing at Resolve time, that's an executor invariant
violation; today's behavior returns nil — Phase 5's preflight ensures the case
is unreachable.

### D2 — Parameter defaults

```go
type Parameter struct {
    Name    string
    Type    reflect.Type
    Default *any  // nil = required; non-nil = optional with *Default as fallback
}
```

Pointer-to-interface sentinel: `Default == nil` → required. `Default != nil` →
optional; `*Default` is the fallback value.

Validation at two points:

- **Preflight.** For each declared parameter with a non-nil `Default`,
  type-assert the default's runtime type against `Type`. Mismatches aggregate
  into the D5 envelope.
- **Slot-fill time.** The resolved `Variable.Value` is type-asserted against
  `Type` before being placed into the slot. Defense-in-depth — preflight
  should catch type problems first, but a slot-fill assertion mismatch is a
  `VariableResolver` bug and should fail immediately rather than panic inside
  a provider method.

### D3 — `ExecutableUnit.Parameters()` bubble-up

```go
type ExecutableUnit interface {
    // ... existing methods ...
    Parameters() []op.Parameter
}
```

- **Node.** `Parameters()` returns the method's parameters whose slot's
  `SlotValue` is `Variable`. The `Parameter.Type` is the method's declared
  parameter type; the `Parameter.Default` is propagated from the declared
  default (or nil if the parameter is required).
- **Subgraph.** `Parameters()` walks the topological-root children (children
  with no incoming edges within the subgraph), unions their `Parameters()`
  outputs, and deduplicates by name. Multi-root collision (two roots declare
  the same parameter name with incompatible types) is a plan-time error.

This matches the Phase 7 invariant
(`docs/plans/extract-starlark-from-op.md:1215-1225`): "Executable units
declare a parameter surface via `Parameters() []op.Parameter`; for a node
this is `method.Parameters()`, for a subgraph it is the union (by name) of
every topological root's parameters." Phase 7 deferred the implementation;
13.0(n) lands it.

### D4 — `Namespace`, `Origin`, `Variable` types

The three value types in `pkg/binding` that carry resolution data:

```go
package binding

// Namespace identifies a variable source category. Numeric values ascend
// with precedence — higher value beats lower.
type Namespace int

const (
    NamespaceUnknown  Namespace = iota
    NamespaceDefault            // hardcoded fallback declared on plan.var
    NamespaceConfig             // starlark or YAML config files
    NamespaceEnv                // environment variables
    NamespaceFlag               // command-line arguments
    NamespaceOverride           // explicit runtime force
)

func (n Namespace) String() string {
    return [...]string{"unknown", "default", "config", "env", "flag", "override"}[n]
}

// Origin records where a resolved variable value came from.
type Origin struct {
    Namespace Namespace
    Name      string // the lookup key that matched (e.g., "DEVLORE_WRIT_ROOT" or "config.star:12")
}

func (o Origin) String() string {
    if o.Namespace == NamespaceUnknown {
        return "unknown"
    }
    return o.Namespace.String() + ":" + o.Name
}

// Variable pairs a resolved value with its name and origin.
type Variable struct {
    Name   string
    Value  any
    Origin Origin
}

func (v Variable) String() string {
    return fmt.Sprintf("%s = %v [%s]", v.Name, v.Value, v.Origin)
}
```

#### `Name` convention per `Namespace`

| Namespace | `Origin.Name` = | Example `Origin.String()` |
|---|---|---|
| `NamespaceUnknown` | (empty) | `unknown` |
| `NamespaceDefault` | the parameter's declared name | `default:target_root` |
| `NamespaceConfig` | matched config-map key (or `file:line` if the loader tracks it) | `config:target_root`, `config:config.star:12` |
| `NamespaceEnv` (program-specific hit) | literal env var with program prefix | `env:DEVLORE_WRIT_TARGET_ROOT` |
| `NamespaceEnv` (global cascade hit) | literal env var with global prefix | `env:DEVLORE_TARGET_ROOT` |
| `NamespaceFlag` | the parameter name (Go/starlark form) | `flag:target_root` |
| `NamespaceOverride` | the override-map key | `override:layer` |

The `Name` field is opt-in rich — sources that have richer location data
(file:line, etc.) supply it; flat-map sources just supply the key.

### D5 — `VariableResolver`

```go
package binding

type VariableResolver struct {
    overrides    map[string]any  // NamespaceOverride source
    flags        map[string]any  // NamespaceFlag source
    envPrefix    string          // NamespaceEnv — "DEVLORE_WRIT"
    globalPrefix string          // NamespaceEnv cascade — "DEVLORE"
    config       map[string]any  // NamespaceConfig source

    resolved map[string]Variable // populated by Resolve; nil until then
}

type Option func(*VariableResolver)

func WithOverrides(m map[string]any) Option
func WithFlags(m map[string]any) Option
func WithEnvPrefix(programPrefix string) Option   // "DEVLORE_WRIT"; derives "DEVLORE" as global
func WithConfig(m map[string]any) Option

func NewVariableResolver(opts ...Option) *VariableResolver

// Resolve walks each parameter through the source precedence chain and
// populates the resolver's internal variable map. Returns aggregated errors
// for missing required, type mismatch, default-type mismatch.
func (r *VariableResolver) Resolve(parameters []Parameter) []error

// Variables returns the full resolved variable map. Panics if called before Resolve.
func (r *VariableResolver) Variables() map[string]Variable

// Get returns a single resolved variable. Panics if called before Resolve.
func (r *VariableResolver) Get(name string) (Variable, bool)
```

Construction via functional options; **no exported fields**. All sources
captured at construction; the resolver is read-only after that until `Resolve`
populates `resolved`. After `Resolve` the resolver is fully read-only.

**Resolution order per parameter** (descending precedence, first hit wins):

1. `overrides[name]` → `NamespaceOverride`, `Name=name`.
2. `flags[name]` → `NamespaceFlag`, `Name=name`.
3. `os.Getenv(envPrefix + "_" + camelToSnake(name))` → `NamespaceEnv`,
   `Name="DEVLORE_WRIT_TARGET_ROOT"`.
4. `os.Getenv(globalPrefix + "_" + camelToSnake(name))` → `NamespaceEnv`,
   `Name="DEVLORE_TARGET_ROOT"`.
5. `config[name]` → `NamespaceConfig`, `Name=name`.
6. `*parameter.Default` if non-nil → `NamespaceDefault`, `Name=name`.

After picking the value, the resolver:

- Parses env-sourced values from string to the parameter's declared Go type
  (env always yields a string; other sources supply already-typed Go values).
- Type-asserts the picked value against `parameter.Type`. Mismatch → error.
- Records `resolved[name] = Variable{Name, Value, Origin}`.

For each required parameter (`Default == nil`) that found nothing → error
wrapping the parameter name, target type, and the literal lookup keys the
resolver tried.

`Resolve` returns aggregated `[]error`; the caller (the executor's preflight
pass) folds errors into the D5 envelope. Caller fetches the resolved map via
`Variables()` afterward.

`Get(name)` returns `(Variable, bool)`. Panics if called before `Resolve` —
that's a programmer error.

### D6 — Override routing at `Execute()`

`GraphExecutor.Run` signature changes from `Run(graph)` to
`Run(graph, variables map[string]binding.Variable)`. The executor stashes the
variable map on its receiver; dispatch reaches a slot with `Variable{Name: X}`
and resolves via `variables[X].Value`. No `Properties` interface lookup remains
anywhere in the executor.

Subgraph composition: when a parent graph's slot fill for a child subgraph is
`Variable{Name: Y}`, the variable map at the root carries `Y`. The subgraph's
matching slot resolves to the same value. Bubble-up at runtime mirrors the
bubble-up declared at plan time.

### D7 — Preflight integration

New preflight pass `bindVariables` added to `GraphExecutor.Run`, joining
`ResolvePending` (landed by 13.0(k) k.15) and future orphan / type-check /
platform passes. The executor doesn't construct the resolver itself — the
caller (tool main or test harness) constructs the resolver, supplies sources,
and passes the resolved variable map to `Run`. Preflight at this layer:

1. Compute `graph.Parameters()` — the root's bubble-up surface.
2. For each parameter:
   - Check `variables[name]` exists, OR parameter has a default.
   - If parameter has a default, type-assert `*Default` against `Type`.
   - If variable is present, type-assert `variables[name].Value` against `Type`.
3. Collect errors; aggregate into the D5 envelope.
4. Fail fast on non-empty: graph transitions to `StateFailed`; executor returns
   `errors.Join(errs...)`.

Skipped in dry-run, matching `ResolvePending`'s convention
(`pkg/op/executor.go:79-84`).

## Implementation phases

**Test-first ordering.** Phase 1 builds the foundations needed to author
the integration test that defines the contract: harness extensions, the
`plan.var` stub, the `executor.Run` signature change, and the `.star` test
files themselves. Phase 1 also lands the slot-model rewrite — the three-variant
sealed `SlotValue` with `Variable` replacing `EnvironmentValue` — since the
harness needs the new type to compile. The tests parse and the harness invokes
them end-to-end; they fail at the executor with a "variable not resolvable"
error until later phases land. Subsequent phases build the resolver, bubble-up,
and preflight to satisfy the test.

Each phase is one or more commits on
`refactor/extract-starlark-from-op.phase-8`. Build stays green at every commit
boundary — Phase 1's failing-tests are intentional and the test harness reports
them as expected failures via `t.expect_error`.

## Open questions to lock during implementation

1. **`camelToSnake` rules for env keys.** Edge cases like `ID` (`I_D` or `ID`?),
   `URLPath` (`URL_PATH` or `U_R_L_PATH`?). Pick rules consistent with Go's
   idiomatic patterns or document a custom rule.
2. **Env-string parsing target types.** Resolver parses env values from string
   to the declared Go type at resolve time. Need a small parser registry:
   `string → int`, `string → bool`, `string → os.FileMode`, `string →
   time.Duration`, `string → string`. Initial set covers writ adopt's needs;
   extend as new types appear.
3. **Helper location in `cmd/writ/writ/`.** New file (`adopt_helper.go`?) or
   fold into `commands.go`? Lean toward new file.
4. **Phase 5 `plan.var` kwarg parsing.** Codegen pathway or hand-written
   builtin? Likely hand-written.
