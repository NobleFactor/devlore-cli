# Plan: 13.0(n) — Parameter binding infrastructure

## Summary

Rebuild the slot value model around a PowerShell-style parameter binding
system: graphs, subgraphs, and flow-control containers expose a bubble-up
parameter surface; the caller supplies values at execute time via a single
immutable property map; a new `binding.PropertyResolver` assembles that map
from layered sources (override / flag / env / config / default) with explicit
precedence and origin tracking. Retire the ambient `RuntimeEnvironment.Data`
/ `Properties` machinery. Test-first: write the starlark integration test
that models `writ adopt` before writing the infrastructure. `writ adopt`
running through the new binding model is the exit discriminator.

## Exit criterion

13.0(n) is complete when both of the following hold:

1. **All tests pass**, including a new starlark integration test under
   `cmd/devlore-test/testdata/binding/` that models the `writ adopt`
   workflow end-to-end (mkdir → move → link with bubbled parameter binding).
2. **`writ adopt` runs through the new infrastructure** and produces the
   same behavior the user observes today — the three nil-activation call
   sites in `cmd/writ/writ/commands.go` (`:1400` Mkdir, `:1410` Move,
   `:1422` Link) are gone, replaced by graph construction +
   `PropertyResolver` + `GraphExecutor.Run`.

When both criteria hold, `make build` is clean, `make vet` is clean,
`make test` is green across the full suite.

The remaining 5 nil-activation call sites in `cmd/writ/writ/migrate_cmd.go`
and `cmd/writ/writ/migrate/execute.go` (writ migrate) and the file
provider's defensive nil-activation paths are **out of scope** for 13.0(n) —
they will be addressed in a follow-on step that picks up after 13.0
closes, with the smaller-PR cadence.

## Background

### Today's slot model

Sealed three-variant `op.SlotValue` (`pkg/op/slot.go:40-46`):

| Variant | Resolved at | Source |
|---|---|---|
| `ImmediateValue{Value any}` | plan time | call-site literal or converted Go value |
| `PromiseValue{NodeRef, Slot}` | execute time | `results[NodeRef]` (sibling output) |
| `EnvironmentValue{Key string}` | execute time | `Properties.Property(Key)` reading `RuntimeEnvironment.Data` |

The `Properties` interface and `RuntimeEnvironment.Data map[string]any`
(`pkg/op/runtime_environment.go:48`, `:263`) provide an ambient property
bag populated at process startup from CLI flags, env vars, and user config
files (`docs/architecture/2.1-typed-slots.md:30-36`).
`plan.env("target_root")` creates an `EnvironmentValue` at plan time;
resolution defers to execute time via `Properties.Property(key)`.

### Why the model needs to change

Two structural problems block the bubble-up composition model that
flow-control combinators (`plan.subgraph`, `plan.choose`, `plan.gather`,
`plan.wait_until`) require:

1. **No declared parameter surface.** Today, `overrides` passed to
   `Execute(graph, overrides)` is an opaque `map[string]any`. The
   executor has no declared schema to check against. A missing required
   parameter surfaces as a nil dereference or a type-assertion panic deep
   inside a provider method, mid-execution, after side effects have
   partially committed.
2. **Ambient state is not composable.** `RuntimeEnvironment.Data` is
   process-wide; it cannot be re-scoped per subgraph. A subgraph composed
   inside a parent graph cannot supply a different value for its child's
   `EnvironmentValue` lookup — the lookup hits the shared `Data` bag.

Bubble-up requires both: a typed parameter surface declared at plan
time, and a property-resolution layer that scopes per-call rather than
per-process and tracks where each value came from.

### The unified model

Replace `EnvironmentValue` with `ParameterValue`. The slot's lookup
target moves from "process-wide property bag" to "this call's property
map." The tool main becomes the assembler of the property map — it reads
CLI flags, env, config, and produces a typed `map[string]Property` via
`PropertyResolver`; the executor sees only the resolved map. The
`Properties` abstraction goes away from the slot model entirely.
Subgraph composition becomes recursive bubble-up: a subgraph's
`ParameterValue` slots are bound by its parent graph's slot fill, which
itself may be `ImmediateValue`, `PromiseValue`, or another
`ParameterValue` (bubble continues all the way up to the root graph's
parameter surface).

## Design

### D1 — Sealed three-variant `SlotValue`

```go
type SlotValue interface {
    isSlotValue()
    Resolve(properties map[string]Property, results map[string]any) any
}

type ImmediateValue struct{ Value any }
type PromiseValue   struct{ NodeRef, Slot string }
type ParameterValue struct{ Name string }
```

Signature change: `Resolve(props Properties, results map[string]any)`
becomes `Resolve(properties map[string]Property, results map[string]any)`.
`ImmediateValue` and `PromiseValue` behavior is unchanged;
`ParameterValue.Resolve(properties, _)` returns
`properties[pv.Name].Value`.

`ParameterValue.Resolve` is reachable only after preflight has confirmed
every required `ParameterValue` has a matching property. If
`properties[Name]` is missing at `Resolve` time, that's an executor
invariant violation — **panic**.

### D2 — Parameter defaults

```go
type Parameter struct {
    Name    string
    Type    reflect.Type
    Default *any  // nil = required; non-nil = optional with *Default as fallback
}
```

Pointer-to-interface sentinel: `Default == nil` → required.
`Default != nil` → optional; `*Default` is the fallback value.

Validation at two points:

- **Preflight.** For each declared parameter with a non-nil `Default`,
  type-assert the default's runtime type against `Type`. Mismatches
  aggregate into the D5 envelope.
- **Slot-fill time.** The resolved `Property.Value` is type-asserted
  against `Type` before being placed into the slot. Defense-in-depth —
  preflight should catch type problems first, but a slot-fill
  assertion mismatch is a `PropertyResolver` bug and should fail
  immediately rather than panic inside a provider method.

### D3 — `ExecutableUnit.Parameters()` bubble-up

```go
type ExecutableUnit interface {
    // ... existing methods ...
    Parameters() []op.Parameter
}
```

- **Node.** `Parameters()` returns the method's parameters whose slot's
  `SlotValue` is `ParameterValue`. The `Parameter.Type` is the method's
  declared parameter type; the `Parameter.Default` is propagated from
  the declared default (or nil if the parameter is required).
- **Subgraph.** `Parameters()` walks the topological-root children
  (children with no incoming edges within the subgraph), unions their
  `Parameters()` outputs, and deduplicates by name. Multi-root collision
  (two roots declare the same parameter name with incompatible types)
  is a plan-time error.

This matches the Phase 7 invariant
(`docs/plans/extract-starlark-from-op.md:1215-1225`): "Executable units
declare a parameter surface via `Parameters() []op.Parameter`; for a
node this is `method.Parameters()`, for a subgraph it is the union (by
name) of every topological root's parameters." Phase 7 deferred the
implementation; 13.0(n) lands it.

### D4 — `Namespace`, `Origin`, `Property` types

The three value types in `pkg/binding` that carry resolution data:

```go
package binding

// Namespace identifies a property source category. Numeric values
// ascend with precedence — higher value beats lower.
type Namespace int

const (
    NamespaceUnknown  Namespace = iota // zero value; should not appear post-resolve
    NamespaceDefault                   // hardcoded fallback declared on plan.param
    NamespaceConfig                    // starlark or YAML config files
    NamespaceEnv                       // environment variables
    NamespaceFlag                      // command-line arguments
    NamespaceOverride                  // explicit runtime force
)

func (n Namespace) String() string {
    return [...]string{"unknown", "default", "config", "env", "flag", "override"}[n]
}

// Origin records where a resolved property value came from.
type Origin struct {
    Namespace Namespace
    Name      string // the lookup key that matched (e.g., "DEVLORE_WRIT_ROOT" or "config.star:12")
}

// String formats as "<namespace>:<name>". NamespaceUnknown renders as
// "unknown" alone since no name is meaningful.
func (o Origin) String() string {
    if o.Namespace == NamespaceUnknown {
        return "unknown"
    }
    return o.Namespace.String() + ":" + o.Name
}

// Property pairs a resolved value with its name and origin.
type Property struct {
    Name   string
    Value  any
    Origin Origin
}

func (p Property) String() string {
    return fmt.Sprintf("%s = %v [%s]", p.Name, p.Value, p.Origin)
}
```

#### `Name` convention per `Namespace`

| Namespace | `Origin.Name` = | Example `Origin.String()` |
|---|---|---|
| `NamespaceUnknown` | (empty) | `unknown` |
| `NamespaceDefault` | the parameter's declared name | `default:target_root` |
| `NamespaceConfig` | matched config-map key (or `file:line` if the loader tracks it) | `config:target_root`, `config:config.star:12` |
| `NamespaceEnv` (program-specific hit) | literal env var with program prefix | `env:DEVLORE_WRIT_TARGET_ROOT` |
| `NamespaceEnv` (global cascade hit) | literal env var with global prefix | `env:DEVLORE_TARGET_ROOT` |
| `NamespaceFlag` | the parameter name (Go/starlark form) | `flag:target_root` |
| `NamespaceOverride` | the override-map key | `override:layer` |

The `Name` field is opt-in rich — sources that have richer location
data (file:line, etc.) supply it; flat-map sources just supply the key.

### D5 — `PropertyResolver`

```go
package binding

type PropertyResolver struct {
    overrides   map[string]any  // NamespaceOverride source
    flags       map[string]any  // NamespaceFlag source
    envPrefix   string          // NamespaceEnv source — "DEVLORE_WRIT"
    globalPrefix string         // NamespaceEnv cascade source — "DEVLORE"
    config      map[string]any  // NamespaceConfig source

    resolved map[string]Property // populated by Resolve; nil until then
}

type Option func(*PropertyResolver)

func WithOverrides(m map[string]any) Option
func WithFlags(m map[string]any) Option
func WithEnvPrefix(programPrefix string) Option   // "DEVLORE_WRIT"; derives "DEVLORE" as global
func WithConfig(m map[string]any) Option

func NewPropertyResolver(opts ...Option) *PropertyResolver

// Resolve walks each parameter through the source precedence chain and
// populates the resolver's internal property map. Returns aggregated
// errors for missing required, type mismatch, default-type mismatch.
func (r *PropertyResolver) Resolve(parameters []op.Parameter) []error

// Properties returns the full resolved property map. Panics if called
// before Resolve.
func (r *PropertyResolver) Properties() map[string]Property

// Get returns a single resolved property. Panics if called before Resolve.
func (r *PropertyResolver) Get(name string) (Property, bool)
```

Construction via functional options; **no exported fields**. All sources
captured at construction; the resolver is read-only after that until
`Resolve` populates `resolved`. After `Resolve` the resolver is fully
read-only.

**Resolution order per parameter** (descending precedence, first hit wins):

1. `overrides[name]` → `NamespaceOverride`, `Name=name`.
2. `flags[name]` → `NamespaceFlag`, `Name=name`.
3. `os.Getenv(envPrefix + "_" + camelToSnake(name))` → `NamespaceEnv`,
   `Name="DEVLORE_WRIT_TARGET_ROOT"`.
4. `os.Getenv(globalPrefix + "_" + camelToSnake(name))` → `NamespaceEnv`,
   `Name="DEVLORE_TARGET_ROOT"`.
5. `config[name]` → `NamespaceConfig`, `Name=name`.
6. `*parameter.Default` if non-nil → `NamespaceDefault`, `Name=name`.

After picking the value, the resolver:

- Parses env-sourced values from string to the parameter's declared Go
  type (env always yields a string; other sources supply already-typed
  Go values).
- Type-asserts the picked value against `parameter.Type`. Mismatch →
  error.
- Records `resolved[name] = Property{Name, Value, Origin}`.

For each required parameter (`Default == nil`) that found nothing →
error wrapping the parameter name, target type, and the literal lookup
keys the resolver tried (`"missing required parameter "target_root"
(string); tried: override, flag, env:DEVLORE_WRIT_TARGET_ROOT,
env:DEVLORE_TARGET_ROOT, config"`).

`Resolve` returns aggregated `[]error`; the caller (the executor's
preflight pass) folds errors into the D5 envelope. Caller fetches the
resolved map via `Properties()` afterward.

`Get(name)` returns `(Property, bool)`. Panics if called before
`Resolve` — that's a programmer error.

### D6 — Override routing at `Execute()`

`GraphExecutor.Run` signature changes from `Run(graph)` to
`Run(graph, properties map[string]Property)`. Dispatch reaches a slot
with `ParameterValue{Name: X}` and resolves via `properties[X].Value`.
No `Properties` interface lookup remains anywhere in the executor.

Subgraph composition: when a parent graph's slot fill for a child
subgraph is `ParameterValue{Name: Y}`, the property map at the root
carries `Y`. The subgraph's matching slot resolves to the same value.
Bubble-up at runtime mirrors the bubble-up declared at plan time.

### D7 — Preflight integration

New preflight pass `bindProperties` added to `GraphExecutor.Run`,
joining `ResolvePending` (landed by 13.0(k) k.15) and future orphan /
type-check / platform passes. The executor doesn't construct the
resolver itself — the caller (tool main or test harness) constructs the
resolver, supplies sources, and passes the resolved property map to
`Run`. Preflight at this layer:

1. Compute `graph.Parameters()` — the root's bubble-up surface.
2. For each parameter:
   - Check `properties[name]` exists, OR parameter has a default.
   - If parameter has a default, type-assert `*Default` against `Type`.
   - If property is present, type-assert `properties[name].Value`
     against `Type`.
3. Collect errors; aggregate into the D5 envelope.
4. Fail fast on non-empty: graph transitions to `StateFailed`; executor
   returns `errors.Join(errs...)`.

Skipped in dry-run, matching `ResolvePending`'s convention
(`pkg/op/executor.go:79-84`).

The error path on resolver failures (caller called `resolver.Resolve` and
got errors) is upstream of the executor — caller handles those before
even calling `Run`. The executor's preflight catches the case where the
caller passed an incomplete property map (which shouldn't happen if the
caller used `PropertyResolver`, but the executor doesn't trust).

## Implementation phases

**Test-first ordering.** Phase 1 builds the foundations needed to author
the integration test that defines the contract: harness extensions, the
`plan.param` stub, the `executor.Run` signature change, and the `.star`
test files themselves. The tests parse and the harness invokes them
end-to-end; they fail at the executor with a "parameter not resolvable"
error until later phases land. Subsequent phases build the
infrastructure to satisfy the test.

Each phase is one or more commits on
`refactor/extract-starlark-from-op.phase-8`. Build stays green at every
commit boundary — Phase 1's failing-tests are intentional and the test
harness reports them as expected failures via `t.expect_error` or a
similar mechanism.

### Phase 8 — `writ adopt` migration

The three call sites in `runAdopt`'s helper `adoptFile`
(`cmd/writ/writ/commands.go`):

| Line | Method | Slots |
|---|---|---|
| 1400 | `fp.Mkdir(nil, destDir, 0o755, "")` | path, mode |
| 1410 | `fp.Move(nil, &file.Resource{SourcePath: …}, destPath)` | source, destination |
| 1422 | `fp.Link(nil, &file.Resource{SourcePath: …}, filePath)` | source, target |

Rewires to:

1. Construct a `*op.Graph` with three nodes (mkdir → move → link),
   wired by declaration order (no explicit edges).
2. Build a `binding.PropertyResolver` with
   `WithEnvPrefix("DEVLORE_WRIT")` plus caller-known overrides /
   flags from cobra parsing.
3. Call `resolver.Resolve(graph.Parameters())`; aggregate errors as
   needed.
4. Fetch `resolver.Properties()`.
5. Call `executor.Run(graph, properties)`.

Factor the pattern into a helper local to `cmd/writ/writ/` so the
three sites collapse to one-line invocations of the helper.

- **Tests:** the Phase 1 integration test exercises the same graph
  shape — running `writ adopt` on a real path tree should produce
  identical side effects to running the integration test plan. Add a
  writ-side unit/behavioral test covering the helper's graph
  construction.

## Test strategy

### Starlark-first integration tests

Phase 1 authors `.star` tests before any infrastructure exists. The
tests parse and the harness runs them end-to-end; they fail at the
executor until Phase 5 lands. From Phase 5 onward, the tests become
regression guards for retirement (Phase 7) and migration (Phase 8).

The test's role is dual:

1. **Specification.** The starlark surface defines the user-facing
   binding API — what the operator types, what errors they see, what
   data is available post-execution.
2. **Regression guard.** Once passing, the test catches any regression
   introduced by retirement or migration phases.

### Go unit coverage

Each new Go type / function in `pkg/binding` and the modified
`pkg/op` slot machinery gets unit tests. These cover edges the
integration test cannot reach economically — `camelToSnake` edge
cases, panic-on-pre-Resolve, multi-root collision detection, default
type validation against parameter type.

### Writ-side coverage

Phase 8 adds a writ-side test for the graph-construction helper. The
behavioral test for `writ adopt` continues to pass — same observable
filesystem behavior, different internal path.

## Open questions to lock during implementation

1. **`camelToSnake` rules for env keys.** Edge cases like `ID`
   (`I_D` or `ID`?), `URLPath` (`URL_PATH` or `U_R_L_PATH`?). Pick
   rules consistent with Go's idiomatic patterns or document a custom
   rule.
2. **Env-string parsing target types.** Resolver parses env values
   from string to the declared Go type at resolve time. Need a small
   parser registry: `string → int`, `string → bool`, `string →
   os.FileMode`, `string → time.Duration`, `string → string`. Initial
   set covers writ adopt's needs; extend as new types appear.
3. **Helper location in `cmd/writ/writ/`.** New file
   (`adopt_helper.go`?) or fold into `commands.go`? Lean toward new
   file — `commands.go` is already large (~1450 lines).
4. **Phase 6 `plan.param` kwarg parsing.** Codegen pathway or
   hand-written builtin? Likely hand-written — `plan.param` is a
   special-form builtin, not a generated provider method.
5. **`writ_adopt.star` fixture setup.** The test needs real
   filesystem operations (mkdir, move, link). Test harness already
   provides `t.tmp` and `t.write`; confirm these compose cleanly with
   `plan.file.*` for an integration test of this size.
