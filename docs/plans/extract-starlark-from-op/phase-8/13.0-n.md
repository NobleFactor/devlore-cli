# Plan: 13.0(n) — Variable binding infrastructure

## Summary

Rebuild the slot value model around a PowerShell-style variable binding system.
Graphs, subgraphs, and flow-control containers expose a bubble-up parameter
surface; the caller supplies values at execute time via a single immutable
variable map; a new [op.VariableResolver] assembles that map from layered
sources (override / flag / env / config / default) with explicit precedence and
provenance tracking via [op.VariableSource]. Retire the ambient
`RuntimeEnvironment.Data` / `Properties` machinery (done). Test-first: the
starlark integration tests that model `writ adopt` are authored before the
infrastructure that satisfies them. `writ adopt` running through the new
binding model is the exit discriminator.

The binding-layer types live in **`pkg/op`** alongside the rest of the
operations layer; there is no separate `pkg/binding`. Tool-side state (flag /
config / override maps + program name) lives in a new **`pkg/application`**
package that every tool (lore, star, writ, devlore-test) projects into.

`plan.run` invocation materialization is **in scope** for 13.0(n): the starlark
bridge collects [starlarkbridge.Invocation] entries while a `.star` plan
executes; `plan.run` walks the reachable invocation set and materializes
[op.Node] children onto the bound graph's root. Without this step the
integration-test contract authored under Phase 1 cannot be satisfied.

## Exit criterion

13.0(n) is complete when both of the following hold:

1. **All tests pass**, including the seven starlark integration tests under
   `cmd/devlore-test/devloretest/data/test_writ_adopt*.star` that model the
   `writ adopt` workflow (mkdir → move → link) with bubbled parameter binding,
   subgraph composition, precedence and origin assertions, and failure-mode
   coverage (missing-required, type-mismatch).
2. **`writ adopt` runs through the new infrastructure** and produces the same
   behavior the user observes today — the three nil-activation call sites in
   `cmd/writ/writ/commands.go` (`:1406` Mkdir, `:1416` Move, `:1428` Link) are
   gone, replaced by graph construction + [op.VariableResolver] +
   [op.GraphExecutor.Run].

When both criteria hold, `make build` is clean, `make vet` is clean, and
`make test` is green across the full suite.

The 5 remaining nil-activation call sites in `cmd/writ/writ/migrate_cmd.go`
and `cmd/writ/writ/migrate/execute.go` and the file provider's defensive
nil-activation paths are **out of scope** for 13.0(n) — they land in a
follow-on PR at the smaller-PR cadence.

## Scope

  [op.ImmediateValue], [op.PromiseValue], [op.VariableValue]. `EnvironmentValue`
  and the `Properties` interface are gone (deleted; no migration window).
  `SlotValue.Resolve` signature is `(variables map[string]Variable, results map[string]any) any`.

  the session's [op.Root] and is the only type that holds Close responsibility

## Background

### Today's slot model (pre-13.0(n))

Sealed three-variant `op.SlotValue` (the original shape, pre-rewrite):

| Variant | Resolved at | Source |
|---|---|---|
| `ImmediateValue{Value any}` | plan time | call-site literal or converted Go value |
| `PromiseValue{NodeRef, Slot}` | execute time | `results[NodeRef]` (sibling output) |
| `EnvironmentValue{Key string}` | execute time | `Properties.Property(Key)` reading `RuntimeEnvironment.Data` |

The `Properties` interface and `RuntimeEnvironment.Data map[string]any`
provided an ambient property bag populated at process startup from CLI flags,
env vars, and user config. `plan.env("target_root")` created an
`EnvironmentValue` at plan time; resolution deferred to execute time via
`Properties.Property(key)`.

### Why the model needed to change

Two structural problems blocked the bubble-up composition model that
flow-control combinators require:

1. **No declared parameter surface.** Today, "overrides" passed to
   `Execute(graph, overrides)` was an opaque `map[string]any`. The executor
   had no declared schema to check against. A missing required parameter
   surfaced as a nil dereference or a type-assertion panic deep inside a
   provider method, mid-execution, after side effects had partially
   committed.
2. **Ambient state was not composable.** `RuntimeEnvironment.Data` was
   process-wide; it could not be re-scoped per subgraph. A subgraph composed
   inside a parent graph could not supply a different value for its child's
   `EnvironmentValue` lookup — the lookup hit the shared `Data` bag.

Bubble-up requires both: a typed parameter surface declared at plan time, and
a variable-resolution layer that scopes per-call rather than per-process and
tracks where each value came from.

### The unified model (what landed)

Replace `EnvironmentValue` with `VariableValue`. The slot's lookup target moves
from "process-wide property bag" to "this session's variable map." The tool
main becomes the assembler of the variable map — it parses CLI flags, env,
and config and stamps them into an [application.Application]; the
[op.VariableResolver] reads from `app.Flags` / `app.Config` / `app.Overrides`
plus the process environment (prefix derived from `app.Name`) and produces a
typed `map[string]op.Variable` with provenance. The executor sees only the
resolved map. The `Properties` abstraction is gone from the slot model
entirely. Subgraph composition becomes recursive bubble-up: a subgraph's
[op.VariableValue] slots are bound by its parent graph's slot fill (which may
be `ImmediateValue`, `PromiseValue`, or another `VariableValue`); the bubble
continues all the way up to the root graph's parameter surface.

## Design

### D1 — Sealed three-variant `SlotValue` (landed)

```go
type SlotValue interface {
    isSlotValue()
    Resolve(variables map[string]Variable, results map[string]any) any
}

type ImmediateValue struct{ Value any }
type PromiseValue   struct{ NodeRef, Slot string }
type VariableValue  struct{ Name string }
```

`ImmediateValue` and `PromiseValue` ignore the variables map;
`VariableValue.Resolve` returns `variables[v.Name].Value` (nil-safe on map
absence).

`VariableValue.Resolve` is reachable only after preflight has confirmed every
required `VariableValue` has a matching entry in the variable map. If
`variables[Name]` is missing at Resolve time, today's behavior returns nil —
Phase 4's preflight ensures the case is unreachable.

### D2 — Parameter (landed)

`op.Parameter` (in [`pkg/op/action.go`](../../../../pkg/op/action.go)):

```go
type Parameter struct {
    Name     string
    Type     reflect.Type
    Optional bool
    Variadic bool
    Kwargs   bool
    Default  any
}
```

`Optional` discriminates required-vs-optional (not a sentinel on `Default`).
`Default` is the parameter's declared default value, already at
`Parameter.Type` (parsed once at announce time by `parseDefaultExpression`).
Variadic and Kwargs cannot carry a default — the grammar rejects
`*parts?` / `**kwargs?=foo`.

Validation:

- **At resolve time.** [op.assignToType] asserts every non-env source value
  against the declared `Type` via `reflect.TypeOf(raw).AssignableTo(declared)`.
  Mismatches return an error before the variable is stored. Nilability is
  handled per `reflect.Kind` (pointer / interface / slice / map / chan / func
  kinds accept nil; everything else does not).
- **At slot-fill time.** Defense-in-depth — preflight catches issues first,
  but a slot-fill assertion mismatch is a `VariableResolver` bug and fails
  immediately rather than panicking inside a provider method.

### D3 — `ExecutableUnit.Parameters()` (asymmetric by type)

```go
type ExecutableUnit interface {
    ID() string
    Parameters() []op.Parameter
}
```

`Parameters()` answers "what's the input surface of this executable?" — but
the answer is computed differently for Node and Subgraph because the
underlying shape differs.

- **Node.** `Parameters()` returns the **full method signature** — every
  parameter the bound Go method declares, populated by `Node.Bind` from
  `method.Parameters()`. This is the Go-level reality of what the node can
  accept. Slot fills (`ImmediateValue` / `PromiseValue` / `VariableValue`)
  do not filter this view; the method signature is the contract regardless
  of how individual slots are wired.

- **Subgraph.** `Parameters()` returns the **bubble-up variable surface** —
  the deduplicated set of `VariableValue` references across the subgraph's
  children's slots (recursing into nested subgraphs). Discovery is a
  graph-walk:

  1. For each child node, iterate its slots; for each slot whose `Value` is
     `VariableValue{Name: N}`, contribute `slot.Parameter` (Name + Type +
     Default) under name N.
  2. For each child subgraph, recurse — its `Parameters()` already yields
     deduplicated, type-checked bubble-ups; merge into the parent's working
     set.
  3. Dedup by name. Same-name + same-`Type` collapses to one entry. Same
     name + different `Type` is a plan-time error (the variable map at
     runtime is keyed by name and carries one value).

  No topological-root algorithm is needed — `ImmediateValue` and
  `PromiseValue` slots are intrinsically resolved (inline / sibling output)
  and do not contribute to the external surface. The walk discovers
  exactly the slot fills that demand external values: the variable map
  keys the subgraph needs the resolver to provide.

**Why the asymmetry is intentional.** For a Node — atomic, Go-method-bound
— the input surface IS the method signature. For a Subgraph — composite
— the input surface IS the set of bubbled-up variables. Both answer
"what's the input surface?" using the most natural representation for
their respective shape. `Parameters()` is a uniform interface accessor;
its semantic varies with the type because the type's nature varies.

Phase 1 lands a stub on `*Subgraph` (parameters left nil); Phase 3 lands
the graph-walk and the name-collision check. Node.Parameters() needs no
Phase 3 work — `Node.Bind` already does the right thing.

**Note on Gather and other flow combinators.** A gather operation reads
its body's bubble-up surface to know which variable name(s) to bind per
iteration. Since gather bodies are typically Subgraphs (or wrap Nodes in
an implicit one), `Subgraph.Parameters()` is the discovery surface for
gather. The full per-combinator design (Gather, Choose, WaitUntil) sits
in Step 13/14/15 and is foreshadowed in D11 (stack-frame execution model
+ subgraph ownership).

### D4 — `Variable`, `VariableSource`, `VariableSourceKind` (landed)

In [`pkg/op/variable.go`](../../../../pkg/op/variable.go):

```go
// Ascending precedence; higher beats lower.
type VariableSourceKind int

const (
    VariableSourceKindUnknown VariableSourceKind = iota
    VariableSourceKindDefault
    VariableSourceKindConfig
    VariableSourceKindEnv
    VariableSourceKindFlag
    VariableSourceKindOverride
)

type VariableSource struct {
    Kind VariableSourceKind
    Name string  // lookup key that matched
}

type Variable struct {
    Name   string
    Value  any
    Source VariableSource
}
```

All three carry `String()` for diagnostics. `VariableSource.String()` formats
as `<kind>:<name>` (or `"unknown"` for the zero value). `Variable.String()`
formats as `<name> = <value> [<source>]`.

`Name` convention per `Kind`:

| Kind | `Name` = | Example `String()` |
|---|---|---|
| `Unknown` | (empty) | `unknown` |
| `Default` | parameter name | `default:target_root` |
| `Config` | config-map key | `config:target_root` |
| `Env` | literal env var with program prefix | `env:WRIT_TARGET_ROOT` |
| `Flag` | normalized snake-case flag name | `flag:target_root` |
| `Override` | override-map key | `override:layer` |

### D5 — `VariableResolver` (skeleton landed; cascade in Phase 2)

In [`pkg/op/variable_resolver.go`](../../../../pkg/op/variable_resolver.go):

```go
type VariableResolver struct {
    app      *application.Application
    resolved map[string]Variable
}

func NewVariableResolver(app *application.Application) *VariableResolver

func (r *VariableResolver) EnvPrefix() string
func (r *VariableResolver) Get(name string) (Variable, bool)
func (r *VariableResolver) Variables() map[string]Variable
func (r *VariableResolver) Resolve(parameters []Parameter) []error
```

The resolver captures a pointer to the [application.Application] at
construction. At resolve time it reads `app.Flags`, `app.Config`,
`app.Overrides` directly (not snapshots — late binding lets per-command flag
adjustments take effect). The env-var prefix is derived from
`strings.ToUpper(app.Name) + "_"`.

**Resolution order per parameter** (descending precedence; first hit wins):

1. `app.Overrides[name]` → `VariableSourceKindOverride`.
2. `app.Flags[name]` → `VariableSourceKindFlag`. `app.Flags` keys are
   snake-case (normalized at [application.NewApplication] time —
   `dry-run` → `dry_run`, `target-root` → `target_root`); the resolver
   matches the parameter name verbatim. Framework code that reads specific
   flags (e.g., [application.Application.DryRun] for `dry-run`) likewise
   reads the snake-case key (`Flags["dry_run"]`).
3. `os.Getenv(EnvPrefix() + camelToSnake(name))` → `VariableSourceKindEnv`.
   `camelToSnake` uses Go's idiomatic greedy-acronym rule: consecutive
   uppercase letters form a single acronym; an acronym ends when a
   lowercase letter follows. Examples: `targetRoot` → `TARGET_ROOT`,
   `sourceURL` → `SOURCE_URL`, `URLPath` → `URL_PATH`, `UserID` →
   `USER_ID`. Mixed acronyms like `IPv4Address` produce slightly imperfect
   output (`I_PV4_ADDRESS`) and are tolerated; the rule serves the 99%
   case at zero maintenance cost.
4. `app.Config[name]` → `VariableSourceKindConfig`.
5. `Parameter.Default` (when `Optional == true` and `Default != nil`) →
   `VariableSourceKindDefault`.

After picking the value, the resolver:

- For env-sourced values, wraps the raw string in a private `envValue` type
  that implements [op.SourceConverter]. The [op.Convert] cascade picks the
  wrapper up at step 5 (source-side opt-in); `envValue.ConvertTo` handles
  primitives (bool / int* / uint* / float* / string via the existing
  `Kind` switch shared with `parseDefaultExpression`), stdlib special-cases
  (`time.Duration` via `time.ParseDuration`, `os.FileMode` via
  `strconv.ParseUint(base 0)`), then probes the target for
  [encoding.TextUnmarshaler], then falls through to `json.Unmarshal` for
  arbitrary Go types. `envValue.CanConvertTo` returns `false` for targets
  implementing [op.Resource] so [op.Convert] step 7 (registered Resource
  construction) wins the routing — env strings for Resource-typed
  parameters feed the same constructor path that tag URIs use today.
- For non-env sources, type-asserts the picked value against
  `parameter.Type` via [op.assignToType]. Mismatch → error.
- Records `resolved[name] = Variable{Name, Value, Source}`.

[op.Convert] itself is not modified. Adding a string-source step there would
silently parse stray starlark string literals reaching primitive-typed
slots — the codebase already documented this layering decision at
[`pkg/op/parameter.go:236`](../../../../pkg/op/parameter.go). `envValue` is
a distinct type from `string`; starlark slot fills produce plain `string`
and continue to error correctly when a primitive target is required.

For each required parameter (`Optional == false`) that found nothing →
aggregated error naming the parameter, target type, and the literal lookup
keys the resolver tried.

`Get` and `Variables` panic if called before `Resolve` — that's a programmer
error. After `Resolve` the resolver is fully read-only.

Phase 1 skeleton: `Resolve` initializes the resolved map to empty and
returns no errors. Phase 2 lands the cascade walk.

### D6 — `RegisterParameter` cascade on the runtime environment (landed)

Providers that need a binding-layer parameter immediately (at construct time,
not at preflight) call `env.RegisterParameter(p)`. The method:

1. Records the parameter in `env.declaredParameters`. Re-registration with
   the same name + same `Type` is idempotent; same name + different `Type`
   returns a type-mismatch error.
2. Walks the cascade: `Overrides → Flags → Env (deferred) → Config → Default`
   on `env.Application` directly (the resolver is shared, but
   `RegisterParameter` is a synchronous helper that doesn't require a full
   `Resolve` pass to inspect a single name).
3. Type-asserts the picked value via [op.assignToType]. Mismatch → error
   without overwriting state.
4. Stores the result in `env.variables[name]` so subsequent
   [op.RuntimeEnvironment.VariableByName] calls return it.

This shape lets providers participate in variable binding **before** the
graph's preflight pass — the four star providers (`config`, `setup`, `goast`,
`commands`) currently rely on this path for tool-side parameters.

The env source step in `RegisterParameter` is a deliberate skip until the
resolver's full env-string-parsing logic lands in Phase 2.
[op.VariableResolver.Resolve] is the canonical path for graph-time parameter
resolution; `RegisterParameter` is the construct-time fast path.

### D7 — Session ownership, planning, execution (landed)

```go
func NewRuntimeEnvironment(ctx context.Context, spec *RuntimeEnvironmentSpec) *RuntimeEnvironment
```

`Plan` and `GraphExecutor` are the two session-owners. Each builds an env

resolver pass before dispatch; today's callers pass `nil`).

### D8 — Plan terminals: `plan.assemble`, `plan.run`, `plan.save`, `plan.load`, `plan.clear` (Phase 5)

Four create-and-X terminals plus a registry-reset helper. The four-terminal
model and the multi-graph-from-one-registry pattern were settled during the
2026-05 spec session that closes 13.0(k); this section captures the
settlements load-bearing for Phase 5.

```python
# Creation / composition (returns an immutable Graph)
graph = plan.assemble(body=[...], retry_policy=...)

# I/O (symmetric)
plan.save(graph, "workflow.json")
graph = plan.load("workflow.json")

# Execution — overloaded
result = plan.run(body=[...])  # compose + run
result = plan.run(graph)       # run pre-built graph

# Reset the bridge's plan state
plan.clear()
```

**`plan.assemble(body=[…], …kwargs)`** — the canonical Graph constructor.

- `body=` lists the parentless invocations to root in `graph.Root.Children`.
  If omitted, defaults to ALL parentless invocations in the bridge's
  registry.
- `body=` elements with an existing `parentID` (already included in some
  subgraph's body) error — nested invocations cannot be promoted to root.
- Remaining kwargs populate `graph.Root.Frame.Variables` (the root frame's
  bindings).
- Reserved kwargs `retry_policy=` and `error_action=` stamp those fields on
  `graph.Root`.
- Reachability walk follows: child containment (`body`), PromiseValue
  NodeRef edges, Resource `producerID` edges, and `error_action` edges.
- **Orphan check.** Every Invocation in the bridge's registry must be
  reachable in the assembled Graph. If any registry entry is not reached,
  `plan.assemble` errors and returns nil. To assemble divergent graphs from
  the same registry, call `plan.clear()` between assembles.
- Children are placed in `graph.Root.Children` in **topological order**.

**`plan.run`** — overloaded.

- `plan.run(body=[…], …)` is shorthand for `plan.run(plan.assemble(body=[…], …))`.
- `plan.run(graph)` accepts a pre-built Graph (from `plan.assemble` or
  `plan.load`).
- Returns the graph's return value.

**`plan.save(graph, path)`** — serializes a Graph to `path`. Returns
`None`. Saving an orphan-bearing Graph is unreachable (assemble already
refused), so plan.save trusts the Graph it receives.

**`plan.load(path)`** — deserializes a Graph from `path`. Returns the
Graph. The inverse of `plan.save`.

**`plan.clear()`** — resets the plan to its initial state: purges the
bridge's invocation registry, resets parent-stamping state, clears bridge-
internal caches. Does NOT invalidate previously-assembled Graphs (those
are immutable artifacts the user holds). Does NOT trigger orphan detection
(orphans are detected at `plan.assemble`, not at `plan.clear`).

**Multi-graph from one registry.** The invocation registry is a template
pool. The same Invocation can be referenced from multiple `plan.assemble`
calls' `body=` lists, producing per-Graph `op.Node` instances:

```python
cleanup_foo = plan.file.remove("foo")
cleanup_bar = plan.file.remove("bar")
prod_graph    = plan.assemble([cleanup_foo, cleanup_bar], environment="prod")
staging_graph = plan.assemble([cleanup_foo, cleanup_bar], environment="staging")
```

Both assembles cover the full registry contents (no orphans), and produce
graphs with distinct root frames (`environment` differs per assemble).
Per-Graph `op.Node` instances are settled for now (execution state stays
on the Node struct). plan.clear()'s implementation phase migrates
execution state off the Graph if shared-template Nodes become desirable.

**Parent identity is by ID, not pointer.** Pointers don't serialize.
`graph.Root.ID` is a constant ("root"), so a child invocation parented to
two graphs both stamp `parentID = "root"` — idempotent. Containment
(`Subgraph.Children []NodeID`) is the canonical serialized form; the
runtime parent reference is derivable from containment at load time. The
parent-pointer landing in Phase 3 is refactored to this ID-based model in
**Phase 3.5** (precursor commit before Phase 5 starts).

**Reserved kwargs** (consistent across containers AND `plan.assemble`):

| Kwarg | Role | Destination |
|---|---|---|
| `body=` | Structural children | `Subgraph.Children`. Container methods stamp each child's `parentID`; `plan.assemble` does not (root-level children remain parentless until consumed). |
| `items=` | Iteration / dispatch data | The Subgraph's `items` parameter, passed at execute time. |
| `error_action=` | Failure handler — itself an `ExecutableUnit` | `executableUnit.errorAction`. Default: `flow.Provider.Failed` (built-in sentinel, no registry entry). |
| `retry_policy=` | Retry behavior | `executableUnit.retryPolicy`. |
| (anything else) | Frame bindings | The invoked subgraph's `Frame.Variables`. |

**Containers.** The four containers (`subgraph`, `choose`, `gather`,
`wait_until`) share the uniform Go-side signature
`(items []any, kwargs map[string]any)` and all materialize to
`op.Subgraph`. The receiver-type dispatch at execute time differs per
container (subgraph runs all children sequentially; choose runs one
selected branch; gather runs body once per item with per-iteration child
frames; wait_until polls). Per-iteration frame minting is execute-time
behavior owned by Steps 14/15; Phase 5 produces only the static graph.

**`op.Node` is a goroutine-safe template.** Per the project memory
[execution units are goroutine-safe templates], a single materialized
`op.Node` can be safely dispatched concurrently. Iteration/parallel
constructs dispatch one body N times with per-dispatch frame minting —
the body is never cloned per iteration.

### D9 — Override routing at `Execute()` (landed signature; Phase 4 wiring)

`GraphExecutor.Run` signature is `Run(graph, variables map[string]Variable)`.
The executor stashes the variable map on its receiver; dispatch reaches a
slot with `VariableValue{Name: X}` and resolves via `variables[X].Value`. No
`Properties` interface lookup remains anywhere in the executor.

Subgraph composition: when a parent graph's slot fill for a child subgraph
is `VariableValue{Name: Y}`, the variable map at the root carries `Y`. The
subgraph's matching slot resolves to the same value. Bubble-up at runtime
mirrors the bubble-up declared at plan time.

Phase 4 wires the preflight pass that populates `variables` from the
executor's own [op.VariableResolver] before dispatch. Today's callers (writ
adopt, tests) pass `nil` or supply a pre-resolved map directly.

### D10 — Preflight integration (Phase 4, landed)

New preflight pass [GraphExecutor.bindVariables] added to
[GraphExecutor.Run], running **before** [ResourceCatalog.ResolvePending]
(landed by 13.0(k) k.15) and ahead of future orphan / type-check /
platform passes. The executor doesn't construct the resolver itself — the
env's `variableResolver` (wired by [NewRuntimeEnvironment] from
`spec.Application`) is the one source. The pass:

1. Compute `graph.Parameters()` — the root subgraph's bubble-up variable
   surface (per D3).
2. Call `env.variableResolver.Resolve(env, params)`. The resolver walks
   each parameter through the cascade (Override → Flag → Env → Config →
   Default), type-asserts source values, and aggregates errors for
   missing-required / type-mismatch / default-type-mismatch.
3. If `len(errs) > 0`: `graph.State = StateFailed`; return
   `errors.Join(errs...)` to the caller. No dispatch runs.
4. On success: merge `resolver.Variables()` into `env.variables`. If
   the caller passed a non-nil `callerVariables` map to `Run`, layer
   those entries on top as the highest-priority override (test injection
   surface; production callers pass nil).
5. Mirror `env.variables` into `e.variables` so [Node.ResolveSlots] sees
   the same map [GraphExecutor.bindVariables] just populated.

**Runs in every mode, including dry-run.** Variable resolution is pure —
it reads in-memory `Application` maps and the process env; no
filesystem or network probes. Dry-run output that renders slot values
needs them resolved, so the pass runs unconditionally. This is the one
divergence from `ResolvePending`'s convention (which skips dry-run
because catalog resolution touches target-machine state).

### D11 — Stack-frame execution model + subgraph ownership (forward-looking)

The execution model 13.0(n) prepares the ground for — and that Step 13/14/15
implement in full — is **stack-frame-based dispatch with per-type subgraph
executors and explicit ownership pointers**. 13.0(n) lands the ownership
pointer in Phase 3 as forward-compat scaffolding; the frame machinery and
the per-type executors are downstream work. This section documents the
target so the Phase 3 scaffolding doesn't need a backwards-incompatible
rename later.

#### Stack frames

Every invocation of any subgraph (root, plain composition, gather body
iteration, choose branch, wait_until polling cycle) creates a **frame**:

```go
type Frame struct {
    Subgraph  *Subgraph          // the static template being invoked
    Parent    *Frame             // the calling frame; nil only at the root invocation
    Variables map[string]Variable // overlay on Parent.Variables; lookups fall through
    Results   map[string]any     // node ID → output, scoped to this invocation
    Attempts  []Attempt          // per-invocation retry history
    Status    SubgraphStatus     // pending → completed | failed | rolled_back | skipped
}
```

Frames are dynamic; the `Subgraph` struct is a **template** (its current
`Status`, `Attempts`, `State` fields are conceptually frame state, not
template state, and will move to the frame in Step 13). The same Subgraph
template can be invoked many times concurrently — gather iterating 16
items pushes 16 concurrent frames over the same body subgraph; each frame
isolates its own state.

**What frames solve:**

- **Per-iteration variable scoping.** Gather binds `iter.Variables[name] =
  item_i` for each iteration's frame. Other iterations see their own
  bindings via their own frames; the parent's `Variables` is shadowed
  read-only.
- **Variable resolution scope chain.** `VariableValue.Resolve` walks the
  frame chain — current frame first, then parent, then grandparent —
  until it finds a binding for the name or runs out. Nested subgraphs can
  declare local variables that shadow outer ones; gather's iteration
  variable is one example of that.
- **Concurrent state isolation.** Each iteration of a gather body has
  its own frame; its Status / Attempts / Results live there. No race on
  shared `Subgraph.Status` because no concurrent writer touches the
  template — each writes its own frame.
- **Compensation and rollback.** Unwinding walks the frame stack; each
  frame knows what it consumed (Variables) and produced (Results), so
  compensation can be sequenced precisely without re-inspecting the
  shared template.

**What stays per-executor (correctly so):**

- **Retry policy.** Gather retries per-iteration. Choose re-chooses
  (doesn't retry the same branch). Wait_until polls (a retry-of-condition
  shape). Root retries the whole subgraph. The MECHANICS (frame.Attempts)
  are uniform; the POLICY is type-specific and rightly so — that's the
  combinator's defining behavior.
- **Status emission.** Each combinator emits its own progress shape
  (gather: per-iteration progress; choose: which branch was taken;
  wait_until: each poll). Frames provide the context label; the
  per-executor decides what to say.
- **Dispatch logic.** The whole point of the per-type split. Frames
  don't subsume it.

#### Subgraph ownership

Every executable unit (Node or Subgraph) carries a **parent ID** to its
containing Subgraph (or to the graph root):

```go
type executableUnit struct {
    id          string
    parameters  []Parameter
    parentID    string         // empty only for the graph root; references enclosing Subgraph's ID
    retryPolicy RetryPolicy    // every unit; default zero-value policy = no retry
    errorAction ExecutableUnit // every unit; default flow.Provider.Failed sentinel
}
```

By-ID rather than by-pointer because pointers don't serialize — the same
constraint that drives the Edge model. Containment
(`Subgraph.Children []NodeID`) is the canonical serialized form; the
runtime parent reference is derivable from containment at load time.
`graph.Root.ID` is a constant ("root"), so a child invocation parented to
two graphs both stamp `parentID = "root"` — idempotent (cf. D8's
multi-graph reuse pattern).

The parent ID is stamped at the moment a child is added to a subgraph —
via the `Subgraph.AddChild` method (Phase 3, refactored from pointer to ID
in Phase 3.5) that centralizes the wiring and replaces today's direct
`Children = append(...)` sites in `graph.go`.

The ownership pointer drives dispatch in the per-type-executor world:

- The top-level executor dispatches **only the root**. It never asks "is
  this branch?"; it just walks `root.Children`.
- Each subgraph's per-type executor dispatches **its own children**, in
  whatever frame-construction pattern that combinator requires.
- Recovery-site walks ownership upward to find the saga-owning frame.
- Variable resolution uses the frame chain (dynamic), not the ownership
  chain (static); the two can diverge (a body subgraph statically owned
  by some parent but dynamically invoked by gather has a frame parented
  to gather's frame).

The current `Subgraph.Branch bool` flag is **deprecated** by this model.
Today it means "choose owns dispatch of me." Tomorrow every flow
combinator owns dispatch of its body; the Branch flag becomes redundant.
13.0(n) leaves Branch in place (it has current callers in
`flow.Provider.Choose`); Step 13 removes it as part of the choose
executor refactor.

#### What 13.0(n) implements

Phase 3 landed the **forward-compatibility scaffolding** as parent
pointers; Phase 3.5 refactors it to the by-ID model the serialization
contract requires; Phase 5 consumes the by-ID model for materialization.
After 13.0(n) closes:

- `parentID string` field on `executableUnit` (refactored from
  `parent *Subgraph` in Phase 3.5).
- `retryPolicy` and `errorAction` fields on `executableUnit` (Phase 3.5
  alongside the parent-ID refactor).
- `(s *Subgraph) AddChild(child SubgraphChild)` stamps the parent ID
  idempotently when wiring children.
- Bridge-side parent-stamping: each Invocation listed in a `body=` kwarg
  on a container call gets its `parentID` stamped to the calling
  subgraph's ID at .star dispatch time (Phase 3.5).
- The four-terminal `plan.assemble` / `plan.run` / `plan.save` /
  `plan.load` / `plan.clear` API (Phase 5, per D8).

Full frame plumbing and the per-type subgraph executors are **out of

### Phase 5 — Plan terminals: `plan.assemble`, `plan.run`, `plan.save`, `plan.load`, `plan.clear`

Implement per D8.

**Shared materialization pipeline** (single function consumed by all
terminals that take `body=`):

1. **Reachability walk** rooted at `body=` (or all parentless invocations
   when `body=` is omitted). Follow child containment, PromiseValue
   NodeRef edges, Resource `producerID` edges, and `error_action` edges.
2. **Orphan check.** Every Invocation in the bridge's registry must be
   marked by the walk. Any unreached registry entry is an orphan error;
   `errors.Join` into the aggregated envelope.
3. **Node materialization.** For each reached Invocation, build the
   corresponding `op.Node` or `op.Subgraph` (container methods become
   `op.Subgraph` by the uniform Go signature; leaf methods become
   `op.Node`). The cached method + slot map from the Invocation struct
   drives slot construction. Slots reference variables as
   `op.VariableValue`, references to other invocations as `PromiseValue`,
   and inline values as `ImmediateValue`.
4. **Edge materialization.** Emit edges from PromiseValue NodeRefs and
   Resource producerIDs.
5. **Children placement.** Topologically sort the root-level reached set
   and insert into `graph.Root.Children` via `AddChild` (which stamps
   `parentID = "root"` per Phase 3.5).
6. **Frame setup.** Populate `graph.Root.Frame.Variables` with the
   kwargs that fell through the bridge's reserved-kwarg classification.
   `retry_policy=` / `error_action=` stamp `graph.Root`'s fields.

**Terminal wrappers:**

- `plan.assemble(body=[…], …kwargs)` → run shared pipeline; on success

- `plan.run(body=[…], …kwargs)` → shared pipeline → executor.Run(graph)

- `plan.run(graph)` → executor.Run(graph) directly (skip materialization).
- `plan.save(graph, path)` → serialize graph to path; return None.
- `plan.load(path)` → deserialize; return Graph.
- `plan.clear()` → purge bridge invocation registry, reset parent-
  stamping state, clear bridge caches. Does NOT touch user-held Graphs.
  Does NOT trigger orphan detection.

## Locked design decisions

Decisions taken during plan finalization (2026-05-15) and locked in the
sections above:

1. **`camelToSnake` rule for env keys.** Greedy acronym (Go-idiomatic):
   consecutive uppercase letters form a single acronym; an acronym ends
   when a lowercase letter follows. `URLPath → URL_PATH`, `targetRoot →
   TARGET_ROOT`. Mixed acronyms (`IPv4Address → I_PV4_ADDRESS`) are
   tolerated. Documented under D5 step 3.
2. **`Application.Flags` key form.** Snake-case, normalized at
   [application.NewApplication] time. `dry-run → dry_run`,
   `target-root → target_root`. The resolver matches the parameter name
   verbatim. Framework code reading specific flags
   ([application.Application.DryRun]) likewise reads the snake-case key.
   Documented under D5 step 2.
3. **Env-string parsing strategy.** Private `envValue` type implements
   [op.SourceConverter]; the resolver wraps raw env strings before calling
   [op.Convert]. The cascade picks up `envValue` at step 5; the wrapper
   handles primitives via the shared `Kind` switch with
   `parseDefaultExpression`, then stdlib special-cases (`time.Duration`,
   `os.FileMode`), then [encoding.TextUnmarshaler], then `json.Unmarshal`
   as the universal fallback. `envValue.CanConvertTo` returns false for
   [op.Resource] targets so step 7 (registered Resource construction)
   wins routing. [op.Convert] itself is not modified — the codebase's
   prior layering decision at [`pkg/op/parameter.go:236`](../../../../pkg/op/parameter.go)
   stands. Documented under D5.
4. **Phase 6 layout.** `cmd/writ/writ/adopt_cmd.go` + `cmd/writ/writ/adopt/`
   subpackage (with `adopt.go`, `plan.go`, `execute.go`) matching the
   existing `migrate_cmd.go` + `migrate/` precedent (and six other
   subpackages: `identity/`, `reconcile/`, `segment/`, `snapshot/`,
   `tree/`). Documented under Phase 6.
5. **Each writ adopt CLI argument is an [op.VariableValue] slot.** No
   user-controlled input lives in an [op.ImmediateValue]. Documented
   under Phase 6.
6. **Execution model — stack frames + subgraph ownership + per-type
   executors.** Every subgraph invocation (root, plain composition,
   gather body iteration, choose branch, wait_until polling cycle)
   creates a frame carrying its own `Variables` overlay, `Results` map,
   and `Attempts` history. Frames isolate state for concurrent
   invocations of the same subgraph template. Subgraphs own their
   children via an explicit `parent *Subgraph` pointer; per-type
   executors (root / gather / choose / wait_until / plain) dispatch
   their own children's frames. The current `Subgraph.Branch` flag is
   deprecated by this model and stays in place only until Step 13.
   Documented under D11. 13.0(n) Phase 3 lands the parent pointer for
   forward-compat; full frame plumbing and the per-type executors are
   Step 13/14/15 work.

7. **`Node.Parameters()` vs `Subgraph.Parameters()` — asymmetric by type.**
   `Node.Parameters()` returns the full Go-method signature (every
   declared parameter, regardless of slot fill). `Subgraph.Parameters()`
   returns the bubble-up variable surface (the deduped, type-checked set
   of `VariableValue` references walked across children's slots, recursing
   into nested subgraphs). Both answer "what's the input surface?" using
   the most natural representation for the type. `Parameters()` stays on
   the `ExecutableUnit` interface; no separate `UnboundSlots()` /
   `BubbleUp()` accessor — variables ARE the bubble-up. Documented under
   D3.
