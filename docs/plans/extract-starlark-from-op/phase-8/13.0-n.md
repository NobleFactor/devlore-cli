# Plan: 13.0(n) — Parameter binding infrastructure

## Summary

Rebuild the slot value model around a PowerShell-style parameter binding system:
graphs, subgraphs, and flow-control containers expose a bubble-up parameter
surface; the caller supplies values at execute time via a single immutable
override map; a new `PropertyResolver` assembles that map from layered sources
(caller / program env / global env / config / default) with explicit precedence
and origin tracking. Retire the ambient `RuntimeEnvironment.Data` / `Properties`
machinery. Test-first: write the starlark integration test plan that models
`writ adopt` before writing the infrastructure. `writ adopt` running through
the new binding model is the exit discriminator.

## Exit criterion

13.0(n) is complete when both of the following hold:

1. **All tests pass**, including a new starlark integration test under
   `cmd/devlore-test/testdata/binding/` that models the `writ adopt`
   workflow end-to-end (mkdir → move → link with bubbled parameter binding).
2. **`writ adopt` runs through the new infrastructure** and produces the same
   behavior the user observes today — the three nil-activation call sites in
   `cmd/writ/writ/commands.go` (`:1400` Mkdir, `:1410` Move, `:1422` Link) are
   gone, replaced by graph construction + `PropertyResolver` + `GraphExecutor.Run`.

When both criteria hold, `make build` is clean, `make vet` is clean, `make test`
is green across the full suite, and the defensive nil-activation paths in
`pkg/op/provider/file/provider.go` are deleted.

The remaining 5 nil-activation call sites in `migrate_cmd.go` and
`migrate/execute.go` (writ migrate) are out of scope for 13.0(n) — they
will be addressed in a follow-on step that picks up after 13.0 closes, with
the smaller-PR cadence.

## Background

### Today's slot model

Sealed three-variant `op.SlotValue` (`pkg/op/slot.go:40-46`):

| Variant | Resolved at | Source |
|---|---|---|
| `ImmediateValue{Value any}` | plan time | call-site literal or converted Go value |
| `PromiseValue{NodeRef, Slot}` | execute time | `results[NodeRef]` (sibling output) |
| `EnvironmentValue{Key string}` | execute time | `Properties.Property(Key)` reading `RuntimeEnvironment.Data` |

The `Properties` interface and `RuntimeEnvironment.Data map[string]any`
(`pkg/op/runtime_environment.go:48`, `:263`) provide an ambient property bag
populated at process startup from CLI flags, env vars, and user config files
(`docs/architecture/2.1-typed-slots.md:30-36`). `plan.env("target_root")`
creates an `EnvironmentValue` at plan time; resolution defers to execute time
via `Properties.Property(key)`.

### Why the model needs to change

Two structural problems block the bubble-up composition model that
flow-control combinators (`plan.subgraph`, `plan.choose`, `plan.gather`,
`plan.wait_until`) require:

1. **No declared parameter surface.** Today, `overrides` passed to
   `Execute(graph, overrides)` is an opaque `map[string]any`. The executor has
   no declared schema to check against. A missing required parameter surfaces
   as a nil dereference or a type-assertion panic deep inside a provider
   method, mid-execution, after side effects have partially committed.
2. **Ambient state is not composable.** `RuntimeEnvironment.Data` is
   process-wide; it cannot be re-scoped per subgraph. A subgraph composed
   inside a parent graph cannot supply a different value for its child's
   `EnvironmentValue` lookup — the lookup hits the shared `Data` bag.

Bubble-up requires both: a typed parameter surface declared at plan time, and
an override mechanism that scopes per-call rather than per-process.

### The unified model

Replace `EnvironmentValue` with `ParameterValue`. The slot's lookup target
moves from "process-wide property bag" to "this call's override map." The tool
main becomes the assembler of the override map — it reads CLI flags, env,
config, and produces the map; the executor sees only the resolved map. The
`Properties` abstraction goes away from the slot model entirely. Subgraph
composition becomes recursive bubble-up: a subgraph's `ParameterValue` slots
are bound by its parent graph's slot fill, which itself may be `ImmediateValue`,
`PromiseValue`, or another `ParameterValue` (bubble continues all the way up
to the root graph's parameter surface).

## Design

### D1 — Sealed three-variant `SlotValue`

```go
type SlotValue interface {
    isSlotValue()
    Resolve(overrides, results map[string]any) any
}

type ImmediateValue struct{ Value any }
type PromiseValue   struct{ NodeRef, Slot string }
type ParameterValue struct{ Name string }
```

Signature change: `Resolve(props Properties, results map[string]any)` becomes
`Resolve(overrides, results map[string]any)`. `ImmediateValue` and
`PromiseValue` behavior is unchanged; `ParameterValue.Resolve(overrides, _)`
returns `overrides[pv.Name]`.

`ParameterValue.Resolve` is reachable only after preflight has confirmed every
required `ParameterValue` has a matching override key. If `overrides[Name]` is
missing at `Resolve` time, that's an executor invariant violation — **panic**.

### D2 — Parameter defaults

```go
type Parameter struct {
    Name    string
    Type    reflect.Type
    Default *any   // nil = required; non-nil = optional with *Default as fallback
}
```

Pointer-to-interface sentinel: `Default == nil` → required. `Default != nil` →
optional; `*Default` is the fallback value.

Validation at two points:

- **Preflight.** For each declared parameter with a non-nil `Default`,
  type-assert the default's Go type against `Type`. Mismatches aggregate into
  the D5 envelope.
- **Slot-fill time.** The resolved value (override / env / config / default)
  is type-asserted against `Type` before being placed into the slot.
  Defense-in-depth — preflight should catch type problems first, but a slot-
  fill assertion mismatch is a `PropertyResolver` bug and should fail
  immediately rather than panic inside a provider method.

### D3 — `ExecutableUnit.Parameters()` bubble-up

```go
type ExecutableUnit interface {
    // ... existing methods ...
    Parameters() []op.Parameter
}
```

- **Node.** `Parameters()` returns the method's parameters whose slot's
  `SlotValue` is `ParameterValue`. The `Parameter.Type` is the method's
  declared parameter type; the `Parameter.Default` is propagated from the
  declared default (or nil if the parameter is required).
- **Subgraph.** `Parameters()` walks the topological-root children (children
  with no incoming edges within the subgraph), unions their `Parameters()`
  outputs, and deduplicates by name. Multi-root collision (two roots declare
  the same parameter name with incompatible types) is a plan-time error.

This matches the Phase 7 invariant
(`docs/plans/extract-starlark-from-op.md:1215-1225`): "Executable units declare
a parameter surface via `Parameters() []op.Parameter`; for a node this is
`method.Parameters()`, for a subgraph it is the union (by name) of every
topological root's parameters." Phase 7 deferred the implementation; 13.0(n)
lands it.

### D4 — `PropertyResolver`

```go
package binding

type PropertyResolver struct {
    callerOverrides map[string]any
    envPrefix       string  // "DEVLORE_WRIT" — derived from WithEnvPrefix
    globalEnvPrefix string  // "DEVLORE"      — derived from WithEnvPrefix
    configValues    map[string]any

    resolved map[string]property  // populated by Resolve; nil until then
}

// property is the per-parameter resolved record. Unexported.
type property struct {
    value  any
    origin Origin
}

type Origin struct {
    Source string  // "caller" | "env:DEVLORE_WRIT_ROOT" | "env:DEVLORE_ROOT" | "config:root" | "default"
}

type Option func(*PropertyResolver)

func WithCallerOverrides(m map[string]any) Option
func WithEnvPrefix(programPrefix string) Option   // "DEVLORE_WRIT"; derives "DEVLORE" as global
func WithConfigValues(m map[string]any) Option

func NewPropertyResolver(opts ...Option) *PropertyResolver

func (r *PropertyResolver) Resolve(parameters []op.Parameter) (map[string]any, []error)
func (r *PropertyResolver) Get(name string) (any, Origin, bool)
```

Construction via functional options; **no exported fields**. All sources
captured at construction; the resolver is read-only after that until `Resolve`
populates `resolved`. After `Resolve` the resolver is fully read-only.

**Resolution order per parameter:**

1. `callerOverrides[name]` if present.
2. `os.Getenv(envPrefix + "_" + camelToSnake(name))` — e.g.,
   `DEVLORE_WRIT_TARGET_ROOT`.
3. `os.Getenv(globalEnvPrefix + "_" + camelToSnake(name))` — e.g.,
   `DEVLORE_TARGET_ROOT`.
4. `configValues[name]` if present.
5. `*parameter.Default` if non-nil.

First hit wins. After the walk completes per parameter:

- Type-assert the picked value against `parameter.Type`. Mismatch → error.
- Record `resolved[name] = property{value, origin}`.
- For each required parameter (`Default == nil`) that found nothing → error
  ("missing required parameter %q (type %s); set via override, %s, %s, or
  config", name, typeName, programEnvKey, globalEnvKey).

`Resolve` returns the flat `map[string]any` for
`GraphExecutor.Run(graph, overrides)` and an aggregated `[]error`. Caller
folds errors into the D5 preflight envelope.

`Get(name)` returns `(value, origin, ok)`. Panics if called before `Resolve` —
that's a programmer error.

### D5 — Origin tracking

Single `Source` string identifies both source kind and literal key used:

- `"caller"` — explicit `callerOverrides[name]`
- `"env:DEVLORE_WRIT_TARGET_ROOT"` — program-specific env hit
- `"env:DEVLORE_TARGET_ROOT"` — global env fallback hit
- `"config:target_root"` — config map key
- `"default"` — declared default

One field keeps the type simple and diagnostic output cheap. Structured access
(e.g., parsing `"env:..."` to discriminate program vs global env) is rarely
needed in practice — when it is, callers can `strings.HasPrefix`.

### D6 — Override routing at `Execute()`

`GraphExecutor.Run(graph, overrides)` signature is unchanged. Dispatch reaches
a slot with `ParameterValue{Name: X}` and resolves via `overrides[X]`. No
`Properties` lookup remains anywhere in the executor.

Subgraph composition: when a parent graph's slot fill for a child subgraph is
`ParameterValue{Name: Y}`, the override map at the root carries `Y`. The
subgraph's matching slot resolves to the same value. Bubble-up at runtime
mirrors the bubble-up declared at plan time.

### D7 — Preflight integration

New preflight pass `bindParameters` added to `GraphExecutor.Run`, joining
`ResolvePending` (landed by 13.0(k) k.15) and future orphan / type-check /
platform passes:

1. Compute `graph.Parameters()` — the root's bubble-up surface.
2. For each parameter:
   - Check `overrides[name]` exists, OR parameter has a default.
   - If parameter has a default, type-assert `*Default` against `Type`.
   - If override is present, type-assert `overrides[name]` against `Type`.
3. Collect errors; aggregate into the D5 envelope.
4. Fail fast on non-empty: graph transitions to `StateFailed`; executor returns
   `errors.Join(errs...)`.

Skipped in dry-run, matching `ResolvePending`'s convention
(`pkg/op/executor.go:79-84`).

## Test strategy

### Starlark-first integration test

Phase 1 writes the test before the infrastructure exists. The test fails
loudly; subsequent phases reduce failures until all pass. By Phase 6 the
integration test passes, and the remaining phases (7–8) preserve that pass
while retiring the old model and wiring writ.

The test's role is dual:

1. **Specification.** The starlark surface defines what the binding API looks
   like end-to-end — what the user types, what errors they see, what data is
   available post-execution.
2. **Regression guard.** Once passing, the test catches any regression
   introduced by retirement or migration phases.

### Go unit coverage

Each new Go type / function in `pkg/binding` and the modified `pkg/op` slot
machinery gets unit tests in the standard `*_test.go` pattern. These tests
exercise edges the integration test cannot reach economically (camelCase →
SNAKE_CASE rules, panic-on-pre-Resolve, multi-root collision detection).

### Writ-side coverage

Phase 8 adds a writ-side test for the graph-construction helper. The
behavioral test for `writ adopt` continues to pass — same observable
filesystem behavior, different internal path.

## Open questions to lock during implementation

1. **`camelToSnake` rules for env keys.** Edge cases like `ID` (`I_D` or
   `ID`?), `URLPath` (`URL_PATH` or `U_R_L_PATH`?). Pick rules consistent with
   Go's `strings.ToSnakeCase` conventions or document a custom rule.
2. **`plan.param()` default kwarg semantics.** Kwarg-presence vs explicit
   `None`: `plan.param("x")` vs `plan.param("x", default=None)` vs
   `plan.param("x", default=value)`. Locked: kwarg-presence drives the
   distinction; `default=None` is equivalent to absence.
3. **Helper location in `cmd/writ/writ/`.** New file (`adopt_helper.go`?) or
   fold into `commands.go`? Lean toward new file — `commands.go` is already
   large (~1450 lines).
4. **Phase 6 `plan.param` kwarg parsing.** Codegen pathway or hand-written
   builtin? Likely hand-written — `plan.param` is a special-form builtin, not
   a generated provider method.
