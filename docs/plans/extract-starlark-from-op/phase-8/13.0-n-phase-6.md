# Plan: 13.0(n) Phase 6 — `writ adopt` migration

## Summary

Rewire the three nil-activation file-provider call sites in `cmd/writ/writ/commands.go`'s `adoptFile` helper to use the Phase-1-through-Phase-5 infrastructure (`op.Plan`, `op.VariableValue`, `op.GraphExecutor.Run`). The same observable filesystem behavior — create destination directory, move file into repo, symlink back to original location — is now produced by a real materialized graph with bubble-up parameter binding and a single executor preflight pass. `writ adopt` becomes the integration discriminator that proves the new binding model carries a real-world workflow end-to-end. Phase 6 is the exit gate for 13.0(n); when it lands, 13.0(n) closes.

## Goals

1. **Delete the three nil-activation patterns.** `fp.Mkdir(nil, …)`, `fp.Move(nil, …)`, `fp.Link(nil, …)` at `commands.go:1406/1416/1428` are the last instances of the old "construct provider, call method, pass nil activation" pattern in `writ adopt`'s hot path. After this PR, that pattern doesn't exist in `cmd/writ/writ/` for adopt.
2. **Realize each CLI argument as an `op.VariableValue` slot.** No user-controlled input lives in `op.ImmediateValue`. Cobra-parsed flags land in `Application.Flags` under snake-cased names; the executor's preflight resolves them through the cascade per Phase 4 and fills `env.variables`; dispatch populates slots at run time.
3. **Turn the seven `test_writ_adopt*.star` integration tests green.** Written test-first in Phase 1, currently outside `runner_test.go` because the infrastructure didn't support them. Phase 6 wires them in and they become regression guards for the migration.
4. **Match the writ-side existing-command package layout.** `cmd/writ/writ/migrate_cmd.go` + `cmd/writ/writ/migrate/` is the precedent; six other commands follow the same shape. Adopt joins.
5. **Preserve every observable `writ adopt` behavior.** Same exit code on missing-args, same `cli.Success`/`cli.Warn` output, same on-disk state at every step, same dry-run shape.

## Requirements

### R1 — Package extraction

`cmd/writ/writ/commands.go` shrinks by ~500 lines (the full adopt surface moves out). Layout:

```
cmd/writ/writ/
  adopt_cmd.go          ← NEW. newAdoptCmd, runAdopt, AdoptConfig (cobra glue,
                          flag parsing, validation, dry-run handling). All
                          extracted from commands.go.
  adopt/                ← NEW subpackage.
    adopt.go            ← AdoptFiles, AdoptItem — per-file iteration scaffolding
                          (walks the input list, calls plan→execute per file).
    plan.go             ← BuildGraph(env, src, dest, mode) — the op.Plan closure
                          factored out for isolated unit testability. Each adopt
                          argument enters as op.VariableValue{Name: "..."}.
    execute.go          ← Run(executor, graph) — wraps executor.Run with
                          adopt-specific error mapping (so the caller still sees

```

This mirrors the existing `migrate_cmd.go` + `migrate/` precedent — six other commands in `cmd/writ/writ/` already follow this shape.

### R2 — `adopt.BuildGraph` shape

The graph's `Root` carries three children (mkdir → move → link). Variable references on every user-controlled slot:

```go
func BuildGraph(env *op.RuntimeEnvironment, _ AdoptConfig) (*op.Graph, error) {
    graph := op.NewGraph()
    graph.Rebind(env)

    mkdirInv := plan.file.mkdir(
        path:  op.VariableValue{Name: "dest_dir"},
        chmod: op.ImmediateValue{Value: os.FileMode(0o755)},  // tool-set, not user
    )
    moveInv := plan.file.move(
        source:           op.VariableValue{Name: "source_path"},
        destination_path: op.VariableValue{Name: "dest_path"},
    )
    linkInv := plan.file.link(
        source:      op.VariableValue{Name: "dest_path"},
        target_path: op.VariableValue{Name: "source_path"},
    )
    // ... assemble onto graph.Root via plan.Provider.Assemble or direct AddChild
    return graph, nil
}
```

The Go-side construction uses the planning provider's APIs (`plan.Provider.Variable`, `plan.Provider.Assemble`, etc.) directly — type-safe, no `.star` runtime dependency in the `writ` binary, debugger steps through end-to-end. **Decision locked (Q1):** Go closure, not embedded `.star`.

### R3 — `adopt.Run` shape

```go
func Run(executor *op.GraphExecutor, graph *op.Graph) error {
    _, err := executor.Run(ctx, nil)  // variables come from env.Application
    if err != nil {
        return mapAdoptError(err)  // translate to commands.go-style messages
    }
    return nil
}
```

`executor.Run(ctx, nil)` triggers preflight (variable resolution + `ResolvePending`). The resolver reads `env.Application.Flags` for cobra-parsed values. `mapAdoptError` translates `op`-layer errors into the `cli.Note`/`cli.Warn`-friendly shape the existing UX expects.

### R4 — Cross-filesystem move: no special-case

The current `adoptFile` has an inline `copyFile + os.Remove` fallback when `file.Move` returns `EXDEV`. **Decision locked (Q2):** delete the fallback. `file.Move` keeps its atomic-rename-or-fail contract. `writ adopt` operates within a constrained `os.Root` (Home root = `~`; System root = `/`); for the Home root, source and destination are typically on the same filesystem and EXDEV is a deep edge case. When it does occur, the user sees a clear "invalid cross-device link" error. Users who want recovery wire an `error_action=` subgraph on the move invocation — that's the framework's universal escape hatch and doesn't need to be baked into `writ adopt`'s default path. A `--on-cross-fs=copy-remove` CLI flag is a possible follow-up if real-world use surfaces the need.

### R5 — Dry-run preservation

The existing `if dryRun { cli.Note("Would adopt ..."); return 1, nil }` short-circuit becomes either:

- **(a) Pre-graph** — `adopt.Run` checks `executor.DryRun()` (or equivalent) and prints the would-do narrative without dispatching. Cheap; preserves the existing UX exactly.
- **(b) Through the graph** — every provider method implements its own dry-run output; `executor.Run` skips the actual filesystem effects.

(b) is the right long-term shape but Phase 6's scope is the migration, not the dry-run audit. **Use (a) for Phase 6.**

## Implementation Phases

### Phase 6.0 — Convertibility-aware bubble-up + provider TargetConverter contract (framework prerequisite)

Framework change required before any `writ adopt` migration code lands. The bubble-up parameter consistency check (`Subgraph.mergeBubbled`, `pkg/op/subgraph.go:472`) currently uses strict type equality: same-named variable bound to slots of different types is a hard error. That's wrong — slot-fill is *defined* as the conversion site, and the framework already has `op.Convert` + `SourceConverter` + `TargetConverter` for value-side conversions. The plan-time consistency check needs to use the same conversion table.

**Principle (locked):** A variable carries one type. Slot-fill converts values on demand. Same-named variable bound to differently-typed slots is **not** a collision when a registered conversion bridges them — in either direction.

**Contract for providers:** A Resource type opts into source-type coercion by implementing `TargetConverter` (`CanConvertFrom(reflect.Type) bool` + `ConvertFrom(value any) (any, error)`) or by being the source-side value implementing `SourceConverter`. The framework wires both: `op.Convert` consults them at slot-fill (dispatch time), and `Subgraph.mergeBubbled` consults the same convertibility relation at plan time. No per-provider plumbing.

- [ ] **6.0.1** Add `typesAreInterconvertible(a, b reflect.Type) bool` in `pkg/op/`. Probes both directions: instantiate a fresh `b`, call `(b).(TargetConverter).CanConvertFrom(a)`; instantiate a fresh value of type `a`, call `(a).(SourceConverter).CanConvertTo(b)`. Either-true ⇒ compatible. Document the cheap-probe contract (no actual conversion executed).
- [ ] **6.0.2** Update `Subgraph.mergeBubbled` (`subgraph.go:472`) to consult `typesAreInterconvertible` before declaring "incompatible types ... across slots". When types differ but are interconvertible, keep the first-seen `Parameter` in `seen` (the variable's reported type stays stable across the walk).
- [ ] **6.0.3** Cross-provider `TargetConverter` audit. For each Resource-bearing provider — `file`, `git`, `appnet`, `pkg`, `service`, `mem`, `function`, `json`, `yaml` — decide whether the Resource absorbs `string` sources (and/or `[]byte` for content-addressed types). Implement `TargetConverter` per the decision; document the "no, intentional" choices in code where applicable. Audit matrix (informed by today's catalog model):
  - `file.Resource` — `string` (path) → yes; matches today's `commands.go:1438` pattern.
  - `git.Resource` — `string` (URL or local path) → yes.
  - `appnet.Resource` — `string` (URL) → yes.
  - `pkg.Resource` — `string` (package name) → yes.

  - `function.Resource` — `string` (source text or symbol) → yes (via embedded `mem.Resource`).
  - `json.Resource` / `yaml.Resource` — `[]byte` / `string` / `io.Reader` → yes; already accept these via `NewResource`.
- [ ] **6.0.4** Convertibility contract documentation. Code-side: doc-comments on `SourceConverter`/`TargetConverter` interfaces explain the bubble-up consistency obligation, not just the dispatch-time use. Plan-side: a "Convertibility contract" section in this document (or a dedicated `docs/design/convertibility.md`) for provider developers.
- [ ] **6.0.5** Tests: `typesAreInterconvertible` unit tests (bidirectional, one-direction, neither-direction); `mergeBubbled` collision-suppression test (string + `*file.Resource` no longer errors); each new provider `TargetConverter` round-trip test.
