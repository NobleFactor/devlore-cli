# Draft: System Model from Execution Graphs

Design topic for a future plan. Not yet approved for implementation.

---

## Core Insight

The execution graph is not just a deployment artifact — it is the system model.
Each node (machine) in a distributed system gets its own execution graph. The
collection of receipts across all nodes provides drift detection and a unified
system model for manageability.

This is code-focused: Starlark scripts define intent, graphs are the concrete
artifacts, receipts capture realized state.

---

## The Architecture of Emergent Reality

The process creates a continuous feedback loop:

1. **The Intent (The Code):** The coordinator sends a logic-bound DAG, not
   commands.
2. **The Realization (The Node):** The node traverses the graph through phases
   (Prepare, Install, Provision, Verify).
3. **The Artifact (The Receipt):** The resulting graph is a reified model of
   the node's state — not a log, but a structured record with provenance.

This is an inversion of traditional top-down configuration management. Instead
of telling a system what it should be and hoping it complies, the execution
record is the source of truth. The receipt (the completed subgraph) gives the
system built-in provenance.

---

## Why Receipts Are Superior to State Files

In systems like Terraform or SaltStack, a "state file" acts as a middleman. If
that file gets corrupted or out of sync with hardware, recovery is difficult.
In the receipt model:

- **Verifiable dependencies:** Because the model IS the execution graph, you
  know exactly why Component B exists — it was a child of Component A in the
  graph.
- **Phase-aware recovery:** If a node fails during "Provision," the receipt
  tells the coordinator exactly where the graph stopped, enabling surgical
  remediation rather than full wipe-and-reinstall. (See also: `draft.md` —
  Recovery Stack Serialization & Restart.)
- **Natural drift detection:** Re-run the "Verify" phase of the existing
  receipt. If reality on disk doesn't match the verify nodes in the graph,
  you have immediate, code-defined drift detection.

## Cross-Node Dependencies: The Global Graph

When the coordinator downloads receipts from N nodes, it possesses N "local
truths." Cross-node dependencies require a global graph.

**Example:** Node A's "Provision" phase requires an API key generated by Node
B's "Install" phase. The coordinator must pause Node A's subgraph until it
receives the success receipt (and the necessary data) from Node B.

### How This Maps to Existing Primitives

The orchestration primitives already support this at a different scale:

| Local (single machine) | Global (distributed) |
|---|---|
| Node = action | Node = machine subgraph |
| Edge = data dependency | Edge = cross-machine dependency |
| Phase = lifecycle boundary | Phase = deployment wave |
| Gather = parallel items | Gather = parallel machine provisioning |
| Choose = conditional branch | Choose = platform-specific provisioning |
| Result = action output | Result = machine receipt |
| Promise slot = upstream ref | Promise slot = cross-machine data ref |

The coordinator holds a **global execution graph** where:

- Each "node" is a machine's subgraph (or a reference to one)
- Edges represent cross-node dependencies
- Gather runs machine subgraphs in parallel with concurrency limits
- Promise slots carry data between machine receipts (e.g., API keys, IPs)
- The global receipt is the system model

### Open Questions

1. **Graph granularity:** Is each machine a single node in the global graph, or
   does the global graph have visibility into per-machine phases?
2. **Data flow between machines:** How does Node B's result (API key) flow to
   Node A's slots? Through the coordinator's results map, or through a shared
   data store?
3. **Partial failure:** If 3 of 100 machines fail, the global graph has 97
   successful receipts and 3 partial receipts. How does drift detection handle
   the partial state?
4. **Reconciliation frequency:** Is drift detection periodic (cron), event-driven
   (webhook), or on-demand?
5. **Global graph serialization:** The global graph + all machine receipts = the
   system model. What's the storage format? A single document, or a directory
   of per-machine receipts with a global index?

## External Dependencies and Data Propagation

Since we're moving from local node "truths" to a distributed "global truth," the
graph needs a way to handle **External Dependencies** (blocking edges) and **Data
Propagation** (lazy-loaded values).

In a system where the model *emerges* from the execution, these dependencies
aren't just strings; they are **Promises** that the Coordinator must resolve by
bridging two different subgraphs.

### The "Interface" Node Pattern

To keep the node-level execution clean, introduce **Interface Nodes** (Input and
Output) within the subgraph.

- **Input Nodes (Requirement):** A node that blocks execution of its children
  until the Coordinator injects a value (e.g., `db_endpoint`).
- **Output Nodes (Provision):** A node that, upon successful "Verify" phase,
  exports a value to the Coordinator (e.g., `api_key`).

### The Dependency Query Mechanism

For the "emergent model" to be useful, the Coordinator needs a way to traverse
the collective receipts. This requires a **Distributed Graph Query**:

1. **Selection:** Find all nodes of type `LoadBalancer`.
2. **Traversal:** Follow the `depends_on` edge to find their `TargetGroups`.
3. **Resolution:** Extract the `ip_address` attribute from the "Verify" phase
   data in the receipt.

### Handling the "Wait-State"

How does the Orchestration Coordinator manage the lifecycle of a cross-node
dependency?

| Strategy | Mechanism | Pro/Con |
|---|---|---|
| **Reactive (Push)** | Node B finishes → Coordinator updates Node A's pending Input Node → Node A resumes. | **Pro:** Lower latency. **Con:** Requires persistent connections or webhooks. |
| **Polling (Pull)** | Node A reaches a "Wait" node and asks the Coordinator: "Is `db_ready` true yet?" | **Pro:** Simpler node logic. **Con:** Can create "thundering herd" traffic on the Coordinator. |

### Late-Binding

One of the defining properties of an "emergent" model is **Late-Binding**. If
the graph is built in code, the "Address" of a dependency doesn't need to exist
when the graph is *deployed* — only when that specific *phase* of the graph is
*executed*.

The current code-based graph builder supports "Future" / "Promise" types
(SlotValue with NodeRef) that represent these cross-node values. The "Wait"
state is handled by the `flow.wait_until` primitive, which provides event-driven
synchronization with polling fallback and timeout.

## Slots, Promises, and Proxies: JIT Infrastructure

The codebase is rooted in Slots that hold three kinds of values:

1. **Immediate values** — known at plan time
2. **Promises** — references to upstream node results, resolved at execution time
3. **Proxies to promises** — a specialization for subroutine behavior (gather
   iteration binding)

By using Proxies to Promises, the system functions as a "Just-In-Time"
infrastructure compiler. The set of target servers can be gathered *during*
execution rather than being hard-coded in a static inventory. This treats
**compute resources as a first-class variable** in the graph logic.

### The Lifecycle of a Slot

1. **Resolution:** An execution node hits a Slot.
2. **The Proxy Trigger:** If the Slot holds a Proxy, it triggers the subroutine
   (e.g., a discovery script or a cloud API call).
3. **The Promise:** The Proxy returns a Promise while discovery is in flight.
4. **The Fulfillment:** Once the servers are identified, the Promise resolves,
   and the subgraphs are dispatched to the newly discovered fleet.

### Procedural-Emergent vs. Declarative

Traditional models are **declarative** (State A must exist). This architecture
is **procedural-emergent** (Execute Logic B to discover State A).

| Element | Static Architecture | Slot & Proxy Architecture |
|---|---|---|
| **Inventory** | Known at plan time. | Emerges at execution time. |
| **Dependency** | Hardcoded (ID-based). | Functional (Query-based). |
| **Drift** | "The config changed." | "The logic-path resulted in a different graph." |

### The Phantom Node Problem

If a Proxy triggers a subroutine that deploys to 10 servers, but 2 fail the
Verify phase, how does "Partial Success" reflect back into the parent Slot?

Options:

- **Atomic transaction:** The Slot remains "Pending" because the full promise
  isn't fulfilled. The entire subgraph invocation is treated as all-or-nothing.
- **Shattered Result:** The parent graph records 8 successful branches and 2
  "Error" receipts. The emergent model shows partial resolution.

### Open Question: Partial Resolution

Do Receipts allow for "Partial Resolution" of a Slot? Current design leans
toward the Shattered Result model — GatherUndoState preserves per-iteration
results and recovery entries, which naturally supports partial success/failure.
The Gather action can report both successful iterations and failed ones,
letting the coordinator decide the policy (fail-fast, best-effort, or
threshold-based).
