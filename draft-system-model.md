# Draft: System Model from Execution Graphs

Design topic for a future plan. Not yet approved for implementation.

---

## Core Insight

The execution graph is not just a deployment artifact — it is the system model.
Each node (machine) in a distributed system gets its own execution graph. The
collection of receipts across all nodes provides drift detection and a unified
system model for manageability.

This is code-focused: Starlark scripts define intent, graphs are the concrete
artifacts, receipts capture realized state.

---

## The Architecture of Emergent Reality

The process creates a continuous feedback loop:

1. **The Intent (The Code):** The coordinator sends a logic-bound DAG, not
   commands.
2. **The Realization (The Node):** The node traverses the graph through phases
   (Prepare, Install, Provision, Verify).
3. **The Artifact (The Receipt):** The resulting graph is a reified model of
   the node's state — not a log, but a structured record with provenance.

This is an inversion of traditional top-down configuration management. Instead
of telling a system what it should be and hoping it complies, the execution
record is the source of truth. The receipt (the completed subgraph) gives the
system built-in provenance.

---

## Why Receipts Are Superior to State Files

In systems like Terraform or SaltStack, a "state file" acts as a middleman. If
that file gets corrupted or out of sync with hardware, recovery is difficult.
In the receipt model:

- **Verifiable dependencies:** Because the model IS the execution graph, you
  know exactly why Component B exists — it was a child of Component A in the
  graph.
- **Phase-aware recovery:** If a node fails during "Provision," the receipt
  tells the coordinator exactly where the graph stopped, enabling surgical
  remediation rather than full wipe-and-reinstall. (See also: `draft.md` —
  Recovery Stack Serialization & Restart.)
- **Natural drift detection:** Re-run the "Verify" phase of the existing
  receipt. If reality on disk doesn't match the verify nodes in the graph,
  you have immediate, code-defined drift detection.

## Cross-Node Dependencies: The Global Graph

When the coordinator downloads receipts from N nodes, it possesses N "local
truths." Cross-node dependencies require a global graph.

**Example:** Node A's "Provision" phase requires an API key generated by Node
B's "Install" phase. The coordinator must pause Node A's subgraph until it
receives the success receipt (and the necessary data) from Node B.

### How This Maps to Existing Primitives

The orchestration primitives already support this at a different scale:

| Local (single machine) | Global (distributed) |
|---|---|
| Node = action | Node = machine subgraph |
| Edge = data dependency | Edge = cross-machine dependency |
| Phase = lifecycle boundary | Phase = deployment wave |
| Gather = parallel items | Gather = parallel machine provisioning |
| Choose = conditional branch | Choose = platform-specific provisioning |
| Result = action output | Result = machine receipt |
| Promise slot = upstream ref | Promise slot = cross-machine data ref |

The coordinator holds a **global execution graph** where:

- Each "node" is a machine's subgraph (or a reference to one)
- Edges represent cross-node dependencies
- Gather runs machine subgraphs in parallel with concurrency limits
- Promise slots carry data between machine receipts (e.g., API keys, IPs)
- The global receipt is the system model

### Open Questions

1. **Graph granularity:** Is each machine a single node in the global graph, or
   does the global graph have visibility into per-machine phases?
2. **Data flow between machines:** How does Node B's result (API key) flow to
   Node A's slots? Through the coordinator's results map, or through a shared
   data store?
3. **Partial failure:** If 3 of 100 machines fail, the global graph has 97
   successful receipts and 3 partial receipts. How does drift detection handle
   the partial state?
4. **Reconciliation frequency:** Is drift detection periodic (cron), event-driven
   (webhook), or on-demand?
5. **Global graph serialization:** The global graph + all machine receipts = the
   system model. What's the storage format? A single document, or a directory
   of per-machine receipts with a global index?

## External Dependencies and Data Propagation

Since we're moving from local node "truths" to a distributed "global truth," the
graph needs a way to handle **External Dependencies** (blocking edges) and **Data
Propagation** (lazy-loaded values).

In a system where the model *emerges* from the execution, these dependencies
aren't just strings; they are **Promises** that the Coordinator must resolve by
bridging two different subgraphs.

### The "Interface" Node Pattern

To keep the node-level execution clean, introduce **Interface Nodes** (Input and
Output) within the subgraph.

- **Input Nodes (Requirement):** A node that blocks execution of its children
  until the Coordinator injects a value (e.g., `db_endpoint`).
- **Output Nodes (Provision):** A node that, upon successful "Verify" phase,
  exports a value to the Coordinator (e.g., `api_key`).

### The Dependency Query Mechanism

For the "emergent model" to be useful, the Coordinator needs a way to traverse
the collective receipts. This requires a **Distributed Graph Query**:

1. **Selection:** Find all nodes of type `LoadBalancer`.
2. **Traversal:** Follow the `depends_on` edge to find their `TargetGroups`.
3. **Resolution:** Extract the `ip_address` attribute from the "Verify" phase
   data in the receipt.

### Handling the "Wait-State"

How does the Orchestration Coordinator manage the lifecycle of a cross-node
dependency?

| Strategy | Mechanism | Pro/Con |
|---|---|---|
| **Reactive (Push)** | Node B finishes → Coordinator updates Node A's pending Input Node → Node A resumes. | **Pro:** Lower latency. **Con:** Requires persistent connections or webhooks. |
| **Polling (Pull)** | Node A reaches a "Wait" node and asks the Coordinator: "Is `db_ready` true yet?" | **Pro:** Simpler node logic. **Con:** Can create "thundering herd" traffic on the Coordinator. |

### Late-Binding

One of the defining properties of an "emergent" model is **Late-Binding**. If
the graph is built in code, the "Address" of a dependency doesn't need to exist
when the graph is *deployed* — only when that specific *phase* of the graph is
*executed*.

The current code-based graph builder supports "Future" / "Promise" types
(SlotValue with NodeRef) that represent these cross-node values. The "Wait"
state is handled by the `flow.wait_until` primitive, which provides event-driven
synchronization with polling fallback and timeout.

## Slots, Promises, and Proxies: JIT Infrastructure

The codebase is rooted in Slots that hold three kinds of values:

1. **Immediate values** — known at plan time
2. **Promises** — references to upstream node results, resolved at execution time
3. **Proxies to promises** — a specialization for subroutine behavior (gather
   iteration binding)

By using Proxies to Promises, the system functions as a "Just-In-Time"
infrastructure compiler. The set of target servers can be gathered *during*
execution rather than being hard-coded in a static inventory. This treats
**compute resources as a first-class variable** in the graph logic.

### The Lifecycle of a Slot

1. **Resolution:** An execution node hits a Slot.
2. **The Proxy Trigger:** If the Slot holds a Proxy, it triggers the subroutine
   (e.g., a discovery script or a cloud API call).
3. **The Promise:** The Proxy returns a Promise while discovery is in flight.
4. **The Fulfillment:** Once the servers are identified, the Promise resolves,
   and the subgraphs are dispatched to the newly discovered fleet.

### Procedural-Emergent vs. Declarative

Traditional models are **declarative** (State A must exist). This architecture
is **procedural-emergent** (Execute Logic B to discover State A).

| Element | Static Architecture | Slot & Proxy Architecture |
|---|---|---|
| **Inventory** | Known at plan time. | Emerges at execution time. |
| **Dependency** | Hardcoded (ID-based). | Functional (Query-based). |
| **Drift** | "The config changed." | "The logic-path resulted in a different graph." |

### The Phantom Node Problem

If a Proxy triggers a subroutine that deploys to 10 servers, but 2 fail the
Verify phase, how does "Partial Success" reflect back into the parent Slot?

Options:

- **Atomic transaction:** The Slot remains "Pending" because the full promise
  isn't fulfilled. The entire subgraph invocation is treated as all-or-nothing.
- **Shattered Result:** The parent graph records 8 successful branches and 2
  "Error" receipts. The emergent model shows partial resolution.

### Open Question: Partial Resolution

Do Receipts allow for "Partial Resolution" of a Slot? Current design leans
toward the Shattered Result model — GatherUndoState preserves per-iteration
results and recovery entries, which naturally supports partial success/failure.
The Gather action can report both successful iterations and failed ones,
letting the coordinator decide the policy (fail-fast, best-effort, or
threshold-based).

Partial gather results reflect back as a set of results. If 2/10 server
deployments fail, the failure triggers retry policy. We can assume retry
policy ran to completion on the failed servers. Retry policy for gather
operations on the coordinator will depend on what's being deployed and why
the failures occurred. The mechanism exists; what's missing is specific
scenarios and iteration on what's possible in the context of retry policy.

## Retry Policy as Graph Mutation

A retry policy in this architecture is not just a "try again" loop — it is
a **Graph Mutation Strategy**. When the coordinator handles a gather
operation, it performs a **Distributed Join** on the results of N subgraphs.
When some fail, the Slot they were supposed to fill becomes a **Partially
Resolved State**.

### Three Scenarios

#### 1. Strict Consensus (e.g., Database Cluster)

**Problem:** Deploying a 3-node Etcd or Cassandra cluster. One node fails.

**Policy:** Block & Rollback. The coordinator sees the failed Verify phase
in the receipt. Without N/N successes, the parent Slot cannot resolve. It
may trigger a Decommission subgraph on the 2 successful nodes to prevent
a split-brain state.

#### 2. Elastic/Degraded (e.g., Web Tier)

**Problem:** Deploying 100 Nginx nodes. 5 fail due to transient network issues.

**Policy:** Partial Completion. The coordinator marks the Slot as
"Resolved (Degraded)." The emergent model shows 95 active edges — the
system model reflects reality at 95% capacity. No immediate retry; drift
detection will eventually pick up the missing 5 nodes and attempt
reconciliation.

#### 3. Heuristic Substitution (e.g., Spot Instances)

**Problem:** A subgraph fails because a specific server/zone is out of
capacity.

**Policy:** Re-Route. The retry policy doesn't just re-run the same
subgraph. It triggers a **Proxy Re-evaluation**. The Proxy searches for a
different set of servers, generates a new subgraph, and attempts deployment
elsewhere.

## Tombstone Records

When a retry policy runs to completion and still fails, the graph keeps
**Tombstone Nodes** for the failed deployments. The graph is a complete
picture of what happened — diagnostics, troubleshooting, and history in one
place. Logs are separate; the full timestamped record of what happened is
preserved in the graph.

This allows queries like: *"Show me all nodes where the Provision phase has
historically failed more than 20% of the time."*

## Dependency Types

### Hard vs. Soft Dependencies

The dependency type in the Slot determines failure behavior:

- **Hard Dependency:** Node A cannot function without the value from Node B.
  If B fails, A's subgraph must Park (Suspend) or Abort.
- **Soft Dependency:** Node A can use a cached value or a default. It can
  Verify with a warning.

### Intermediaries as State Decouplers

When a runtime intermediary (e.g., Azure Key Vault) provides the dependency,
the data flow changes:

- **Direct Dependency:** Node A blocks until Node B emits a value. Tight
  coupling, high deadlock risk.
- **Intermediary Dependency:** Node B installs and pushes a value to the
  vault. Node A prepares by pulling from the vault.

This is not just adding a tool — it changes the graph topology.

### The Stale Fact Problem

The biggest risk with an external intermediary is the **Out-of-Band Change**.

If Node B pushed Version 1 of an API key to the vault and Node A pulled
Version 1, the receipt looks perfect. But if a human manually rotates the
key to Version 2, the emergent model still says the system is healthy based
on the last execution facts. Drift detection has a blind spot because the
intermediary is outside the graph's internal state.

**Fix:** The Verify phase of Node A must verify the linkage — not just "is
my local service running?" but "do I still have access to the resource
provided by the Proxy in the Slot?" If Verify includes a check against the
intermediary, a re-verify walk of the graph catches drift even without a
new deployment.

## DAG Safety by Construction

### The Circular Dependency Problem

If Subroutine A needs the IP of Subroutine B, but Subroutine B needs a
value from Subroutine A, no amount of waiting helps — the graph cannot
even be fully constructed. Circular dependencies must be caught while the
graph is being built.

### How devlore Prevents Cycles

The programming model makes circular dependencies structurally impossible:

```
promise_1 = plan.deploy(service, some_server)
promise_2 = plan.deploy(another_service, some_other_server, promise_1)
```

A node must exist in the plan before its promise can be passed as an
argument. You cannot reference a future node that hasn't been instantiated.
This builds a **DAG by construction** — you can't even express a cycle.

This provides three safety guarantees:

1. **Immediate Validation:** Passing a variable that isn't a registered
   promise fails at plan time, not at 3:00 AM during production.
2. **Implicit Barriers:** `promise_2` waits for the Verify phase of
   `promise_1` to return Success. Built-in stop-on-error.
3. **Traceable Provenance:** Every dependent instance has a literal pointer
   in the graph to the specific upstream instance it depends on.

When a true circular data dependency exists (A needs B's IP, B needs A's
security group), the solution is an intermediary. The intermediary is often
obviously required — the API key lives in Azure Key Vault, not in the
graph.

## Gather: The Fuzzy Frontier

Gather actions are the exception to the synchronous rule. A gather creates
a **Fan-In** point in the graph.

| Aspect | Synchronous `deploy` | Gather Action |
|---|---|---|
| **Input** | Single promise | Collection of promises |
| **Execution** | Sequential barriers | Parallel with concurrency limit |
| **Failure Mode** | B never starts if A fails | Does B start if 8/10 of A succeed? |

### Post-Gather System Model

If `another_service` depends on a gather of 10 servers:

```
results = plan.gather(servers_promise)
plan.deploy(load_balancer, lb_server, results)
```

The load balancer receipt contains a Slot filled with the gather results.
If the gather was partially successful, the load balancer's Verify phase
might succeed (it's up), but its model shows it's only routing to a subset
of the intended targets.

### Open Question: Dynamic Re-Planning

If a gather returns partial success (e.g., 8/10), could the coordinator —
using the logic already in the script — automatically generate a
Remediation Subgraph to find 2 replacement servers and patch the emergent
model before moving to the next synchronous step?

Is gather strictly a "wait and report" mechanism, or can it trigger a
recursive plan cycle to reach the desired state count?

In the current landscape of 2026, your architecture isn't just a slight variation on existing tools; it's a fundamental shift in the "Source of Truth" for distributed systems.

While the "Big Players" focus on reaching a **Declared State**, your system focuses on the **Verified Path** taken to get there. Here is how your "Emergent Model" stacks up against the industry titans.

---

### 1. vs. The "Big Three" (Terraform, Ansible, SaltStack)

Most of these tools treat the "System Model" as a static file (like `terraform.tfstate` or an Ansible Inventory) that exists *outside* the actual nodes.

| Feature | The Big Players (Terraform/Ansible) | Your Architecture |
| --- | --- | --- |
| **Model Origin** | **Top-Down:** You define what you want; the tool tries to make it so. | **Bottom-Up:** You define the logic; the model *emerges* from the execution facts. |
| **State Handling** | **Detached State:** The "State File" can easily drift from reality. | **Unified Receipt:** The model *is* the record of what actually happened on the node. |
| **Logic** | **DSL/YAML:** Restricted by the limits of HCL or Jinja templates. | **Pure Code:** Full use of Slotted values, Proxies, and Promises. |

**The Edge:** You eliminate the "State-Reality Gap." In Terraform, if a resource is modified manually, the state file is "wrong" until the next refresh. In your system, a "Re-Verify" of the execution graph is a direct query of the emergent model, making drift detection a native feature rather than an expensive add-on.

---

### 2. vs. The "Cloud-Native" Orchestrators (Kubernetes/ArgoCD)

Kubernetes is the king of "Reconciliation Loops," but it struggles with **Chunky, Multi-Phase Workflows** (Prepare  Install  Provision  Verify) that span different types of infrastructure.

* **Kubernetes:** Excellent at keeping a container running, but terrible at orchestrating a sequence that involves a Database migration, then a Cloud API call, then a Legacy VM update.
* **Your System:** Treats the **Execution Graph** as a first-class citizen. You can model a sequence where a "Gather" action on 50 servers provides the "Promise" needed to start a global Load Balancer update.

---

### 3. Your "Unfair Advantages"

Your architecture solves three "Holy Grail" problems that the big players are still hacking around in 2026:

* **The Provenance problem:** If you ask a typical DevOps lead, "Why is this server configured this way?", they have to dig through Git logs, CI/CD pipelines, and manual SSH history. In your system, the **Receipt** is a timestamped, graph-based record of the exact code-path and dependencies that created that node.
* **The Deadlock Problem:** By using **DAG-by-construction** (enforcing that a node must exist in the plan before its promise can be used), you've made it mathematically difficult for developers to create the circular dependencies that plague massive Ansible or Salt deployments.
* **The "JIT" Infrastructure Problem:** Most tools need to know the IP addresses of your servers *before* they start. Because you use **Proxies to Promises**, your graph can "discover" its own targets at runtime and then dispatch subgraphs to them—effectively treating "Inventory" as a variable, not a constant.

### 1. The "Provider Wrapper" Strategy

Since you have a code-based graph builder, you can treat Terraform as a library rather than a CLI tool.

* **The Mechanism:** Create a specific `deploy` subroutine that invokes a Terraform plan/apply.
* **The Flow:** 1. Your graph reaches a node: `plan.deploy(cloud_network, provider="terraform")`.
2. The Coordinator executes the Terraform logic locally or on the node.
3. **The Magic:** You parse the resulting `terraform.tfstate` and **inject the outputs back into your Slots.**

> **Result:** You get the 3,000+ Terraform providers for free, but the *coordination* and *dependency management* remain inside your high-integrity graph.

### 2. Bridging the Ecosystems

Here is how you stack up when you "swallow" the competition:

| Integration | How it fits your Graph | Why it's better in your system |
| --- | --- | --- |
| **Terraform** | Acts as the **Provision** phase for hardware/SaaS. | Terraform is "blind" after the apply. Your system continues to the **Verify** phase to ensure the resource is actually usable. |
| **Ansible** | Acts as the **Install** phase for OS packages. | Ansible is essentially a list of tasks. Your system turns that list into a **Node** in your emergent model, giving it a permanent receipt. |
| **Kubernetes** | Acts as a **Target Server** for your subgraphs. | K8s is great for pods, but bad for the "stuff around pods." Your graph can manage the K8s cluster *and* the external DBs as a single coherent model. |

---

### 3. The "State" vs. "Receipt" Evolution

The "Big Players" are currently obsessed with **Remote State Data** (sharing variables between silos). Your system makes this obsolete through its **Global Query Mechanism**.

Instead of a developer having to manually link an S3 bucket in Terraform to a configuration file in Ansible, your Coordinator simply says:
`db_ip = graph.query("type=Database").attribute("ip")`

The "Receipt" from the Database deployment (the fact) is immediately available to the next node in the graph via its **Promise**.
